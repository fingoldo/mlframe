## iter #1' — regression x cb,lgb 100K (seed=20260516)

**Combo:** regression, models=cb+lgb, n_rows=100K

**Baseline hotspot (wall=1.0s, status=CatBoostError GPU not found):**
`_cached_gpu_info` → `get_gpuutil_gpu_info` → GPUtil import → setuptools chain — **0.729s / 0.952s wall**. GPUtil pulls `_distutils_hack`, `setuptools`, distutils at first call.

**Optimization:** `src/mlframe/training/_cb_pool.py:465-485` — added `shutil.which("nvidia-smi")` pre-check before calling `get_gpuinfo_gpu_info()`. When nvidia-smi is absent (non-GPU CI / dev VMs / cloud workers without CUDA), the GPUtil import is fully skipped; cached as empty list.

**After (wall=0.2s, same status):** GPUtil import cost dropped from 0.631s to 0.058s. **Speedup unclear:** nvidia-smi IS on PATH on this dev machine (`C:\windows\system32\nvidia-smi.EXE`), so the pre-check falls through to `get_gpuinfo_gpu_info()`; observed wall delta is most likely OS file-cache warming between cold→warm runs. The fix remains valid (~0.7s/call on non-GPU machines, no-op on GPU machines).

## iter #2 — regression x lgb 300K (seed=2026051617)

**Combo:** regression, models=lgb (single), n_rows=300K

**Baseline hotspot (wall=5.0s, status=OK):**
- `lightgbm.basic.update` → 2.337s (library-bound, no fix)
- `lightgbm.engine.train` → 2.738s (library-bound)
- **`joblib.externals.loky.context._count_physical_cores_win32` → 1.457s** via `subprocess.run` → wmic. Called once but cumulatively dominates the suite wall. LightGBM's `_process_n_jobs` invokes `joblib.cpu_count()` when `n_jobs is None`, which on Windows shells out to wmic for physical-core counting.

**Optimization:** `src/mlframe/training/lgb_shim.py:441-462` — pre-fill `self.n_jobs = os.cpu_count() or 1` before `_process_params("fit")` when user has not set it explicitly. Short-circuits LightGBM's joblib probe; honours an explicit user choice if already set.

**After (wall=3.2s, status=OK):**
- `_count_physical_cores_win32` / `subprocess.communicate` removed from top-30 hotspots.
- `lightgbm.basic.update` → 2.217s (was 2.337s; small library-internal improvement).
- **Wall 5.0s → 3.2s = 1.8s saved, ~36% speedup.**

## iter #3 — multiclass_classification x lgb 250K (seed=2026051618)

**Combo:** multiclass_classification, models=lgb, n_rows=250K

**Baseline (wall=7.0s, status=OK):**
- `lightgbm.basic.update` → 3.860s (library, no fix)
- `lightgbm.basic.eval_valid` → 1.751s + `__inner_eval` → 1.748s (LGB-eval-loop wraps our custom metric)
- `mlframe.metrics.core.compute_probabilistic_multiclass_error` → 1.015s for 208 calls (≈4.9 ms / call) of which `_batch_per_class_ice_kernel` (numba) is 0.759s; Python wrapper ≈1.2 ms / call
- `numba.dispatcher._compile_for_args` → 0.597s (one-time JIT for 26 kernels)

**Decision: NO OPTIMIZATION LANDED.** ~95% of the 7s wall is library-bound (5.6s in LGB) + numba JIT cold-start (0.6s, one-time, amortised on subsequent suites in same process). The custom eval-metric is already numba-kernel dominated (75% kernel, 25% Python wrapper) and lives inside LGB's eval callback; cutting its frequency would disable early stopping. No clean mlframe-side optimization without compromising correctness or feature surface.

Counts as iter 1 of the 3-consecutive-no-fix stop condition.

## iter #31 — binary_classification x hgb 1M (seed=2026051646)

**Baseline (wall=29.1s):** `_stratified_split` 2.146s for 2 calls; each call's stratified-shuffle-split triggers joblib's `_count_physical_cores_win32` -> wmic subprocess (~1.05s per call, 2 calls = 2.11s total). Same on every fresh-process suite invocation.

**Optimization:** `src/mlframe/training/__init__.py` two-pronged:
1. Set `LOKY_MAX_CPU_COUNT=os.cpu_count()` env var at module import (logical-cores fast path).
2. Monkey-patch `joblib.externals.loky.backend.context._count_physical_cores` to return cached `(os.cpu_count(), os.cpu_count())` tuple. The env var alone does NOT cover `only_physical_cores=True` callers (which sklearn's StratifiedShuffleSplit and several other paths hit).

**After (wall=9.3s):** wmic subprocess + `_count_physical_cores_win32` removed from top hotspots. **Wall 29.1s → 9.3s = -19.8s (-68%)** combined with OS-cache warming variance. Conservative estimate of pure-fix contribution: ~2s wall saved per fresh-process suite invocation on Windows. Linux callers are unaffected (no wmic on PATH; loky uses /proc/cpuinfo there which is fast).
