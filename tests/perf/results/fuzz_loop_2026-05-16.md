## iter #189 — binary x lgb,linear,ridge 1M x onehot+TruncatedSVD (seed=2026051901)

**Combo:** binary, models=lgb+linear+ridge, n_rows=1M, axes (seed-derived from enhanced harness): mrmr_fs=True, outlier=none, ensembles=False, cat_enc=onehot, scaler=standard, dim_reducer=TruncatedSVD.

**Bug surfaced (wall=112s train, predict=0.04s, status=OK on train but ALL 6 model predictions FAILED):**
- Error per model: `Model X expects features missing from input: [...]`. Some missed only `truncatedsvd0..9`, others missed both `cat_low/cat_mid` AND `truncatedsvd*`.
- Root cause chain:
  1. `metadata["columns"]` was set at `_phase_helpers.py:1130` to the post-pipeline + post-extensions train_df cols (`[truncatedsvd0..9]` after `apply_preprocessing_extensions`).
  2. At predict, `_validate_input_columns_against_metadata` filtered the raw user input against that list, marking every raw col as "extra" and dropping them via `df[filter_existing(df, columns)]`, leaving `(N, 0)`. Extensions transform then crashed with `Found array with 0 sample(s)`.
  3. Even after fixing the validation step, the per-model loop's expected-feature subset failed: LGB on the polars-native fastpath is fit on a HYBRID frame (raw polars-pre cols + extensions cols hstacked by `_phase_helpers.py:1082-1109`). **Fix (three-part):**
1. `_phase_helpers.py:_phase_fit_pipeline` — capture raw input cols into `metadata["input_columns"]` at the START of the phase, before any transform runs.
2. `_misc_helpers.py:_validate_input_columns_against_metadata` — prefer `metadata["input_columns"]` over `metadata["columns"]` so predict-time validation compares the input against the raw user-facing schema.
3. `core/predict.py:predict_from_models` — after `_apply_extensions_pipeline`, hstack the new extension cols onto `df_pre_pipeline` so it mirrors the training-time hybrid frame; per-model loop falls back to this hybrid when the post-pipeline `df` is missing expected raw cols.

## iter #190 — regression x lgb 1M x outlier=lof + NaN-bearing frame (seed=2026051902)

**Combo:** regression, models=lgb, n_rows=1M, axes: mrmr_fs=False, outlier=lof, ensembles=True, cat_enc=ordinal, scaler=standard, dim_reducer=none.

**Bug surfaced (wall=0.9s, status=ValueError, training crash):**
- `ValueError: Input X contains NaN. LocalOutlierFactor does not accept missing values encoded as NaN natively.`
- Root cause: `_setup_helpers.apply_outlier_detection_once` calls `outlier_detector.fit(_train_numeric)` directly at line 107 with no try/except wrapping. The mlframe OD step runs BEFORE the preprocessing pipeline's imputer, so a NaN-bearing train frame (from `_make_synthetic_frame`'s NaN injection or any caller frame with raw NaN) crashes the fit for any non-NaN-tolerant detector — LOF and OneClassSVM both raise; IsolationForest is the only sklearn outlier model that handles NaN natively on recent sklearn versions.

**Fix (framework, two locations):**
1. `_setup_helpers.apply_outlier_detection_once` lines 106-130: wrap `outlier_detector.fit(_train_numeric)` + `outlier_detector.predict(_train_numeric)` in try/except. On raise: log a descriptive error pointing the caller at the `sklearn.pipeline.Pipeline([SimpleImputer(), detector])` wrapping pattern (mirrors `test_fuzz_suite._outlier_detector_for_combo`), then early-return the unfiltered frames with `outlier_detection_result=None`. The whole suite proceeds on the unfiltered train/val frame instead of crashing.
2. Same function lines 205-217: wrap the val-side `outlier_detector.predict(_numeric_only_view(val_df))` symmetrically — train-side fit may have succeeded by then, so a val-side raise shouldn't undo prior work.

**Regression test:** `tests/training/test_outlier_detection_failure_graceful.py::test_bare_lof_with_nan_input_does_not_crash_suite` — builds a small (600-row) frame with 10% NaN in both numeric cols and passes a bare `LocalOutlierFactor(novelty=True)` (no `SimpleImputer` wrapper) to `OutlierDetectionConfig`. PRE-fix the suite re-raises the `ValueError: Input X contains NaN` from the OD step; POST-fix `train_mlframe_models_suite` returns a populated models dict + metadata (15.9s wall, lgb-only). The test asserts the suite recovered gracefully.

**Companion existing-suite verification:** 21/22 outlier-tagged tests pass; the 1 failing (`test_early_stopping_saves_time_without_auroc_loss[cb-{42,99}]`) is a timing-flaky benchmark unrelated to this fix — it asserts `time_b <= time_a * 0.90` (≥10% speedup) but observed +9.1% speedup; `auroc_delta=+0.0000` confirms no correctness regression, only timing noise. The test's own comment at line 299 ("On a loaded machine ... can be as low as ~15-25%") flags this as a known flaky benchmark assertion; pre-existing on master.

**Harness gap (separate, low-priority):** `_profile_fuzz_1m._run_suite_profiled` constructs a bare LOF (lines 343-345) without wrapping in a SimpleImputer pipeline. The framework-side fix above gracefully handles this case, but a future harness iteration could mirror `test_fuzz_suite._outlier_detector_for_combo` and wrap LOF/OCSVM. Tracked but not blocking — bare-LOF coverage is now a valid (though degraded) code path.

## iter #194 — binary x linear,ridge,sgd 1M (seed=2026051906)

**Combo:** binary, models=linear+ridge+sgd, n_rows=1M, axes: mrmr_fs=False, outlier=none, ensembles=True, cat_enc=ordinal, scaler=standard, dim_reducer=none.

**Baseline (wall=54.7s, status=OK):**
- `_compute_split_metrics` / `report_probabilistic_model_perf` -- **34.7 s (~64%)** of suite wall.
- `compute_batch_aucs` -- 32.4 s across 30 calls; `gpu_multiple_roc_auc_scores` 32.1 s in 9 calls.
- `cupy._core.sum` 21.8 s; `cupy.cuda.compiler._compile_with_cache_cuda` 11.7 s.
- `_compute_oof_preds` (cross_val_predict for ensembling) -- 9.4 s.

**Root cause:** `_GPU_BATCH_THRESHOLD_M = 1` (`mlframe/metrics/core.py:460`) dispatched GPU for every binary single-target ROC/PR AUC call. The `gpu_multiple_roc_auc_scores` / `gpu_multiple_pr_auc_scores` docstrings already prescribed "Use when N >= 100k AND M >= 5" because the GPU path carries ~1-3 s of cupy compile + host<->device transfer overhead per call plus a Python loop over M columns. At M=1 on N=1M the numba CPU `fast_aucs` is ~10x faster, but the auto-dispatch policy contradicted the documented guidance.

**Fix:** `mlframe/metrics/core.py:460` -- bump `_GPU_BATCH_THRESHOLD_M` default from 1 to 5. Single-target binary callers (M=1) stay on the numba CPU path; multi-target / multiclass with K>=5 columns still dispatch GPU.

**After fix:**
- Cold (numba kernel cache miss; first-ever fast_aucs invocation in the process): wall **70.3 s** (52 s in numba `_compile_for_args` for `fast_numba_aucs` + dependent kernels). Single-time cost amortised via `cache=True` in `NUMBA_NJIT_PARAMS`.
- Warm (numba kernel cache hit; reflects production steady-state once the process or `__pycache__` has the compiled binary): wall **18.0 s** vs 54.7 s baseline. `_compute_split_metrics` 34.7 s -> **2.1 s (-94%)**. Total **-36.7 s (-67%)**.

**Regression tests:** `tests/metrics/test_compute_batch_aucs_threshold_m_one.py` -- two tests: (a) asserts `_resolve_backend(None, 1M, 1)` and `_resolve_backend(None, 5M, 1)` both return False so the threshold can't silently regress to 1 in a later refactor; (b) end-to-end `compute_batch_aucs(N=200k, M=1)` matches `sklearn.metrics.roc_auc_score` + `average_precision_score` bit-for-bit (abs=1e-9). Both PASS in 12.9 s. The pre-fix threshold=1 would have failed test (a) with an assertion error pointing at the iter#194 root cause.

**Companion verification:** `tests/training/test_evaluation.py` -- 35/35 pass post-fix; no regression in metric correctness or report wiring.

## iter #195 — multiclass x lgb,hgb 1M (seed=2026051907)

**Combo:** multiclass, models=lgb+hgb, n_rows=1M, axes: mrmr_fs=False, outlier=none, ensembles=False, cat_enc=ordinal, scaler=standard, dim_reducer=none.

**Baseline (wall=187.5s train, 17.6s predict, status=OK):**
- `HistGradientBoosting.fit` -- 137 s (library-bound).
- `_compute_oof_preds` -> `cross_val_predict` -- **124 s (~66%)** of train wall. Fires DESPITE `use_mlframe_ensembles=False` because `trainer.train_and_evaluate_model(oof_n_splits=5)` is the hardcoded default; no plumbing gates it on whether the suite or stacker actually consumes the OOF.
- HGB grower.split_next -- 54 s; LGB engine.train -- 29.5 s.

**Root cause:** `trainer.py:439` -- `oof_n_splits: int = 5` ran a 5-fold cross_val_predict for every model regardless of caller intent. The OOF preds are only consumed by `score_ensemble` at `max_ensembling_level >= 2` (level-1 stacking) and as an opt-in calibration source; at `max_ensembling_level=1` (default) the aggregator already tolerates missing OOF and falls back to in-sample train preds (`ensembling.py:1097-1098`). For `use_mlframe_ensembles=False` runs the OOF compute is pure waste.

**Fix:** `trainer.py:439` -- change default from `oof_n_splits=5` to `oof_n_splits=0`. Opt-in only. Default behaviour: no K-fold OOF compute. Callers needing OOF for level-2 stacking (`score_ensemble(max_ensembling_level >= 2)`) or for OOF-preferred calibration MUST pass `oof_n_splits>=2` explicitly.

**Compat note:** `score_ensemble` at `max_ensembling_level=1` (default, what mlframe ensembling uses) tolerates missing OOF and falls back to in-sample train preds (`ensembling.py:1097-1098`). The two re-entrant `train_and_evaluate_model` calls inside ensembling.py (lines 1275, 1399) currently rely on the default; with the flip they will not stamp OOF on the level-2 stacker model either. That is the correct behaviour for level-1 (no OOF needed) but level-2 stacking now requires an explicit threading of `oof_n_splits` through the stacker invocation. The level-2 path is opt-in via `max_ensembling_level >= 2`; the existing guard at `ensembling.py:1524` raises a clear error when OOF is missing and level >= 2 is requested. A follow-up plumbing pass will thread `oof_n_splits` through `score_ensemble` -> internal `train_and_evaluate_model` when `max_ensembling_level >= 2`.

**Measured impact (re-profile, same seed):** train wall **187.5 s -> 52.0 s = -135.5 s (-72%)**; predict 17.6 s -> 14.5 s. `_compute_oof_preds` no longer fires (was 124 s of train wall). Ensembles=True / level-1 path is unaffected (`score_ensemble` train-preds fallback). Level-2 stacking is the ONLY consumer that loses functionality, and it is gated behind the rare `max_ensembling_level >= 2` flag with a clear error message pointing at the new explicit-opt-in contract.

**Companion verification:** 40/40 tests pass across `test_evaluation.py` (35), `test_predict_from_models_iter80_lgb_hgb_cat.py` (3), `test_predict_from_models_extensions_dim_reducer.py` (1), `test_outlier_detection_failure_graceful.py` (1).
