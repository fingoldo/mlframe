# Changelog

All notable user-facing changes to **mlframe** are documented here.

The format is based on [Keep a Changelog](https://keepachangelog.com/en/1.1.0/),
and this project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

This file is intentionally lean and user-focused. For the full engineering
record (per-commit kernel tuning, profiling numbers, audit notes) see the git
history.

## [Unreleased]

### Fixed

- MRMR's raw-feature floor-drop protection (`_fit_impl_core.py`'s `_rp_r2`) re-solved `np.linalg.lstsq` on `[base | one extra column]` from scratch for every candidate raw feature checked, even though the base design (already-selected columns) is IDENTICAL across all candidates in one protection pass -- at production scale (p~120 base columns, n_tr~53k rows, ~109 candidates) this cost 36.2s of a 100k-row run's wall time, the single largest unaddressed `np.linalg.lstsq` contributor. Fixed by computing the base design's QR decomposition ONCE and extending it per candidate via `scipy.linalg.qr_insert` (an O(n*p) column-insert update) instead of a fresh O(n*p^2) solve -- mathematically equivalent to the original per-candidate lstsq (verified: max coefficient difference ~6e-17, machine-epsilon-level agreement, not an approximation). Measured 20.36x faster at production shape on synthetic data of that shape (91s -> 4.5s for 109 candidates); 200/200 R^2-threshold accept/reject decisions matched across a stress test including near-collinear candidates. Falls back to `-inf` (reject) on genuine rank-deficiency (an exact or near-machine-epsilon duplicate of an existing base column), matching the practical outcome of the original SVD-based lstsq for a candidate that carries zero new information.

- `tests/feature_selection/mrmr/biz_val/test_biz_value_mrmr_underselection.py`'s `_support_names` test helper used `list(getattr(m, "feature_names_in_", []) or [])`, which raises `ValueError: The truth value of an array with more than one element is ambiguous` whenever `feature_names_in_` is a non-empty array (i.e. on every successful `MRMR.fit()`) -- masking every sensor in the file, including its own documented "positive sensor" (`test_core_screen_selects_signal_with_fe_off`), behind a spurious infrastructure crash instead of their real assertions. Fixed with an explicit `is not None` check instead of `or`. With the crash gone, all 22 sensors in the file now pass -- Fix A1 (fallback fires on 0-raw), Fix A2 (floor never empties), and Fix B (wide-pool MI debiasing), the three production bugs this file's sensors were written to track, have evidently already landed via earlier work in this codebase; the broken helper had been hiding that they were fixed. Updated the file's docstring (previously "EXPECTED TO FAIL until Fix A1/A2/B land") to reflect the current all-green state.

- `score_prospective_pairs` (`_step_pairs_rank.py`) called `usability_operand_continuous` twice per candidate pair to extract each raw operand's continuous values, but a raw column appears in O(n_candidates) pairs, so the SAME column was re-extracted from `X` (pandas getitem + dtype cast + ravel) once per pair it participates in -- measured 170,160 calls / 3.27s cumtime on a 100k-row production profile. Fixed with a local per-call memoization dict (`_cached_operand`), safe because `score_prospective_pairs` runs serially (no threading/loky dispatch at this level, unlike the numba-typed-dict MI caches elsewhere in this package that DO cross worker threads and need explicit per-worker copies). Pure call-count reduction -- returns byte-identical values, selection unchanged.

- Validated the cumulative effect of the `_rp_r2` QR fix and the `usability_operand_continuous` cache together on a full 100k-row production run: `np.linalg.lstsq` tottime dropped 52.7s -> 4.8s (call count 2192 -> 2012, the remainder from unrelated lstsq sites), `usability_operand_continuous` no longer appears in the top-25 by tottime at all. Wall-clock landed at 794s -- matching the best pre-session baseline (798s) despite this run also carrying every GPU fix from this session (which independently improved GPU kernel time 2.8x and CUDA-API time 2.1x, confirmed via nsys/ncu). MAE/RMSE/MaxError/R^2 identical to every prior round.

- `mlframe.metrics.core.prewarm_numba_cache` eagerly JIT-compiles both the serial and parallel (`_par`) numba variant of every metric it covers on process start, costing ~50-60s per fresh process regardless of numba's disk cache (`cache=True`) -- confirmed via three separate fresh-process measurements (~51-53s each, no improvement from a supposedly-warm on-disk cache) that this is inherent `parallel=True` compile/link overhead, not a caching bug. The size-gated dispatchers (`_PARALLEL_REDUCTION_THRESHOLD=100_000`) only ever call the `_par` variant once a fold reaches 100k rows, so for any run whose data is known to stay below that threshold, warming the `_par` variants is pure overhead with zero runtime payoff. Added `MLFRAME_NUMBA_WARMUP_SKIP_PARALLEL=1` (opt-in, default OFF) to skip only the individual `_par` warmup calls (the interleaved `_seq` calls are never skipped); measured ~6-10s faster warmup with the flag on. `wellbore_train.py` now sets this automatically when `TARGET_ROWS` is below mlframe's own `_PARALLEL_REDUCTION_THRESHOLD` (imported directly, not duplicated as a hardcoded constant). A `_par` kernel skipped at warmup still compiles correctly and returns identical results on its first real (lazy) call -- the flag only changes when compilation happens, never whether it works.

- MRMR FE screen-subsample: the classification stratified subsampler could return ~n rows (one per class) for a very high-cardinality / near-continuous target, silently defeating the 30k screen subsample and running the MI/CMI screen at full n (~33x slower at n~1M). It now falls back to a uniform draw of the requested size when the per-class floor cannot fit the budget; genuine low-cardinality classification still stratifies (rare classes preserved). The bare `except` fallbacks that make the whole screen silently run at full n now log a WARNING so the regression is diagnosable instead of an invisible slowdown.

- `kernel_tuning_cache` dispatch helpers (`lookup_joint_hist`, `lookup_mi_classif_backend`, `lookup_fe_mi_split_backend`, `fe_rung_keep_frac`'s dispatcher): the default (`run_auto_tune=False`) path routed through `get_or_tune` with a permanent no-op tuner, which unconditionally logged a misleading "sweep starting" INFO banner on every cache miss and, for `fe_rung_keep_frac` specifically, spawned a background async-sweep thread on every fresh MRMR fit that could never populate the cache (no tuner is registered for that kernel). The default path now uses a pure cache lookup instead (mirroring the existing `lookup_pairwise_corr_backend` / composite `_ktc_dispatch._lookup_backend` pattern), preserving the same stale-entry safety via an explicit `code_version_stale()` check; `get_or_tune` remains only for the explicit offline-sweep (`run_auto_tune=True`) path, now with `async_sweep=True` so a future real tuner can never block a live fit. Also fixed `lookup_pairwise_corr_backend` passing its size dims wrapped in a `dims=` kwarg instead of unpacked (`KernelTuningCache.lookup` takes `**dims`), which silently made every cached region match regardless of actual (p, n) and always returned the first region.

- `dispatch_batch_pair_mi` (batch pair-MI precompute) and `discretize_2d_array` (MRMR's discretization step, runs earlier in `categorize_dataset`) were the two remaining GPU dispatch sites in the FE package without a VRAM budget check before uploading data to the device. At production scale (millions of rows) the upload alone can consume a small card's entire VRAM; on Windows/WDDM an oversized upload can transparently over-subscribe device memory via host-paging instead of raising a catchable CUDA OOM, so the kernel launch then grinds through PCIe-paged memory for minutes before the OS silently kills the process (no Python exception, no traceback, no Windows Event Log entry -- reproduced live on a 2.4M-row/423-column production fit). Both are now gated by the same `_fe_gpu_vram.fe_gpu_has_vram_cushion` absolute-cushion check every other GPU-FE dispatch site already uses, applied even to an explicit `force_backend="cuda"`/`"cupy"` request. A rejection is never silent: it logs the full sizing context (rows, columns, pairs, requested GB, free/total VRAM) at WARNING. Also widened `dispatch_batch_pair_mi`'s CUDA except clause from `(ValueError, RuntimeError)` to `Exception` -- numba's `CudaAPIError`/`CudaDriverError` derive directly from `Exception`, so a genuine CUDA driver fault used to skip the fallback handler entirely.

- `batch_pair_mi_cuda_row_chunked`'s histogram accumulator was sized by the FULL `n_pairs` passed to one call regardless of row-chunking, so a wide candidate pool (real case: 86,736 pairs x max_joint=441 x n_classes_y=20) needed a ~6GB accumulator -- bigger than the entire 4GB card -- no matter how small the row-chunk was. cProfile on a 100k-row production run showed the likely-oversubscribed allocation thrashing: 165 `to_device` calls averaging 4.6s each (an isolated same-shape microbench shows 1-30ms), 772s of a 1633s wall, the single largest hotspot in the run. Fixed by adding a second, independent chunking dimension (`_choose_pair_subchunk_rows`): pair-subchunks as the outer loop, row-chunks as the inner, each with its own dedicated VRAM budget fraction. Measured on the exact failing shape: 772s-scale pathology -> ~40s, correctness unchanged (bit-identical to the CPU reference).

- `wellbore_train.py`'s own `_HWSampler` diagnostic thread crashed a live 3M-row validation run with repeated `Windows fatal exception: access violation` (Windows' equivalent of SIGSEGV) -- every occurrence coincided with this thread's in-process NVML call (`pynvml.nvmlDeviceGetUtilizationRates`) racing the main thread's concurrent CUDA work. NVML and CUDA driver API calls from different OS threads are not safe to interleave on this WDDM setup, and since it's a native fault, no Python `except Exception:` can catch it -- the only fix is not making the call while real GPU work may be in flight. GPU/NVML polling is now off by default (`WELLBORE_HW_SAMPLE_GPU=1` to opt back in for isolated profiling runs); CPU-only `psutil` sampling is unaffected and stays on.

- `batch_pair_mi_cuda_row_chunked`'s two independent chunk-size choosers (row-chunk rows, pair-subchunk rows) each clamp to a safe-but-degenerate floor when free VRAM is itself near-zero (a real 1M-row run hit `free_vram=0.00GB` after a cupy pool cap + other resident allocations) -- individually correct (neither ever over-allocates), but their PRODUCT can reach tens of millions of kernel launches (measured live: 796 row-chunks x 89,676 pair-subchunks = ~71M), each paying real per-launch upload/dispatch overhead that would take many hours for work the CPU kernel finishes in seconds. Now bounded by a total-launch-count sanity check (`_MAX_REASONABLE_ROW_CHUNK_LAUNCHES=2000`): past that ceiling the function raises immediately instead of grinding, so `dispatch_batch_pair_mi`'s existing CPU-njit fallback takes over right away.

- `MRMR.transform`'s chained-engineered-recipe replay (`_append_engineered`) called pandas `.assign()` once PER RECIPE inside its dependency-resolution loop. `.assign()` always returns a full block-manager copy, so replaying a realistic ~100-recipe candidate pool re-copied the (growing) working frame ~100 times per transform call -- cProfile on a 100k-row run attributed 16,204 block-copy calls / 36.5s to this pattern, the single largest `.copy()` contributor in the run. Fixed by taking one private copy of the working frame before the loop (preserving the existing never-mutate-the-caller's-X contract) and using cheap in-place column assignment inside the loop instead. Measured 57.7x faster at production-representative scale (99,401 rows / 500 base columns / 103 recipes: 14.5s -> 0.25s), output unchanged.

- `batch_pair_mi_cuda_row_chunked` zeroed each pair-subchunk's histogram accumulator by allocating a full host-side `np.zeros` buffer the same size as the device array (up to ~400MB at production shape) and shipping it over PCIe, just to zero a buffer whose content is immediately discarded. nsys profiling on a 100k-row run showed the accumulate kernel at 71.8% of GPU kernel time (17 launches, 35ms-3.96s each, huge variance) and the accumulator's later device-to-host readback (`cuMemcpyDtoH_v2`) at 55% of CUDA-API time (51 calls, 266ms avg, 4.16s max, against an isolated ~1-30ms baseline) -- consistent with the same WDDM VRAM-oversubscription paging hazard documented above, with the redundant zero-upload doubling transient memory pressure at exactly the allocation moment most likely to trigger it. Fixed via `_new_zeroed_device_array`: zeroes device-side instead (cupy's `cp.zeros`, a device-side `cudaMemsetAsync` with no host transfer, when cupy is available; a trivial numba.cuda zero-fill kernel otherwise) -- measured 60.78x faster than the host-upload pattern in isolation at the production shape (254ms -> 4.18ms per accumulator pair, ~4s saved per full `batch_pair_mi_cuda_row_chunked` call), bit-identical results confirmed on real hardware through both the cupy and numba-only code paths.

- `batch_pair_mi_cuda_row_chunked`'s histogram-accumulate kernel used plain GLOBAL-memory atomics (2 atomic adds per row processed per thread). ncu profiling (admin-elevated to get past the default `ERR_NVGPUCTRPERM` permission block) on the same 100k-row shape showed 99.22% occupancy but only 2.53% compute throughput, with Warp State Statistics attributing 93.5% of the average 703.7-cycle stall between issued instructions to "LG Throttle" (the L1 instruction queue for local/global memory operations staying full) -- ncu's own guidance: "avoid redundant global memory accesses... combine multiple lower-width memory operations into fewer wider memory operations". Fixed by mirroring what the non-chunked kernel (`_cuda_kernel_factory`) already does: a new shared-memory-staged kernel variant (`_cuda_hist_kernel_shared_factory`) accumulates each block's contribution in a per-block dynamic shared-memory buffer (int32, on-chip) and flushes to the persistent global int64 accumulator once per histogram cell, cutting global atomic traffic from O(chunk_rows) to O(max_joint*n_classes_y) per block (~17x fewer at the profiled shape). Gated by `_hist_kernel_shared_fits_budget` on the device's actual per-block shared-memory budget -- falls back to the original global-atomics kernel when the histogram is too large to stage, so this never risks a launch failure or an over-allocation. Measured 1.34x faster end-to-end at the production shape (2.157s -> 1.613s for one full pair-subchunk), bit-identical to both the CPU reference and the global-only kernel across pair- and row-subchunk boundaries.

- MRMR audit finding #5 ("unresolved" doubt whether `_screen_predictors.py`'s `backend="threading"` joblib dispatch actually benefits from the njit kernels it wraps "releasing the GIL") is now resolved with a direct measurement: `compute_mi_from_classes`, `shuffle_arr`, and `parallel_mi` were decorated `@njit(cache=True)` WITHOUT `nogil=True`, despite a code comment claiming they "release the GIL, so threading gives near-process-pool speedup". Measured threading-backend dispatch through `compute_mi_from_classes` at 0.84-0.94x vs a serial loop (worse than serial -- pure GIL-serialized overhead) at n=30000/8 concurrent tasks. Added `nogil=True` to `compute_mi_from_classes`, `compute_mi_mm_from_classes`, `_mm_bias`, `shuffle_arr`, `shuffle_arr_lcg`'s kernel, `parallel_mi`, and `parallel_mi_with_null` (all pure-numeric numpy-array kernels with no Python-object access -- safe per the same pattern already used by 269 other `nogil=True` sites in this codebase, including these functions' own siblings `compute_su_from_classes`/`compute_relevance_score` in the same file, which already had it). Measured 1.55-1.91x real speedup at the same shape post-fix. Thread-safety of numba's internal `np.random` state under genuinely concurrent nogil execution verified directly (8 threads x 200 shuffles, every output a valid permutation).

- Found (via the above investigation's more thorough concurrent-execution testing) and fixed a separate, pre-existing bug in `shuffle_arr_lcg`: numba boxes a `uint64` return value as a plain Python `int`, not an `np.uint64` scalar. The function's docstring documents chaining its returned state across calls (`state = shuffle_arr_lcg(arr, state)`), but re-passing that bare `int` into another njit call made numba infer `int64` (a raw Python int carries no dtype) -- which fails to unbox with `OverflowError: int too big to convert` as soon as the LCG state's high bit sets (typically within the first 1-5 calls of any seed, confirmed independent of threading -- reproduces identically single-threaded, and pre-dates this session's `nogil=True` addition). Fixed by wrapping the kernel (renamed `_shuffle_arr_lcg_kernel`) in a thin Python function that casts both the input and the returned value through `np.uint64(...)` at the Python level, which numba always unboxes correctly regardless of magnitude. Verified over 5000 chained calls with zero failures (pre-fix: failed within 1-5 calls for nearly every seed tried). No production caller currently chains state through this function (only a single fixed-seed warmup call in `_prewarm.py`), so this was latent, not yet user-visible -- but the fix makes the documented public contract actually true.

- Both of the above now try a ROW-CHUNKED GPU path before dropping all the way to CPU when the full upload doesn't fit VRAM, instead of losing the GPU speed win outright. `batch_pair_mi_cuda_row_chunked` uploads `factors_data` in VRAM-sized row-blocks and accumulates the joint/marginal histogram across them (counts are additive over rows; the finalize reduction uses the identical formula/order as the non-chunked kernel, so results match to within a single-ULP libm rounding difference -- measured max abs diff ~1e-17 on real hardware). `discretize_2d_array_cuda_row_chunked`'s `uniform` method is similarly EXACT (column min/max are genuinely reducible across row-chunks); its `quantile` method computes bin edges from a GPU-resident random subsample (default 500k rows) instead, since exact quantiles need the full column's order statistics -- this is an approximation by construction, matching the project's documented FE/MRMR exception (binning speed levers are held to selection-equivalence, not bit-identical MI); measured >99% code match against the full computation at 1.5M rows. Both row-chunked paths were measured faster than the CPU fallback at production scale (pair-MI: 4.5x at 1.5M rows/60 cols; discretize: 14.7x at 1.5M rows/40 cols, plus the subsample actually makes quantile-edge computation itself cheaper than the full-array percentile).

- `run_polynom_pair_fe`'s loky worker pool (`LokyBackend(inner_max_num_threads=1, initializer=disable_cuda_in_worker)`) had no earlier user in a typical fit (the sibling CPU pair-MI-sweep pool in `_step_pairmi.py` only engages when the GPU MI path fails), so it paid a full cold start -- 16 fresh Python processes each re-importing mlframe/numba/cupy -- synchronously inside the polynom-pair-FE phase every single fit. Root-caused via `py-spy` attached to a live production run: the round-11 cProfile's 71.8s of `_pickle.dumps` tottime with only 6.6% caller attribution turned out to be real work on a genuinely different OS thread (`QueueFeederThread`, continuously `active+gil`) that cProfile (main-thread-only by default) can't attribute cleanly -- not the cross-thread accounting artifact it superficially resembled. An isolated A/B on the real (79237, 544) production shape then isolated the actual cost: a cold pool took 28.1s, the identical pool warm took 0.7s -- a 40x gap dominated by process spawn + reimport, not by pickling per se. Added `maybe_prewarm_polynom_loky_pool` (`_joblib_safe.py`), called from `MRMR._fit_impl` -- see the two follow-up fixes below for where/how, both found via a REAL production A/B, not assumed correct from the isolated microbenchmark alone.
  - First version called this immediately after `fe_smart_polynom_iters`/`n_jobs` are read (fit-entry, well before categorization), reasoning that categorization + GPU pair-MI screening would leave CPU idle for it to overlap with. A full 100k-row production A/B (round 12) PROVED this wrong: wall-clock REGRESSED from 712s to 935s (+223s). Two compounding bugs, both found via `[HW] span=`/`avg_cpu=`/idle-gap telemetry the run already logs: (1) categorization is itself CPU-active, not idle -- the pre-warm's 16 concurrent worker-process spawns contended with it (the categorization-to-GPU-screen gap grew 85.3s -> 153.1s, with exactly 16 new `NumbaPerformanceWarning: Grid size 1` lines appearing in that window, matching the 16 workers bootstrapping). Fixed by moving the call site to right after `categorize_dataset` completes, where the next phase (GPU pair-MI screening) is genuinely GPU-bound (blocks on `.get()`/`copy_to_host`) and leaves CPU actually free. (2) loky's default `idle_worker_timeout=300` is shorter than the real gap to `run_polynom_pair_fe`'s actual dispatch (measured ~418s with the fit-entry call site) -- the pre-warmed pool self-terminated before it was ever used, paying the contention cost above for zero benefit. Fixed by adding `POLYNOM_LOKY_IDLE_WORKER_TIMEOUT` (`_joblib_safe.py`), passed on BOTH the pre-warm's `LokyBackend` and `run_polynom_pair_fe`'s (they must match for `get_reusable_executor`'s reuse-key to hit the same pool).
  - Round 13 (both fixes above, timeout=900s): wall-clock improved to 862s (still +150s vs the 712s baseline) despite the polynom-pair-FE phase itself genuinely being faster (125.4s -> 117.4s, only an ~8s win once real Optuna/CMA-ES trial work -- not pool startup -- was confirmed to dominate the phase's wall-clock). Root cause: the relocated call site shrank the real pre-warm-to-dispatch gap to ~236s, but the 900s timeout (sized against the STALE 418s figure) let the 16 idle workers linger for up to 900s past their one use, contending with the REST of the fit (second target's categorization/training etc.) for far longer than necessary.
  - Round 14 (timeout reduced to 450s, comfortably above the ~236-274s measured gap while roughly halving idle-lingering time): wall-clock 753.4s (+41.4s vs the 712s baseline, down from +223s and +150s in the two prior rounds), polynom-pair-FE phase 103.0s -- the best of all 14 rounds this session, 22.4s faster than the original no-prewarm baseline. `MAE=6.03 RMSE=8.50 MaxError=26.06 R2=1.00` identical to all 14 rounds. Net effect now close to break-even (~5% overhead in a complex multi-phase run, within typical run-to-run variance) while delivering a real, consistent win on the phase this fix specifically targets -- kept as-is; further timeout tuning would chase a small remaining margin with diminishing returns.

- `abs_pearson` (`_fe_usability_signal.py`) showed 60.9s tottime / 61.8s cumtime across ~767k calls on a 100k-row production cProfile -- read initially (from the tottime/cumtime split alone) as "the `np.asarray`/dtype-branch/`np.ascontiguousarray` wrapper checks cost ~79us/call, the njit kernel itself only ~1.2us". Direct A/B microbenchmarking disproved that read: a no-validation internal fast path (`_abs_pearson_fast`, used by `usability_form_corrs`'s 9 internal calls per pair, which can guarantee already-contiguous same-dtype input by construction) is only ~3.8% faster (58.2us/call vs 60.5us/call at the real n=30000 post-subsample shape) -- the wrapper checks were never the bulk of the cost. What actually happened: numba dispatcher calls apparently don't get their own separate frame in cProfile's stats here, so the njit kernel's REAL two-pass float64 reduction cost (~58us at n=30000) was silently folded into the caller's tottime. Kept `_abs_pearson_fast` anyway as a free, bit-identical ~3.8% win (11 tests, `tests/feature_selection/filters/test_abs_pearson_njit.py`). Investigated (and rejected) two bigger swings at the real dominant cost: a batched-9-forms-in-one-kernel-call variant measured 1.5x SLOWER (the per-row inner loop over forms defeats the original kernel's SIMD/reassoc vectorisation -- independently reproduces an already-documented 2026-07 rejection of the same idea); a numerically-stable-looking shifted one-pass formula measured 2.19x faster but reproduced the exact catastrophic-cancellation failure (`|corr|` ~0.029 instead of ~0.0 on a near-constant column) the current two-pass mean-then-center design was written to fix -- UNSAFE, not shipped. Both attempts + numbers documented in `_benchmarks/bench_abs_pearson_fastmath.py` and `_abs_pearson_fast`'s docstring.

- `usability_form_corrs` (`_fe_usability_signal.py`) recomputed its SINGLE-operand `|corr|` baseline (`_cs` -- the max of an operand's raw value and its square, each correlated against y) from scratch on EVERY candidate pair, even though a small pool of raw operands recurs across a much larger pool of pairs -- the exact same redundancy shape as the `usability_operand_continuous` extraction cache fixed earlier in this file's history, just one layer further down the pipeline. `pair_is_tail_concentrated_rankaware` (the caller, ~85k calls in a wide 100k-row production fit) showed up at 105.6s cumtime in a round-14 profile, largely driven by this. Decomposed the single-operand half into its own function, `_single_operand_usability_corr(y, x)` (`max(|corr(y,x)|, |corr(y,x**2)|)`), mathematically identical to `usability_form_corrs`'s internal computation (verified bit-identical: `max` over 4 values via nested vs flat grouping is exact for float comparisons, no accumulation involved), and added an optional `precomputed_single_corr` parameter to both `usability_form_corrs` and `pair_is_tail_concentrated_rankaware` so a caller with a per-operand cache can skip the redundant half entirely. Wired a `_cached_single_corr` memoization (mirroring the existing `_cached_operand` pattern -- safe for the same reason: `score_prospective_pairs` runs serially, no threading/loky dispatch at this level) into `score_prospective_pairs`'s hot call site. Measured 1.51x at a representative production-shaped synthetic benchmark (120 operands, ~5000 pairs, n=30000 post-subsample: 786.7us/pair -> 519.5us/pair). 22 new/updated tests (bit-identity of the decomposition, cache-avoids-redundant-computation, end-to-end decision equivalence with and without the cache) plus the existing operand-cache tests, all passing; mypy clean.

- Investigated replacing `run_polynom_pair_fe`'s loky-based per-pair dispatch with the existing (but completely untested and never wired in) `_numba_polynom_optimizer.py` -- a full `@njit(parallel=True)` port of the `random_batch` search strategy that `prange`s over ALL pairs in one kernel call (no GIL, no worker spawn, no pickling, no memmap; its own docstring says "this is the path the polynom-pair FE dispatch should call... to fully eliminate joblib"). Added the missing test suite it shipped without (`tests/feature_selection/filters/test_numba_polynom_optimizer.py`, 7 tests): recovers a strong synthetic signal, the single-pair and multi-pair (P=1 vs batched) entry points agree, `prange` thread-isolation confirmed via a direct low-level-kernel comparison with manually-sliced identical RNG streams (no cross-thread aliasing), run-to-run reproducibility (no races), degenerate-input robustness, and comparable solution quality to the existing `_run_random_batch_search` it mirrors. All pass -- the kernel itself is now genuinely trustworthy. NOT wired into `run_polynom_pair_fe` yet: wellbore's actual (default, unoverridden) config uses `fe_multi_fidelity=True`, which this kernel's docstring explicitly lists as unsupported ("eval_pair_fn closures (multi-fidelity...) NOT supported -- the kernel uses the full dataset for every evaluation"). Multi-fidelity's inner search subsample is 1500 rows vs the outer `UNIFIED_FE_SUBSAMPLE_N=30_000` -- switching optimizers would mean every trial runs on 20x more data, which could offset or reverse the joblib-avoidance win and needs a real A/B measurement, not an assumption, before it can safely become the default. FUTURE: either add multi-fidelity (subsample-then-refine) support to the kernel, or run the A/B with multi-fidelity disabled to quantify the selection-quality/wall-clock tradeoff.

- `fe_gpu_strict_enabled`'s AUTO (unset `MLFRAME_FE_GPU_STRICT`) default gated ONLY on the fit's row count (`_auto_min_n()`, 100k), ignoring column count entirely -- a wide-but-under-100k-rows production fit (real shape: 79,237 rows x 544 columns, ~43M total elements, comfortably clearing the SAME `n*p>=1_000_000 and p>=64` work floor a per-call dispatch already uses) stayed on the exact CPU path for every FE-family dispatch this whole session purely for sitting under an arbitrary row mark -- despite the codebase's own docstrings claiming STRICT is validated selection-equivalent from ~50k rows already (100k was an extra conservative margin, not the actual proven safety boundary). Added `_auto_gate_passes` (`_fe_gpu_strict.py`): engages when EITHER the pure row rule holds (unchanged -- never narrows existing behavior) OR the fit is at/above the validated ~50k floor AND its total (row, column) work clears the per-call floor. `set_auto_fit_n` now accepts an optional column count, threaded from `MRMR.fit`'s entry (`_mrmr_class.py`). 23 tests (16 existing + 7 new), mypy clean. ### Changed

At the default 30k screen-subsample size the `prange` loop beats the serial loop 2.4-7.8x for every candidate count `p>=4`, so small candidate pools no longer run single-core. Both branches are exact CMI (selection-equivalent).

- MRMR GPU-resident FE (`MLFRAME_FE_GPU_STRICT`) now has an AUTO size-gated default: on fits at/above `MLFRAME_FE_GPU_STRICT_AUTO_MIN_N` rows (default 100 000, the production regime) with a usable CUDA device, STRICT engages automatically (~2.5x faster FE and measured selection-equivalent to the CPU path at that scale). Below the threshold, or with no GPU, the exact CPU path runs unchanged (byte-identical legacy). Set `MLFRAME_FE_GPU_STRICT=0` to pin the CPU path at any n, or `=1` to force STRICT. The small-n divergence that keeps STRICT gated below the threshold is finite-sample MI-estimation variance (features with near-tied relevance), which fades as n grows — verified converged across scenarios by ~50k; the 100k default sits comfortably above that.

### Added

- Recency-weighting primitives (`mlframe.core.recency_weights`): parametric poly/exp/power weight vectors over ordered histories, generalizing the exponential `ewma`. Identity parameters reproduce uniform weights.
- Per-entity recency-weighted feature primitives (`mlframe.feature_engineering`): `per_group_recency_weighted_mean` (weighted mean / event-rate), `per_group_recency_weighted_mode` (weighted Parzen-density mode), and `per_group_behavioral_stability` (density-peak-height predictability score). Opt-in; identity params match the plain unweighted mean. Parallel (`prange`) KDE kernel dispatched by entity count.
- Nadaraya-Watson kernel regression / smoothing (`mlframe.feature_engineering.nadaraya_watson`): `nadaraya_watson_smooth` (flat signal, gaussian/epanechnikov/boxcar/tricube kernels, optional per-sample weight composable with recency weights) and `per_group_nadaraya_watson_smooth` (per-entity denoising). Parallel query kernel dispatched by query count.
- Robust location estimators (`mlframe.core.robust_location`): `robust_mean_mestimator` (redescending Meshalkin/Huber/Tukey M-estimator with MAD scale, prange-dispatched at large n), `geometric_median` (Weiszfeld spatial median), and `trimmed_mean` / `winsorized_mean` (classic tail-robust location). Robust aggregators for contaminated data.
- Proportion statistics (`mlframe.core.proportion_stats`): `wilson_interval`, `required_n_for_proportion`, `proportions_significantly_different`, `z_for_confidence` — confidence intervals and sample-size planning for probability/rate estimates.
- Spectral matrix seriation (`mlframe.core.matrix_seriation`): `spectral_seriation` / `seriate` reorder a similarity/correlation matrix (Fiedler or leading-singular-vector) to surface block structure — for readable correlation heatmaps and feature-block detection.
- Benchmark-relative & threshold regression metrics (`mlframe.metrics.regression`): `fast_epsilon_band_accuracy` (fraction within ±ε, the dunnhumby acceptance functional), `fast_rel_mae` / `fast_mrae` / `fast_percent_better` (error relative to an arbitrary benchmark prediction), and `fast_logcosh_loss` (overflow-safe smooth loss).
- Optimal decision-threshold search (`mlframe.metrics.classification.optimal_threshold`): given scores and binary labels, find the threshold maximizing `f1` / `balanced_accuracy` / `mcc` / `youden` / `accuracy` via one O(n log n) confusion-count sweep — a per-functional operating-point picker (the F1-optimal and BA-optimal cuts differ under class imbalance).
- Single-population targeting curves (`mlframe.metrics.classification`): `cumulative_gains_curve` (CAP/Lorenz), `lift_curve`, `gains_table` (decile marketing table), and `exploss` (exponential proper scoring rule) — for how-deep-to-target model evaluation, distinct from the per-group LTR lift in `metrics.ranking`.
- Weighted / Quadratic Weighted Kappa (`mlframe.metrics.classification.quadratic_weighted_kappa`, `weighted_kappa`): distance-weighted agreement for ordinal multiclass targets (bit-matches `sklearn.cohen_kappa_score(weights=)`).
- Set-similarity coefficients (`mlframe.core.set_similarity`): `jaccard`, `dice`, `overlap`, `braun_blanquet`, `ochiai`, `kulczynski`, `tversky` over two boolean masks or Python sets — for set/interval/cluster-pair targets.
- RMSPE (`mlframe.metrics.regression.fast_rmspe`): root-mean-square percentage error (the Rossmann metric), scale-free squared-relative error excluding zero targets.
- Optimal ordinal cutpoints (`mlframe.metrics.classification.optimal_ordinal_cutpoints`, `apply_cutpoints`): tune the thresholds that digitize a continuous prediction into ordinal grades to maximize QWK/accuracy (the CrowdFlower direct-functional-tuning technique).
- Binning-smoothing (`mlframe.core.binning`): `fit_bin_smoother` / `apply_bin_smoother` / `bin_smooth` replace each value by its bin mean/median/boundary representative on the original scale (Han & Kamber smoothing-by-binning) — a leakage-safe, rank-preserving quantizer/denoiser.
- Categorical co-occurrence SVD embedding (`mlframe.feature_engineering`): `cat_cooccurrence_svd_fit` / `apply_cat_cooccurrence_svd` / `cat_cooccurrence_svd_with_recipes` encode a categorical column by the leading singular vectors of its co-occurrence matrix with another categorical (Dyakonov's `code_factor`) — a target-free structural encoding.
- ACE feature-significance filter (`mlframe.feature_selection.ace_select`): Artificial Contrasts with Ensembles (Tuv et al. 2009) — a parametric t-test of each feature's importance against permuted-contrast importances over replicates, plus a masking-removal loop; complements Boruta's binomial hit-count test with a continuous-margin test.
- Ensemble blending primitives (`mlframe.models.ensembling`): `caruana_greedy_selection` (metric-direct greedy forward selection with replacement, optimizing the actual competition metric) and `rank_average_blend` (scale-invariant AUC-oriented rank blend) over a base-model prediction matrix.
- Random-forest proximity metric (`mlframe.models.rf_proximity`): `rf_proximity_matrix` (Breiman leaf-co-occurrence similarity from any `.apply`-capable forest), `proximity_to_distance`, and `rf_outlier_measure` (Breiman within-class proximity outlier score) — a learned metric for RF-based clustering / MDS / anomaly detection, distinct from the FE proximity-weighted aggregate.
- Graph -> tabular feature generation (`mlframe.feature_engineering`): `graph_neighbor_aggregate` (leakage-safe homophily / social-influence: aggregate a label or feature over a node's graph neighbours) and `graph_structural_features` (per-node degree / weighted strength / local clustering coefficient / triangle count), plus tabular-to-graph constructors `knn_graph_edges` (k-NN similarity graph on float columns) and `shared_attribute_edges` (affiliation graph from a categorical / group column), each with an optional `timestamp=` for directed past-only (leakage-safe temporal) edges. Lets graph structural + homophily features be built for ordinary float/categorical tables.
- Pairwise link-prediction features (`mlframe.feature_engineering.link_prediction_features`): batched common-neighbours / Jaccard / Adamic-Adar / resource-allocation / preferential-attachment for candidate node pairs (numba two-pointer intersection over CSR adjacency) — the tabular feature block for link prediction as supervised classification. Per-node importance (PageRank / HITS / centralities) remains networkx; only these batched pairwise scores are provided here.
- LENKOR composite-similarity metric learning (`mlframe.core.composite_similarity`): `fit_composite_similarity` / `combine_block_similarities` learn a metric as a coordinate-descent-tuned deformed (linear/sqrt) weighted combination of per-attribute-block similarity matrices, optimizing a leave-one-out kNN target functional (Dyakonov's ECML-PKDD-2011-winning technique) — for blending heterogeneous precomputed similarity views, distinct from Mahalanobis metric learning on raw features.
- Fuzzy-partition (POSP) membership encoding (`mlframe.feature_engineering.fuzzy_partition_encode` / `fuzzy_partition_fit` / `fuzzy_partition_transform`): fuzzification of a numeric column into `n_sets` interpretable soft-membership features whose rows sum to 1 (Ruspini partition), `triangular` (exact tents) or `gaussian` (normalized RBFs), numba njit, leakage-safe fit/transform recipe. A smooth alternative to hard one-hot binning: on smooth nonlinear targets a linear model on the soft memberships beats hard `KBinsDiscretizer` one-hot at the same number of bins.
- Per-graph spectral descriptor (`mlframe.feature_engineering.graph_spectral_features`): a compact permutation-invariant (isospectral) fingerprint of a whole graph from its edge list - n_components, algebraic connectivity (Fiedler λ2), spectral radius, graph/Laplacian energy, a bipartiteness signal (largest normalized-Laplacian eigenvalue), triangle/edge spectral moments, and the smallest normalized-Laplacian eigenvalues - for tabular graph-classification where each sample is a graph. Complements the per-node `graph_structural_features`.
- Friedman & Popescu H-statistic (`mlframe.inspection.friedman_h_statistic` / `pairwise_interaction_strength`): model-agnostic measure of how much of a feature pair's joint effect on a trained model is non-additive interaction (H = sqrt of the interaction share of the centred joint partial dependence, in [0,1]). Built on the suite's existing PDP machinery; distinct from sklearn (which has partial dependence but no interaction statistic). High-H pairs are the ones worth an explicit engineered interaction feature.
- `train_mlframe_models_suite` regression reports now also emit RMSPE (Rossmann scale-free error) and LogCosh (overflow-safe smooth loss) alongside the existing extended block, and both are registered as lower-is-better in the metric-direction table.

### Added

- `train_mlframe_models_suite` orchestration for end-to-end multi-model training, calibration, feature selection, and reporting.
