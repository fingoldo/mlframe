# MRMR Feature-Engineering — improvement & synergy backlog (2026-06-03)

Synthesis of a 4-agent ideation sweep over mlframe's MRMR FE, briefed on what
**shipped** and what was **benchmarked-and-rejected** this session. Each idea is
grounded in the actual gate/operator/estimator code (file:function cited).

> **Guiding lesson from this session:** across five rejected ideas the binding
> constraint was always the **admission / selection machinery**, NOT the richness
> of feature construction. Features were constructible but died at a gate, or were
> already implemented / subsumed. So the highest-confidence wins are the ones that
> let a *genuinely-good feature pass a gate it currently fails*. Anything that
> lowers a bar must keep the **order-2 maxT permutation-null floor** as the outer
> guard, and must debias **both** sides of any ratio consistently.

- **Feasibility.** Gate on `n_rare ≳ few hundred` (rare-imbalance-needs-large-n). Integration: `_mi_classif_batch` scoring helpers.

### 19. KSG / k-NN continuous-MI tie-breaker
- **Mechanism.** All FE-gate MIs are binned plug-in; a heavy-tailed engineered column can collapse to 2-3 bins and lose most of its MI (documented at `_feature_engineering_pairs.py:954-959`, 0.14 vs true 0.32). When the binned ratio narrowly fails (0.90-0.97 band), recompute numerator+denominator with KSG on **raw continuous** values and re-test. `sklearn.feature_selection.mutual_info_*` ships KSG (install-before-reimplement).
- **Cheap validation.** `mul(log(c),sin(d))`: binned MI ~0.14 vs KSG ~0.32; ratio passes under KSG where binned fails.
- **Feasibility.** O(n log n)/pair with a k-d tree → restrict to the margin band + top pairs. RISK: KSG noisy at small n → require agreement with binned on noise (don't *raise* the ratio for noise pairs).

### 20. Cross-fit recipe warm-start prior
- **Mechanism.** Within CV/`partial_fit` refits (folds 80%+ overlap), MRMR re-searches FE from scratch. Maintain a recipe-prior store keyed on the X-fingerprint: prior-surviving recipes seed the pool first + seed CMA-ES coefficients → fewer iters, stabler selection. Distinct from the identity-skip cache (no-FE case only) and `fe_warm_start` (within-pair CMA only).
- **Cheap validation.** 5 bootstrap subsamples; CMA iters-to-converge + wall-time cold vs warm.
- **Leak control.** A prior recipe must still pass *that fold's* gates on *that fold's* data — warm-start changes search order/init, never admission. Integration: pool-construction in `_mrmr_fe_step.py`, keyed on `_mrmr_compute_x_fingerprint`.
- **VERDICT (2026-06-10): BENCH-REJECTED, shipped OPT-IN / default-OFF.** The CMA-coefficient half was implemented as `optimise_hermite_pair(cross_fit_prior_seeds=...)` — prior joint-coefficient vectors injected as extra warm-start seeds. Probe `profiling/bench_warmstart_probe.py` on 5 (and 12) bootstrap subsamples (85% overlap, n=4000, non-monotone-inner product target, the regime where CMA must actually search): **median iters COLD=78 / WARM=79** (the prior seed adds ONE eval and saves ZERO generations); **wall −0.3%…−3.7%** (slightly slower); **selection NOT byte-identical on 8/12 folds** (6 higher-MI, 2 lower). Root cause: the per-pair ALS warm-start (`warm_start_als`, default-on since 2026-05) already re-derives the true basin from *this fold's* data each call and lands x0 there, so a cross-fold coefficient prior is strictly subsumed — and the extra seed perturbs the CMA population enough to land on a *different* optimum, failing the "identical-or-stabler selection" ship gate. The leak-control invariant itself is preserved (winner re-scored on the fold; unit test `test_xfit_warm_start_prior.py::test_seeded_path_rescored_on_this_data` proves a strong-target prior cannot manufacture an admission on a noise target). Kept opt-in per keep-all-versions; `None` default ⇒ byte-identical to legacy. Do NOT flip default-on without a regime ALS cannot seed.

### 22. Sufficient-summary early-stop (compare-to-theoretical-max via DPI residual) — ✅ SHIPPED (DEFAULT-ON, 2026-06-10)
- **Mechanism.** After each MRMR feature SELECTION (once per fit/screen pass, NOT per candidate pair), fit a CHEAP ridge of y on the SMALL selected set `E_hat[y|selected]` (1–5 cols — engineered features linearise the signal so a linear fit captures `E[y|selected]`; the design is tiny because the SELECTED set is small) and form the residual `r = y − E_hat`. STOP the FE search iff `MI(r; x_j) ≤ the maxT permutation null` for EVERY raw feature AND the residual is small relative to y (`Var(r)/Var(y) ≤ guard`, the H(y)-relative size guard). By the **Data-Processing Inequality** any future engineered candidate is a function of the raws, so it cannot carry more MI with r than the raws do → a residual that is pure noise w.r.t. the observables certifies the selection reached `I(observables; y)` (the theoretical max) and the remaining search is provably pointless.
- **Why orthogonal.** Changes WHEN the loop stops, gates untouched. Complements #16 (the rung schedule prunes the pair pool WITHIN a step; this skips ENTIRE remaining steps once the selection is already sufficient). The user's "compare-to-theoretical-max" idea realised cheaply: the maxT null is the SHIPPED `pooled_permutation_null_gain_floor`, the residual is one ridge solve.
- **Decisive design points (the user iterated to THIS exact shape).** (1) The residual is tested against the FULL SELECTED set, NOT the 1-D top feature — `y=a+b` stops only after BOTH a,b are selected, never after a alone. (2) The residual-size guard is the GRID-FREE unexplained-variance fraction `Var(r)/Var(y)` (== 1−R²), the faithful realisation of "small relative to H(y)" without the equi-frequency / fixed-grid binning artefacts that make a binned-entropy ratio read ~1.0 for ANY non-constant residual. (3) The leftover test is **MI** (not a linear score) so a NONLINEAR leftover the linear `E_hat` underfits is STILL caught — if a raw explains the residual via MI, the stop is blocked. (4) Cheap precisely because the SELECTED set is small (NOT the full pool, NOT a GBM/kNN — all explicitly rejected). (5) Ridge regularisation handles collinear/redundant selected columns.
- **OUTCOME (validated + shipped default-on).** `_fe_sufficient_summary.py` (pure helper `sufficient_summary_reached` + MRMR adapter `check_sufficient_summary_for_mrmr`), wired into the greedy FE loop in `_mrmr_fit_impl` AFTER `screen_predictors` returns `selected_vars`, BEFORE `_run_fe_step`. Ctor knobs `fe_sufficient_summary_early_stop=True` / `fe_sufficient_summary_residual_frac=0.25` / `fe_sufficient_summary_maxt_permutations=25` / `fe_sufficient_summary_maxt_quantile=0.95` / `fe_sufficient_summary_ridge_alpha=1e-3`. Verdict surfaced on `self.sufficient_summary_`; `self._fe_steps_executed_` counts FE steps run. **SELECTION UNCHANGED:** on the genuine multi-signal F1 fixture the early-stop conservatively does NOT fire (a raw still carries residual signal) → selection is BYTE-IDENTICAL with it on vs off. **WORK-CUT:** on a fully-recoverable linear signal the stop fires before any FE step (0 vs 2 FE steps off), and the feature the off run would have added carries < 1e-3 incremental R² over the kept raws (provably pointless skip). **NO FALSE STOP** on a pure-noise target (variance guard) / **NO PREMATURE STOP** with an unfound second signal. **COST 0.06% of fit** (cProfile). Triad: `tests/feature_selection/test_fe_sufficient_summary.py` (12 unit: positive a/b, negative c/d, adversarial e–i, degenerate guards) + `tests/feature_selection/test_biz_value_mrmr_sufficient_summary.py` (work-cut + byte-identity + no-false/premature-stop matrix + cProfile). Numbers: `D:/Temp/earlystop_results.md`. CHANGELOG under "Added".

## Architectural through-line & iron rules
- **Pipeline shape for the synergy ideas:** *proposer* (GBM #6 / RFF #9 / CMI-lattice #10 / gradient #21) → **II ranking #8** → existing FE search → **order-matched maxT floor (#7 for 3-way)**. Proposers shrink the candidate family from p²/p³ to K, which makes the maxT floor *less* conservative (tighter multiple-comparison correction) — so smart proposal both scales AND improves power, provided each proposer carries its own permuted-y self-gate.
- **Iron rule:** anything that LOWERS a bar (#1, #4, #5, #8) must keep the order-2 maxT floor as the outer guard and debias **both** sides of any ratio/comparison consistently (the floor at `_mrmr_fe_step.py:441,469` is on the uncorrected `batch_pair_mi_prange` scale).
- **MI-invariant operators (#14 isotonic; RankGauss):** must route through the non-MI-gated pool + downstream-lift test, never an MI-uplift gate (DPI makes them invariant — this is why supervised-knots died).
- **Dispatcher/defaults:** the fastest proposer for (n, p, HW, dtype) should be the routed default behind a dispatcher with thresholds in `kernel_tuning_cache`, not an opt-in flag.
