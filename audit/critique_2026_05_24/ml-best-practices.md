# ML Best Practices Audit ‚Äî mlframe (2026-05-24)

NOTE: Persisted from inline agent output (Plan-type subagent ran without Write tool).

## Summary

The mlframe trainer follows mature ML hygiene for the highest-risk leakage vectors: train-only fitting of preprocessing pipelines (`fit_and_transform_pipeline` snapshots train schema, transforms val/test), train-only unsupervised pre-screen with broad protection of targets/cats/text/group/ts columns, an explicit test-row leak guard in `post_calibrate_model` (intersects `calib_idx` with `test_idx` and raises), OOF predictions for level-1 stacking via `cross_val_predict`, and a documented val/test/OOF terminology contract. Several P0/P1 risks remain: drift report and pre-screen rely on numeric-only PSI/variance, the Enum-vs-Categorical guidance is violated in three production paths (`_phase_polars_fixes.py:69`, `preprocessing/transforms.py:112,123`, `core/predict.py:341`), there is no Brier/AUC bootstrap CI in the evaluation surface, RFECV defaults to plain k-fold int CV (auto-upgrade to TimeSeriesSplit only fires when a timestamps hint is threaded), and there is one bias-creep into "val-domain widening" in the Enum cast (`_phase_helpers_fit_split.py:485-501`) that is documented in code but inflates the val ES detector slightly relative to a strict train-only domain. Calibration calibrators are appropriately ordered (Sigmoid / Isotonic / Spline / Beta / VennAbers / Dirichlet) but the default `BinaryPostCalibrator` does not pin a single recommended method and no calibration plot is automatically emitted on the OOF source.

## –ê—Ä—Ö–∏—Ç–µ–∫—Ç—É—Ä–Ω–æ–µ –ø—Ä–µ–¥–ª–æ–∂–µ–Ω–∏–µ

Three directions for sustained ML-discipline improvement:

1. **Make calibration first-class, not opt-in.** Wire a `CalibrationPolicy` config (default: Isotonic on OOF when `n_oof >= 1000`, Beta on OOF otherwise, ECE bootstrap-CI tiebreak across {Sigmoid, Isotonic, Beta, Spline}); stamp the chosen calibrator + its 95% CI ECE into metadata and auto-emit a reliability diagram fitted on OOF. Today the framework supports the right primitives (`post_calibrate_model`'s OOF guard, `get_postcalibrators` palette, ICE metric) but selection is left to the caller ‚Äî so most caller scripts leave calibrator choice implicit and the resulting reports under-utilise the available toolkit.

2. **Lift drift, bootstrap CI, and SHAP background-dataset choice to "report-on-by-default".** The infrastructure exists (`drift_report.py`, `feature_drift_report.py`, FS-side bootstrap, `boruta_shap.py`) but lives behind opt-in flags or in a different package. Add a `ReportingConfig.honest_estimator_diagnostics=True` default that emits: bootstrap CI on each top-line metric, label & feature drift over the train/val/test trio (incl. categorical PSI), a calibration plot, and a "rows scored OOF vs val vs test" disposition table. This turns the project's existing strong primitives into mandatory artefacts of every suite run.

3. **Unify the "what dataset fed this step" provenance trail.** Every step that consumes a y or X (pre-screen, MRMR, RFECV, scaler/imputer, encoder, calibration, target-distribution analyzer, drift report) should stamp `{source: train|train_only|train+val|oof|val|test, n_rows, train_only_seed}` into a dedicated `metadata["provenance"]` namespace. The `_phase_helpers_fit_split.py:485-501` train+val Enum-widening incident is the canonical case where the deviation is documented in code but invisible at audit time. Provenance metadata closes that loop and makes the val/test/OOF discipline machine-checkable.
