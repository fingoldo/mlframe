# A5 — pipeline caching architecture critique

## Summary

mlframe already runs a layered in-process cache stack: `PipelineCache` (suite-scoped pre-pipeline output), `_PRE_PIPELINE_CACHE` (LRU on `(content_fp, sig, target_fp)`), `FeatureCache` (two-tier mem+disk with size-gated eviction), `prepared_frames_cache` / `tier_dfs_cache` / `tier_enum_map_cache` (cross-target Polars), `MRMR._FIT_CACHE` (process-wide LRU on a 5-tuple including full y blake2b), `DiscoveryCache` (disk LRU per composite target), per-class `_INIT_SIG_CACHE` (WeakKey) and `_cached_slugify` LRU. The gap is at the next layer up: nothing turns deterministic suite-level artefacts into a cross-process disk cache. `joblib.Memory` and `sklearn.Pipeline(memory=...)` are unused in `training/`. `_precompute.py` defines `train_df_fingerprint` plus `precompute_dummy_baselines` / `precompute_composite_target_specs` as NotImplementedError stubs — the contract is wired into the suite but the producer side is missing. `trainset_features_stats`, dummy baselines, composite discovery, the fitted preprocessing tuple (`category_encoder` / `imputer` / `scaler`), and `apply_preprocessing_extensions` (PySR / TF-IDF / poly basis / KBins / RBF / PCA — expensive, deterministic) are recomputed every run despite being a pure function of `(df_fingerprint, config_fingerprint, lib_version)`. No `polars.LazyFrame.cache()` is used anywhere in `training/core` (11 hits but they are all `.collect()` or `.lazy()`).

## Архитектурное предложение

Build a `SuiteArtefactCache` thin facade in `src/mlframe/training/_suite_cache.py` that wraps `joblib.Memory(cache_dir, bytes_limit, mmap_mode="r")` and is keyed by a `SuiteKeyBuilder` composing `(ContentFingerprint.short(), pydantic_config_canonical_hash, mlframe_models_frozenset, lib_versions_tuple, random_seed)` via `orjson.dumps(..., option=OPT_SORT_KEYS)` + blake2b — mirroring `_full_y_content_hash` and the existing `make_discovery_cache_key` style. Decorate the five top-leverage producers with `@suite_cache.memoize_persistent(estimate_size=...)` — every estimator MUST size-gate per CLAUDE.md (refuse to cache anything >2 GB, spill to disk-only above 200 MB). Top 5 candidates by leverage: (1) `fit_and_transform_pipeline` + `apply_preprocessing_extensions` (#3, PySR alone is minutes), (2) `trainset_features_stats` (already opt-in via precompute bundle — finish the disk layer per #1), (3) `composite_target_specs` (#1, producer side missing), (4) `dummy_baselines` (#1), (5) `_PRE_PIPELINE_CACHE` promoted to disk per (#7 + #11 with seed/version fold). Unify `DiscoveryCache` and `FeatureCache` on the same `DiskBackend` Protocol (#5). Layer `pyutilz.system.kernel_tuning_cache` per-HW for every cap (`_PRE_PIPELINE_CACHE_MAX`, `_FP_CACHE_MAX`, `_view_cache` size, `fit_cache_max_mb`) — never hardcode thresholds.

## Не нашёл

- Per-target re-fit of `category_encoder` / `imputer` / `scaler` — these are built once at `_main_train_suite.py:514` and reused; concerns about target-level scaler re-fits are unfounded.
- A literal "per-instance MRMR LRU" — the cache is `MRMR._FIT_CACHE` (class attribute, process-wide). Safety key already folds `target_name + full_y_hash + params`, so cross-target sharing is correct by construction.
- `Pipeline(memory=...)` wiring beyond a single mention in `feature_selection/optbinning.py:85` (one local Pipeline with a `memory` arg passed through; not part of the main training pipeline).
- `polars.LazyFrame.cache()` usage anywhere in `training/`.
- Inference-side warmup beyond the per-call `_loaded_models_cache` in `_predict_main_suite.py`; no module-level pre-fit model cache, no `perform_one_predict` cache.
- Any byte-budget enforcement on the suite-level `PipelineCache` (`strategies.py:748`) — only `FeatureCache` has real size gating.
