# A4 perf-hotspots critique 2026-05-24 (read-only)

Repo: `mlframe/`. Scope: `training/core/_phase_*`, `composite_transforms*`, `training/_precompute.py`. Goal: hot-path waste, dispatcher discipline, Polars fastpath gaps, per-target duplication, numba/parallel/CUDA candidates.

## Summary

The per-target hot loop is structurally clean (suite-scoped caches for fingerprint, FS-report, pipeline cache, dataset-reuse) but a few sites still pay avoidable per-target / per-iteration work: `select_target` rebuilds full model templates per target, MRMR / RFECV instances are constructed once but re-trigger per-target fits without a content-key cache, and the per-PP `_filter_polars_cat_features_by_dtype` runs inside the model loop. `get_trainset_features_stats` (pandas backend) still iterates `cat_cols` in pure Python (Polars backend already batches via one `.collect()`). `apply_polars_categorical_fixes` materialises per-column `.unique()` collects in a Python for-loop, and the post-OD recomputed union for `_DICT_ALIGN_SKIP_CARD=50_000`-cardinality cats forces an O(cards) Python sort on every cat column. `_median_residual_fit` and `_quantile_residual_fit` iterate the bin loop in Python instead of `np.bincount`/`pd.groupby` segment reductions. `_run_composite_target_wrapping` predicts on `filtered_train_df` per component without a frame-id cache (only an id(comp) cache, missed on shim wrappers from a different pass). Several dispatcher-discipline / ladder gaps: `_ewma_kernel` and `_frac_diff_inverse_kernel` are njit-only (no `parallel=True` variant, no CUDA, no size-aware dispatcher). The pandas `get_trainset_features_stats` is unconditionally selected when `train_df` lost the polars fastpath upstream - no fallback to the polars backend even when a polars original is still pinned on ctx.

## Не нашёл

- **Non-vectorized `.iter_rows()` / `.iterrows()` / `.apply()` loops on large data**: searched explicitly across `training/core/*` and `training/composite_transforms*`; none. The 19 setup helpers in `_setup_helpers.py` are all per-suite metadata/config code, not per-row.
- **`apply_polars_categorical_fixes` O(n*m) per-row scans**: the per-column path runs `null_count() + drop_nulls().unique()` which are columnar O(n) - no row-wise Python loop. Per-column sync collects (finding #2) are the issue, not per-row.
- **`run_composite_target_discovery` not incremental**: discovery already has DiscoveryCache (disk-backed at `_disc_cache = DiscoveryCache(discovery_cache_dir)`, `_phase_composite_discovery.py:340-372`) with config-signature + data-signature keying. Spec-set caching is solid; incremental discovery (cache only the surviving specs and re-rerank against a new candidate) would be a feature add, not a perf fix on the current contract.
- **Hardcoded numpy-only backend bypassing dispatcher in `_phase_train_one_target_body`**: the body file uses sklearn / strategy abstraction; no raw numpy kernels there. Dispatcher discipline gaps live in `composite_transforms*` (#4) and `_rolling_median` (#21).
- **`compute_simple_stats_numba` / `compute_moments_slope_mi` ladder gaps**: those files (`feature_engineering/_numerical_stable.py`, `_numerical_numba.py`) are out of scope but quick read shows the NaN-gate + Welford pattern from CLAUDE.md is already in place; no perf hotspot caught in this pass.
