## Findings

| ID | Severity | Category | File:Line | Summary |
|----|----------|----------|-----------|---------|
| F1 | P1 | reproducibility | `_pipeline_extensions.py:832` | `random_seed=0` is silently rewritten to `42` via `int(getattr(config, "random_seed", 42) or 42)`, breaking the sklearn-extension pipeline's (RBFSampler/Nystroem/dim-reducer) reproducibility contract for the one seed value most likely to be used deliberately. |
| F2 | P1 | correctness / atomicity | `_categorical_composite_fe.py:108-130` and `:137-154` | The per-split `for _split_name, _df in (("train", train_df), ("val", val_df), ("test", test_df))` loops for both the auto-group-concat and powerset-concat stages mutate `train_df` in place before processing `val_df`/`test_df`; if `_to_pandas_view` raises (or any step raises) on the val/test iteration, the surrounding `try/except Exception: logger.warning(...)` swallows it, leaving `train_df` with the new composite columns while `val_df`/`test_df` do not have them -- schema-drifted splits returned to the caller. |
| F3 | P1 | cache correctness / config-drift | `_pipeline_helpers_apply.py:398-411` | The main pre-pipeline-cache populate path (the hot path exercised on every fresh fit) reimplements LRU insertion inline instead of calling the already-imported `_pre_pipeline_cache_set` (`_pipeline_cache.py:626`), and the inline version omits the byte-budget eviction (`MLFRAME_PRE_PIPELINE_CACHE_MAX_BYTES`) that `_pre_pipeline_cache_set` implements. A user who sets the byte-budget env var to bound cache memory gets no enforcement on this call path -- only the entry-count cap applies. |
| F4 | P1 | ML best practice (temporal continuity) | `_ma_crossover_composite_fe.py:40-63`, `_entity_time_composite_fe.py:113-138` | Rolling/recency/state-duration features (`_rolling_means`, `per_group_recency_weighted_agg`, `time_since_state_change`) are computed **independently per split** -- each split's `_slice`d `group_ids`/`timestamps` start a fresh rolling window with no continuity from the immediately-preceding split's history for the same entity. A val/test row for an entity whose true history begins in train gets an artificially short/degenerate rolling window (e.g. `min_periods=1` on the very first val row of that entity), even though the true recency-aware feature should carry over the entity's train-period history. This under-informs val/test relative to what a real deployment (continuous timeline) would compute, and can bias the reported val/test metric versus true production behavior. |
| F5 | P2 | reliability / silent skip | `_event_proximity_decay_composite_fe.py:75-82` | `_slice()` returns `None` both when `idx is None` (expected/benign — no split) and when `len(idx_arr) != n_rows` or `ts_arr` is too short (a genuine data-alignment bug). Both cases fall through to `out[split_name] = df` (step silently skipped) with no warning distinguishing "no split" from "index/timestamp misalignment". If train_idx/val_idx/test_idx ever desync from the corresponding frames upstream (e.g. an outlier-filter step reindexes one split but not `timestamps`), this stage silently produces a schema mismatch (some splits get `event_proximity_*` columns, others don't) with zero diagnostic, unlike the TF-IDF stage in `_pipeline_extensions.py` which explicitly logs `skipped_split_mismatch` for the analogous situation. |
| F6 | P2 | correctness (edge case) | `_cross_sectional_composite_fe.py:78-84` | `k = min(k, n_snapshots - 1)` is computed independently per split against that split's own `n_snapshots`. If val/test has fewer distinct `snapshot_col` values than train (a realistic edge case for a narrow val window), the neighbor count `k` used for val/test's cross-sectional aggregation differs from the `k` used to produce train's features -- e.g. train computed with `k=10` neighbors, val with `k=3` -- producing features with a different effective smoothing/variance profile across splits, purely as a function of split size rather than intended configuration. |
| F7 | P2 | reproducibility (predict-time parity) | `_target_encoding_composite_fe.py:161` vs `:112,131` | At fit time, `entity_lookup` keys are the raw `group_ids` values (whatever dtype `group_ids` naturally has, e.g. `int64`); at predict-time replay the persisted metadata dict is keyed by `str(k)` and looked up via `entity_lookup.get(str(gid), global_prior)`. If the predict-time `group_ids` array arrives in a different but semantically-equal dtype than at fit time (e.g. `float64` vs `int64`, so `str(5.0) != str(5)`), every entity lookup silently misses and falls back to `global_prior` -- degrading the encoded feature to a constant for every row with no error or warning. |
| F8 | P2 | ML best practice (possible leakage, caller-dependent) | `_latent_interaction_svd_composite_fe.py:97-106` | The SVD/TF-IDF basis is fit once on the entirety of the caller-supplied `auxiliary_events_df` with no time-cutoff filtering inside this module, unlike the sibling `nearest_past_join` step which is leak-safe by construction (backward as-of join). If the caller passes an `auxiliary_events_df` that includes interaction events chronologically concurrent with or after the val/test period (plausible if it is simply "the whole interaction log" rather than a train-cutoff-filtered slice), the row-entity embeddings would encode information not yet available at those rows' timestamps. This depends entirely on what the (out-of-scope) caller passes in; flagging as a latent risk given no defensive check exists in this module itself. |
| F9 | P2 | code duplication / maintainability | `_pipeline_helpers.py:642-645` / `_pipeline_helpers_apply.py:41-53` | `_pipeline_helpers_apply.py` imports the full `_pipeline_cache` symbol set (`_pre_pipeline_cache_set`, `_pre_pipeline_cache_get`, etc.) identically to `_pipeline_helpers.py`, which already re-exports the same names -- two independent import blocks of the same 9 symbols from the same sibling module in two files that already import from each other (`_pipeline_helpers_apply` imports directly from `_pipeline_helpers` at line 56). Harmless today but a maintenance trap: a future signature change to any of these symbols requires checking both files' import lists instead of one canonical re-export point. |

### F1 -- `random_seed=0` silently rewritten to 42

`_ext_random_state = int(getattr(config, "random_seed", 42) or 42)` at `_pipeline_extensions.py:832` combines a `getattr` default (correctly using `42` only when the attribute is absent) with an `or 42` fallback that also fires whenever the *present* value is `0` -- a legitimate, explicit user choice for `random_seed`. `PreprocessingExtensionsConfig.random_seed` is a plain `int` field (`_preprocessing_configs.py:143`) with no documented restriction against `0`. The nearby PySR seed handling two lines away (`_apply_pysr_fe`, line 209: `int(getattr(config, "random_seed", 42))`) and an explicit comment at line 118 of the same file ("Explicit None-check rather than `or \"standard\"`... Same class of bug as the prior `or 42` rewrite for `random_state=0` in `_phase_composite_discovery`") both show the project is aware of and has previously fixed this exact bug class -- this specific instance at line 832 was missed. The existing regression test `tests/training/pipeline/test_pipeline_fe_extensions.py::test_a2_03_nonlinear_random_state_reproducible_from_seed` only exercises seeds `123`/`999`, never `0`, so it does not catch this. Suggested fix: `None if getattr(config, "random_seed", None) is None else int(config.random_seed)`, defaulting to 42 only on `None`.

### F2 -- categorical composite FE: non-atomic per-split mutation

Both composite-categorical stages iterate `("train", train_df), ("val", val_df), ("test", test_df)` and reassign the enclosing `train_df`/`val_df`/`test_df` locals one split at a time, inside a single `try` that wraps the *entire* loop. `_to_pandas_view` raises (`KeyError` for pandas, `ColumnNotFoundError`-style for polars) if a split's frame is missing one of `cat_cols`/`_group`/`_powerset_present` columns -- a realistic scenario when val/test carries a narrower schema than train (dropped columns, an upstream typo, etc., the exact scenario the TF-IDF stage in `_pipeline_extensions.py` explicitly guards against with its `skipped_split_mismatch` check). Because train is processed first in the tuple, an exception on val's/test's turn leaves `train_df` already mutated with the new composite columns while `val_df`/`test_df` are returned unchanged -- the function returns schema-drifted splits with only a swallowed `logger.warning`, no raise. Downstream this either crashes with an opaque sklearn "Feature names should match" / booster feature-count error, or -- worse -- silently trains and evaluates on differently-shaped frames if some other step happens to pad the narrower split back to the same width. This is the same failure mode the PySR-equation step (`_pipeline_extensions.py:265-293`) and the row-wise-summary-stats step (`_pipeline_extensions.py:663-678`, explicitly commented "ATOMIC across train/val/test") were fixed for elsewhere in this same package; this file did not receive the analogous fix. Suggested fix: compute all three splits' new-columns frames into locals first, and only reassign `train_df`/`val_df`/`test_df` after all three succeed (mirroring the `_train_rw`/`_val_rw`/`_test_rw` pattern already used in `_pipeline_extensions.py`).

### F3 -- pre-pipeline cache populate bypasses byte-budget eviction

`_apply_pre_pipeline_transforms` (`_pipeline_helpers_apply.py:398-411`) is the cache-populate call site exercised on every genuine (non-cache-hit) pre-pipeline fit -- the primary/hottest path through this cache. It inserts directly into `_PRE_PIPELINE_CACHE` and evicts only by entry count (`_cap`), even though `_pipeline_cache.py` already defines `_pre_pipeline_cache_set` (imported into this very file at line 51 but never called) which additionally enforces `_PRE_PIPELINE_CACHE_MAX_BYTES` via `_approx_entry_bytes`-based LRU eviction. A user who sets `MLFRAME_PRE_PIPELINE_CACHE_MAX_BYTES` to bound cache RAM on a long-running service gets that bound enforced only on whichever (if any) other call site still goes through `_pre_pipeline_cache_set` (e.g. `trainer.py`, `cb/_cb_pool.py` -- both out of this cluster's scope) -- not on this one, so cache memory can grow unbounded in byte terms as long as the entry count stays under `_cap`, on frames that (per this file's own docstring) can be many GB each. `tests/training/pipeline/test_pre_pipeline_cache_byte_budget.py` only unit-tests `_approx_entry_bytes`/`_read_int_env` in isolation and does not exercise this populate path end-to-end, so the gap is untested. Suggested fix: replace the inline insert/evict block with a call to `_pre_pipeline_cache_set(train_df, val_df, pre_pipeline, train_df, val_df, train_target=..., target_name=..., cache_max=cache_max, sample_weight=sample_weight)`.

### F4 -- rolling/recency features restart at each split boundary

`apply_ma_crossover_composite_fe` and `apply_entity_time_composite_fe` both compute their time-aware aggregates (`_rolling_means`, `per_group_recency_weighted_agg`, `time_since_state_change`) using only the rows of the *current* split, sliced independently via `train_idx`/`val_idx`/`test_idx`. There is no mechanism carrying an entity's rolling/recency state across the train→val→test boundary. Concretely: if entity `E` has rows in both train and val (chronologically train-then-val, the normal time-split case), val's rolling mean / recency-weighted aggregate for `E`'s first val row is computed from scratch (a 1-observation "rolling mean", or a recency aggregate with no prior observations) instead of continuing from `E`'s train-period history, even though that history genuinely precedes the val row in time and is not leakage to use. This produces evaluation-time features that are systematically less informed than what the same code would produce in a real continuous-timeline deployment, which can bias reported val/test metrics pessimistic relative to production behavior. This may be an accepted simplification (no comment in either file states so) rather than an oversight; flagging both readings. Suggested fix (if not intentional): thread each split's aggregation computation over the *union* of all rows up to and including that split (or explicitly document the current per-split-restart behavior as a known limitation with a `# NOTE` at the `_slice` definitions).

### F5 -- event-proximity-decay: no diagnostic on idx/timestamp misalignment

See table above. `_slice` conflates the benign "this split doesn't exist" case with the "idx no longer matches this split's row count / exceeds ts_arr length" case, silently returning `None` for both and producing `out[split_name] = df` (no new columns) either way, with zero log output. Contrast with the TF-IDF-column mismatch handling in `_pipeline_extensions.py:518-533`, which explicitly warns with column names and the reason. Suggested fix: emit a `logger.warning` distinguishing "idx is None (split absent)" from "idx/timestamps misaligned with this split's row count" so an upstream indexing bug doesn't manifest as an unexplained missing-column schema drift several steps downstream.

### F6 -- cross-sectional-neighbor `k` diverges by split size

See table above. Suggested fix: compute (or at minimum log) a single effective `k` from the smallest split's `n_snapshots`, or explicitly document that `k` is best-effort per split.

### F7 -- target-encoding predict-time lookup fragile to `group_ids` dtype drift

See table above. Suggested fix: normalize `group_ids` to a canonical string form (or a fixed dtype) at both fit time (before building `entity_lookup`) and replay time, rather than relying on `str()` producing the identical representation for whatever dtype happens to arrive at each call site.

### F8 -- latent-interaction SVD basis has no explicit train-only cutoff

See table above. Suggested fix (if the caller cannot guarantee `auxiliary_events_df` is already train-cutoff-filtered): accept an optional `cutoff_time`/`train_end_ts` parameter and filter `events_pd` to `<= cutoff` before fitting, mirroring `nearest_past_join`'s explicit leak-safety-by-construction design.

## Coverage notes

Every file in scope was read in full; nothing was skipped. I did not execute any code, run the test suite, or run the `_benchmarks/` scripts (per the read-only audit constraint) -- findings F1-F9 are derived from static reading plus cross-referencing existing tests to confirm coverage gaps (e.g. confirming `test_a2_03_...` never passes `random_seed=0`, confirming `test_pre_pipeline_cache_byte_budget.py` never calls `_apply_pre_pipeline_transforms`). I did not trace every call site of the composite-FE `apply_*` functions outside this cluster (e.g. exactly how `auxiliary_events_df` is assembled upstream for F8, or whether `train_idx`/`val_idx`/`test_idx` can ever desync from their frames in practice for F5) since that logic lives in `src/mlframe/training/core/**`, out of this cluster's scope -- both findings are flagged with that caveat rather than asserted as definitely user-reachable.
