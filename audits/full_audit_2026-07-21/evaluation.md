### F1 -- `pos_label` parameter clobbered by a same-named loop variable (reports.py)

`evaluate_estimators(..., pos_label: int = 1, ...)` uses `pos_label` at line 270 (`preds = (probs[:, pos_label] > threshold).astype(np.int8)`) to pick which column of a binary classifier's `predict_proba` output counts as the positive class. Inside the same function, when `show_calibration_plot=True` and `use_sklearn_calibration=True`, line 399 runs `for pos_label in range(nclasses):` -- a plain Python `for` loop has no block scope, so this reassigns the SAME local variable that is also the function parameter. After that loop finishes (for a binary problem, `pos_label` ends at `1`, which happens to match the common default, masking the bug in the common case), the corrupted value persists into the next iteration of the outer `for est in estimators:` loop. A caller who (a) passes 2+ estimators, (b) sets `use_sklearn_calibration=True`, and (c) explicitly passes `pos_label=0` (to invert which class counts as positive) gets correct behavior for the FIRST estimator only; from the second estimator onward, `pos_label` has been silently reset to `nclasses - 1` regardless of what the caller asked for -- a silently-wrong classification threshold with no error or warning. Fix direction: rename the loop variable (`for _pl in range(nclasses):` and read `probs[:, _pl]` inside), or dereference `pos_label` under a local alias before the loop; add a regression test with 2 estimators + `pos_label=0` + `use_sklearn_calibration=True` asserting the second estimator's predictions are computed with `pos_label=0`.

### F2 -- `optimize_group_blend_weight` doesn't actually hold previously-tuned groups fixed (label_correlation_rerank.py)

The docstring promises "Groups are optimized one at a time, in the given order, each holding previously-optimized groups' weights fixed (coordinate ascent)". The implementation builds `candidate_weights = dict(weights)` (previously-tuned groups) merged with the current group's trial weight, but then calls `label_correlation_rerank(pred_scores[mask], correlated_pairs=[], correlated_groups=[group], group_weights=candidate_weights)` -- passing `correlated_groups=[group]`, i.e. ONLY the group currently being tuned. `label_correlation_rerank` only reranks columns belonging to groups actually present in its `correlated_groups` argument, so the previously-optimized groups' columns are left as raw (unblended) scores during every later group's CV search, even though `weights` was carefully merged into `candidate_weights` as if it would apply. Since the default metric (LRAP) scores the ENTIRE row's ranking across all labels, not just the group under test, the search for group B's weight is evaluated against a scores array that does not match what will actually be fed to `metric_fn`/production once group A's tuned weight is genuinely applied later via `label_correlation_rerank(..., correlated_groups=all_groups, group_weights=weights)`. With 2+ disjoint groups this can pick a suboptimal weight for later groups and silently defeats the stated coordinate-ascent guarantee (it degrades gracefully to "each group tuned independently," which is a real but undocumented and untested behavior difference). Fix direction: pass `correlated_groups=list(weights.keys()) + [group]` (or equivalently pre-apply all previously-decided groups) so the CV evaluation reflects the state the final call will actually produce; add a 2-group biz_value/regression test.

### P1a -- non-NaN-safe overall variance in `constant_group_target_scan` (constant_group_leak_scan.py)

`y = np.asarray(y, dtype=np.float64); overall_var = float(np.var(y, ddof=1)) if len(y) > 1 else 0.0; if overall_var <= 0.0: raise ValueError(...)`. `np.var` does not skip NaN, so a `y` with even one NaN entry makes `overall_var` itself `NaN`. Because `NaN <= 0.0` evaluates to `False` in Python/numpy, the explicit "zero variance -- nothing to compare against" guard never fires, and the function proceeds to compute `min_ratio = eligible["var"].iloc[worst_pos] / overall_var` (`x / NaN = NaN` for every column), so every row of the returned report silently shows `min_group_variance_ratio = NaN` and `flagged = False` -- a leak that exists in the non-NaN rows is invisible, with no error or warning surfaced to the caller. This is the same "min()==max() style NaN blind-spot" bug class flagged repeatedly elsewhere in this codebase's other modules, since pandas' per-group `.var()` (via `groupby(...).agg(["var","size"])`) IS NaN-skipping by default, so the two variance computations (overall vs per-group) have inconsistent NaN semantics. Fix direction: compute `overall_var` with `np.nanvar` (or explicitly `y[~np.isnan(y)]` before `np.var`), or raise a clear "y contains NaN" `ValueError` up front, matching the existing zero-variance guard's intent.

## Coverage notes

- The 18 `_benchmarks/bench_*.py` scripts under `evaluation/_benchmarks/` (~1.98k additional LOC) were intentionally not read in depth -- they are one-off A/B measurement harnesses referenced from docstrings (e.g. `bench_bootstrap_ci_coverage.py`, `bench_auc_ci_delong_vs_bootstrap.py`) rather than library surface, and the task's own file/LOC counts (~21 files / ~4.8k LOC) line up with excluding them, so this was treated as the intended scope boundary rather than a silent skip.
- I did not execute any test (per the read-only mandate), so F1/F2/P1a are traced by careful static reading of the exact code paths plus confirming (via `Grep`/`Read` of the existing test files) that no existing test exercises the failing configuration -- I did not empirically reproduce the failures with a live interpreter.
- `reports.py`'s CatBoost/`Pool`/`imblearn.Pipeline`-specific branches (lines 208-233) were read but not deeply cross-checked against CatBoost's actual `set_baseline`/`eval_set` contract beyond what's inferable from the code and comments (F7 is flagged at reduced confidence for this reason).
- Did not trace call sites of `evaluate_estimators`/`evaluate_grouped` outside `tests/` to gauge how commonly the `pos_label`-corrupting configuration (F1) or the multi-group `optimize_group_blend_weight` call pattern (F2) actually occurs in real notebooks/pipelines -- severity is assigned from blast-radius-at-the-API-contract-level, not from confirmed production incidence.
