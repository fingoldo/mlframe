Below, one paragraph per finding with the concrete failure scenario and a fix direction.

**F1** -- `preprocessing.py`'s own `_process_special_values` docstring (top of file) explains at length why pre-split filling biases the post-split imputer's learned statistic, and applies that principle correctly to NaN handling and to the `fillna_value`-driven inf path (`inf -> NaN`, comment at line 414-416). But the far more commonly exercised default path (`PreprocessingConfig.fix_infinities=True`, `fillna_value=None`, which is the class default) calls `process_infinities(df, fill_value=0.0, ...)` (both in the legacy 3-pass branch at line 420/428 and in the newer "batched" polars fastpath at line 400), which replaces every `+/-inf` cell with a literal `0.0` BEFORE the train/val/test split, injecting a specific wrong numeric value into every split rather than a missing-value marker the post-split imputer can handle correctly. Any feature-engineering step that can produce `inf` (division ratios, `log(0)`, etc.) hits this by default. Fix direction: route the default `fix_infinities=True` path through the same "inf -> NaN, let the post-split imputer decide" logic already implemented for the `fillna_value` branch, or explicitly document/justify why infinities are treated differently from NaN/null.

**F2** -- `compute_signature` is used by `xgb_shim.py`, `lgb_shim.py`, and `cb/_cb_pool.py`/`cb/_cb_pool_build.py` (all outside this cluster's scope, but consuming this in-scope module) to key a process-level cache of the expensive booster-native dataset object across `sklearn.clone()` / `.iloc[idx].reset_index()` calls. The fingerprint samples only 3 rows (`indices = (0, n_rows // 2, n_rows - 1)`) plus columns/shape/dtypes. Two frames that are logically different (e.g. different `train_idx` slices of the same underlying `df`, or two different composite-target-derived frames) but happen to agree on those exact 3 rows across every column, with the same shape/dtype/columns, will collide and the cache will hand back a `Pool`/`DMatrix`/`Dataset` built from the WRONG rows to a `.fit()` call, producing a model silently trained on stale/mismatched data with no error or warning. The composite-target-discovery and ensemble-refit code paths that motivated this cache (many near-identical row-wise-extension frames sharing a base X) are exactly the regime most likely to produce partial row overlap. Fix direction: either widen the sample (e.g. a strided sample proportional to `sqrt(n_rows)`, or a cheap vectorized hash of every row's bytes via `pandas.util.hash_pandas_object`/`polars` hash) or add a documented, verified invariant that callers never key two structurally-similar-but-content-different frames into the same cache scope.

**F3** -- `mc_dropout_predict` is the MC-dropout uncertainty-estimation entry point for torch models. If the forward pass fails on iteration `k<n` of the loop at line 50 (CUDA OOM under repeated forward passes, a malformed `X`, or any other runtime error), the function raises out of the `with torch.no_grad()` block and `module.train(was_training)` at line 51 is skipped. The caller's model is left with its dropout submodules stuck in train mode (stochastic dropout still active) while every other submodule (BatchNorm/LayerNorm) is in eval mode -- an inconsistent, undocumented state that will silently corrupt any subsequent `predict()`/`fit()` call on the same module object without raising. Fix direction: wrap the forward-pass loop and the `module.train(was_training)` restoration in `try/finally` so the mode is always restored, matching the repo's own "mutate-and-restore via try/finally" convention used correctly elsewhere in this cluster (e.g. `_io_save.py`'s compile/bloat-strip restoration).

**F4** -- `save_mlframe_model(..., lean=True)` is documented as stripping "inference-irrelevant heavy fields" to keep on-disk bundles small (the module's own history shows this being tightened once already for `oof_preds`/`oof_probs`, explicitly citing a prod incident where 16-32 MB arrays leaked into "lean" saves). `_trainer_train_and_evaluate.py` stamps `oof_target`, `calib_probs`, `calib_target`, and `calib_preds` onto the same `SimpleNamespace` the strip set targets (lines 986-991), all the same class of per-row/per-calib-slice array as the already-stripped fields, but none of the four are in `_LEAN_STRIP_FIELDS`. Any suite run with `oof_n_splits>=2` (for level-1 stacking) or a non-trivial `TrainingSplitConfig.calib_size` will produce "lean" bundles that still carry these arrays, silently reintroducing the exact bloat class the prior fix targeted. Fix direction: add `"oof_target"`, `"calib_probs"`, `"calib_target"`, `"calib_preds"` to `_LEAN_STRIP_FIELDS` in `io.py`.

**F5** -- `helpers.py` imports `CUDA_IS_AVAILABLE` from `._gpu_probe` (which already performs the guarded `numba.cuda.is_available()` probe once, memoized at `_gpu_probe` module-import time) and then, a few lines later at module scope, recomputes an equivalent value from scratch via a second `try: from numba.cuda import is_available ...` block, silently discarding the imported one. Functionally harmless today (both probes compute the same thing the same way), but it is dead code that duplicates probe cost on every `helpers.py` import and will silently diverge in behavior if either probe's logic changes in only one place in the future. Fix direction: drop the local re-probe and use the imported `CUDA_IS_AVAILABLE` from `._gpu_probe` directly (or drop the unused import if the local probe is intentionally kept).

**F6** -- `optimize_model_for_storage`'s docstring makes an unqualified "For classification models" claim, but `is_classification = target_type == TargetTypes.BINARY_CLASSIFICATION` (line 198) means the `train_preds`/`val_preds`/`test_preds`-stripping block only fires for binary targets; a multiclass or multilabel classification model's namespace keeps its `*_preds` arrays even when `optimize_storage=True`. This is a real (if modest) memory-footprint gap for non-binary classification suites, and a genuine docstring/implementation mismatch that will mislead anyone reading the docstring to believe multiclass storage is optimized. Fix direction: either extend the condition to `target_type.is_classification` (already a property on `TargetTypes` per `_configs_base.py`) or narrow the docstring to say "binary classification models" explicitly.

**F7** -- `evaluate_model` accepts `verbose: int = 1` in its signature and docstring ("Verbosity level") but the function body never references it -- it forwards `**kwargs` straight to `report_model_perf`, and `verbose` is not among the named parameters passed through. Any caller who passes `evaluate_model(..., verbose=0)` expecting suppressed logging silently gets the default-verbosity behavior instead. Fix direction: either forward `verbose` to `report_model_perf` (if it accepts an equivalent knob) or drop the parameter and update the docstring.

**F8** -- Inside `_ChainEnsemble.fit`'s NaN-guard block, the lines `if _arr.dtype.kind == "f" and not _np.all(_np.isfinite(_arr[~_np.isnan(_arr)])): pass  # inf already gone here; nan handled below` compute a boolean and then do nothing with it -- a vestige of an earlier refactor (the comment even documents that the intended inf-handling was moved elsewhere) left behind as dead, confusing code that runs (and allocates a boolean mask) on every `_ChainEnsemble.fit` call. Fix direction: delete the dead `if ... pass` block.

**F9** -- `fix_quantile_crossing(mode="isotonic")` is meant to guarantee monotonic-non-decreasing quantile columns per row. A row containing a NaN quantile prediction (upstream booster instability, e.g. a rare CB/XGB/LGB failure mode on a degenerate fold) is classified `is_mono=False` by the `np.diff(...) >= 0` check (NaN comparisons are always False), so it enters the PAVA kernel; inside the kernel every `val[m-1] > val[m]` comparison involving the NaN is also always False, so no pooling ever happens for that row and it is emitted essentially unchanged (still possibly crossing, still containing the NaN) -- silently defeating the function's documented "monotonically non-decreasing columns enforced row-wise" contract for that row with no warning. Fix direction: explicitly detect and either drop/impute NaN rows before the PAVA call, or document that NaN rows are passed through unfixed.

**F10** -- `configure_training_params` (the primary per-suite-call config assembly function) declares many optional parameters using the bare-`None`-default pattern (`df: pd.DataFrame = None`, `target: pd.Series = None`, `target_label_encoder: object = None`, `train_idx: np.ndarray | None = None` is done correctly for some but not all) instead of `Optional[T] = None` throughout, which is flagged as a hard "write mypy-clean code" convention violation in this repo's own CLAUDE.md. This is purely a static-typing/documentation-accuracy issue (the runtime behavior is unaffected), but it means `mypy` cannot catch a caller passing `None` where a genuinely-required frame is expected, and IDE tooling under-reports the actual accepted type. Fix direction: wrap every implicitly-optional annotation in `Optional[...]`.

**F11** -- `NoiseAugmentedEnsemble` is documented as forwarding `predict_proba` "if the base has it", implying conditional exposure (e.g. via `__getattr__` delegation or a factory that only attaches the method when appropriate), but the class defines `predict_proba` unconditionally as a normal method. Downstream code that duck-types "is this a classifier" via `hasattr(model, "predict_proba")` (a common sklearn-ecosystem pattern, and one this same cluster's `_classif_helpers.py`/reporting dispatch relies on elsewhere) will incorrectly treat a `NoiseAugmentedEnsemble` wrapping a *regressor* as probability-capable, and only discover the mismatch when `predict_proba` is actually invoked and raises `AttributeError` deep inside the list comprehension at line 65, rather than being routed to the regression code path from the start. Fix direction: only define `predict_proba` conditionally (e.g. via `__getattr__` delegation, or by not defining the method on the class and instead attaching it dynamically in `__init__` only when `hasattr(base_estimator, "predict_proba")`).
