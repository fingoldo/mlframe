**F1** (`group_bias_correction.py`): `fit_group_bias_correction` returns `{str(group_value): ratio}` built from `agg.index` (whatever dtype the fit-time `group` array had), and `apply_group_bias_correction` independently does `pd.Series(group).astype(str)` on the apply-time array. `str(np.int64(3)) == "3"` but `str(np.float64(3.0)) == "3.0"` -- these never match. Since the whole point of this API (per its own docstring: "store this and reapply... never recompute on rows without ground truth") is a fit/apply split that can cross process/time boundaries, a realistic production scenario (e.g. a merge or a `NaN` elsewhere in the `store_id` column upcasting it from int to float between the validation run and the scoring run) makes every group silently miss and the correction becomes a systemic no-op with zero error/warning. Suggested fix: canonicalize the group key (e.g. always `str(pd.array(group).astype(str))` at BOTH fit and apply via one shared helper, or coerce a numeric group column to a fixed dtype before keying) and/or log a warning when the apply-time match rate against `ratios` is anomalously low.

**F2** (`post.py`): the identity/leakage guard deliberately raises two `ValueError`s with the strings `"identical"` and `"reordered"` in their messages, then an outer `except (TypeError, ValueError) as _eq_err: if "identical" in str(_eq_err) or "reordered" in str(_eq_err): raise` re-raises only those two, treating everything else as "non-comparable dtypes, skip". This conflates "my own deliberately-raised guard exception" with "a genuine TypeError from incompatible dtypes" via string content instead of a distinct exception type/flag. It is exercised by `tests/calibration/test_train_postcalibrators_leakage_guard_overlap.py` today (which pins the exact wording via `pytest.raises(..., match="reordered|identical")`), so a wording-only edit would be caught by that test running green-to-red -- but if that test file itself is ever touched in the same change (or CI is skipped), the guard against calib/test leakage (a severe methodology bug: honest-metric contamination) degrades silently. Suggested fix: raise a dedicated internal exception subclass (e.g. `_CalibTestOverlapError(ValueError)`) for the deliberate guard trips and catch that type explicitly, instead of string-matching.

**F3** (`quality.py`): `all_performances`/`data`/`performances` accumulate correctly across the `for i in range(nintervals)` loop, but `x`/`y` (the per-interval binned pocket arrays used only for plotting) are reassigned each iteration and never accumulated; the post-loop `x_min, x_max = np.min(x), np.max(x)` (and the `ax.plot([x_min, x_max], [x_min, x_max], "g--", label="Perfect")` diagonal, and the title's point count) reflect only the LAST interval plotted. With `nintervals > 1` and non-uniform per-interval prediction ranges this draws a misleading reference diagonal alongside the other intervals' curves. Suggested fix: track a running `(min, max)` across all intervals (or compute it from the full `y_pred`/`y_true` up front) before drawing the diagonal/labels.

**F4** (`sticky_state_persistence_floor.py`): the docstring states `floor` should be "in `[0, 1)`" but nothing enforces it. If a caller (not the in-repo `optimize_persistence_floor*` sweepers, which are bounded via `np.linspace(0, 1, ..., endpoint=False)`, but any external caller) passes `floor >= 1.0` for a row whose `active_prob < floor`, `target_rest_mass = 1.0 - row_floor` goes to `0` or negative, producing a negative "rest" probability mass in the renormalized output row -- an invalid probability distribution, produced silently. `active_class` indices are likewise never bounds-checked against `probs.shape[1]`; an out-of-range or negative index either raises an opaque `IndexError` deep in the fancy-indexing call or (for small negative values) silently wraps to the wrong column. Suggested fix: validate `0 <= floor < 1` (scalar or per-element) and `0 <= active_class < k` up front with a clear `ValueError`.

**F5** (`probabilities.py`): `generate_similar_probs_by_ranking(predicted_probs, true_outcomes, n_bins=10, noise_scale=0.001, random_state=None)` never reads `true_outcomes` -- the function only rank-bins `predicted_probs` and shuffles within bins. The docstring claims the result "maintain[s] Brier Score and ROC AUC", which is only approximately true as a side-effect of small within-bin shuffles + small noise, never actually checked. This differs from the sibling `generate_similar_probs`, which iterates and explicitly verifies closeness to the true metrics. A caller relying on the docstring's guarantee for anything beyond "roughly" would be misled. Suggested fix: either drop the unused parameter (and soften the docstring's claim) or add the same closeness-check/best-candidate-tracking pattern `generate_similar_probs` already uses.

**F6** (`group_bias_correction.py` / `group_zero_sum_constraint.py`): both `fit_group_bias_correction`'s `df.groupby("group", sort=False)` and `apply_group_zero_sum_constraint`'s `df.groupby("group", sort=False)` use pandas' default `dropna=True`, so any row whose `group` value is `NaN` is invisible to the per-group aggregate. At apply time such a row's group also cannot be found in the resulting lookup dict/Series, so it silently receives `default_ratio`/zero offset. This is plausibly reasonable behavior (an "unknown group -> no-op" default), but it is undocumented in either docstring and untested (`grep` across `tests/calibration/test_biz_val_group_*` found no NaN-group case). Suggested fix: document the NaN-group behavior explicitly, or raise/warn when the input actually contains NaN group labels so a caller notices rather than silently getting fewer corrected rows than expected.

**F7** (`ensembling.py` / `_ktc_dispatch.py`): flagged per the audit checklist's explicit ask about the `MLFRAME_DISABLE_GPU` opt-out convention. `_dispatch`/`_dispatch_weighted` only fall back to CPU when the `cupy` call itself raises (OOM, missing driver, or `CUDA_VISIBLE_DEVICES=""` making no device visible to CUDA) -- there is no explicit check of `MLFRAME_DISABLE_GPU`, the convention documented elsewhere in this same repo (`feature_selection/filters/*`) for forcing CPU-only execution. This module instead defines its own env var, `MLFRAME_ODDS_COMBINE_BACKEND`. Noting the alternative reading explicitly: it's plausible `MLFRAME_DISABLE_GPU` was always meant to be `feature_selection`-scoped rather than a repo-wide contract, in which case this is working as designed and not a bug. Suggested fix (if repo-wide is intended): check `os.environ.get("MLFRAME_DISABLE_GPU")` in `_forced_backend`/`_dispatch` alongside the existing override.

**F8** (`_ktc_dispatch.py`): `_get_cache()`'s two `except Exception` clauses are inconsistent: the first (FS package unavailable) has an inline comment but no `logger` call; the second (the singleton call itself failing) is marked `# pragma: no cover - defensive` and also has zero logging. Every other failure path in this same module (`choose_odds_combine_backend`'s KTC lookup failure, the tuner's cupy-unavailable branch) logs at `debug`. A silently-broken kernel-tuning cache (e.g. a corrupted on-disk cache file, a permissions issue) would degrade every call to the measurement-backed `fallback` threshold with no trace in logs. Suggested fix: add a `logger.debug` in both except branches, matching the rest of the module's convention.

**F9** (`ensembling.py`): `odds_ratio_combine` checks `p.ndim != 2` and the `weights` shape, but not `p.shape[1] >= 1`. For a `(n, 0)` input, `logits.sum(axis=1)` (numpy) or the njit loop's `for j in range(k)` (k=0, no-op) both produce `combined_logit = 0.0` for every row, so the function returns a constant `0.5` for every row rather than raising. A caller who accidentally passes an empty member list (e.g. an upstream filter that dropped all members) gets uniformly "uninformative" 0.5 predictions silently instead of an actionable error.

## Coverage notes

- `src/mlframe/calibration/_benchmarks/**` (20 files, standalone `bench_*.py` scripts, none imported by library code) were NOT deep-audited line-by-line; I confirmed via `wc -l` they sit outside the task's stated "~18 files, ~4.8k LOC" scope, and did a targeted `grep` across all of them for `eval(`/`exec(`/`os.system`/`subprocess` (the one hit, `bench_per_class_method_isotonic_vs_sigmoid.py:104`, is a local function literally named `_eval`, not a call to builtin `eval`) to rule out anything security-relevant. If the intended scope does include benchmarks, they would need a separate pass.
- `mlframe.evaluation.bootstrap` (`_ci_from_samples`, `_jackknife_metric`, used by `policy.py`) and `mlframe.training.evaluation.report_model_perf` (used by `post.py`) are outside `calibration/` and were only read far enough (via their call sites' usage and the existing regression tests in `tests/calibration/test_p1_p2_hardening.py`) to confirm `post.py`'s `PERF_DICT_COL = 1` indexing is correct, not to audit their own internals.
- `mlframe.feature_selection.filters.get_kernel_tuning_cache` (imported lazily by `_ktc_dispatch.py`) and `mlframe.models.ensembling.ensemble_probabilistic_predictions` (imported by `post.py`) are both explicitly excluded from this audit's scope (the former lives in the excluded `feature_selection/filters/**` tree); only their call-site usage within `calibration/` was reviewed.
- I did not execute any test or benchmark (read-only audit per instructions), so all findings are from static/behavioral reasoning over the source and cross-referenced against existing test file contents, not from a live repro run.
