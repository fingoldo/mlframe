Below, one paragraph per finding.

**F1** -- `src/mlframe/calibration/post.py` is grouped by this audit's own package taxonomy (and by the module's own docstring, "post-hoc probability calibration") as a lower/mid-level package that `training/` orchestrates, yet at module scope it does `from mlframe.training import TargetTypes` (line 45) and `from mlframe.training.evaluation import report_model_perf` (line 56), plus `from mlframe.models.ensembling import ensemble_probabilistic_predictions` (line 46). `mlframe.training.evaluation` and `mlframe.training.honest_diagnostics` already import from `mlframe.calibration.policy` -- but they do it lazily, *inside function bodies*, specifically to avoid this exact cycle (confirmed no module-level `from mlframe.calibration` anywhere under `training/`). `calibration/post.py` is the one asymmetric case that imports the "higher" package eagerly. No import crash was observed (`import mlframe.calibration.post` succeeds), because `mlframe.training`'s own `__init__.py` uses a lazy `__getattr__` dispatch table for most of its surface, and none of the eager bottom-of-file imports it does perform (`_conformal_finalize`, `_regression_calibration`, `_tta`, etc.) import `calibration`/`feature_engineering` back -- but the coupling is real, makes `calibration/post.py` unimportable standalone without pulling in a large fraction of the training package, and is a maintenance trap for the next person who tries to add a genuine `training -> calibration` module-level import (which would then create an actual `ImportError` cycle). Suggested fix: move `TargetTypes`, `report_model_perf`, and `ensemble_probabilistic_predictions` to lazy/deferred imports inside the functions that use them (the codebase already does this pattern extensively elsewhere, e.g. `models/ensembling/process_method.py` with an explicit `# joblib-import-race-ok` comment), or relocate the shared pieces (`TargetTypes`) to a lower-level shared module both sides can import from.

**F2/F3** -- `feature_engineering/categorical_powerset_concat.py:34` and `feature_engineering/two_step_target_encode.py:17` both do `from mlframe.training.feature_handling.ordered_target_encoder import ...` at module scope. `mlframe.training.feature_handling.ordered_target_encoder` itself has no back-imports into `feature_engineering`/`calibration` (checked), so this does not currently cycle, but it does mean importing either `feature_engineering` module first triggers execution of `mlframe/training/__init__.py`'s eager bottom-of-file import block (`_conformal_finalize`, `_regression_calibration`, `_tta`, `_uncertainty_eval`, `_mc_dropout`, `_noise_ensemble`, `_aggregate_cv_early_stopping`, `_pseudo_group_reconstruction`, `_overlapping_walk_forward_cv`, `_easy_ensemble`, `_direct_horizon_bucket_forecaster` -- 11 submodules) just to reach one target-encoding helper. Same direction-of-dependency concern as F1: `feature_engineering` is supposed to be usable independently of the `training` orchestrator. Suggested fix: same as F1 -- defer the import inside the function, or move `ordered_target_encode`/`ordered_target_encode_batch` to a shared lower-level location (e.g. under `feature_engineering/` or a new small `mlframe.encoding` module) that both `training.feature_handling` and `feature_engineering` can import without a directional dependency.

**F4** -- `preprocessing/outlier_detector_zoo.py:16` imports `mlframe.models.ensembling.selection.rank_average_blend` at module scope. `preprocessing/` is meant to be a foundational data-prep layer; reaching into the ensembling/modeling layer for a blending helper inverts that. No back-cycle was found (`models/ensembling/selection.py` and `models/ensembling/__init__.py` do not import `mlframe.preprocessing`), so this is a design-direction smell rather than an active bug. Suggested fix: either move `rank_average_blend` to a shared numeric-utilities module both `preprocessing` and `models.ensembling` can depend on, or keep the import local to the one factory function that needs it (deferred import), matching the pattern already used elsewhere in this codebase for genuine cross-layer needs.

**F5** -- `core/__init__.py`, `data/__init__.py`, `estimators/__init__.py`, `inference/__init__.py`, `models/__init__.py`, `preprocessing/__init__.py`, `training/composite/transforms/__init__.py`, and `utils/__init__.py` all assemble their public surface purely via `from .submodule import *`, and none of the star-imported submodules (verified: `core/arrays.py`, `core/stats.py`, `data/datasets.py`, `estimators/base.py`, `inference/predict.py`, `models/ensembling/__init__.py`, `preprocessing/cleaning.py`, `utils/eda.py`) define `__all__` either. `metrics/__init__.py` is the one package in scope that does this correctly: it computes `__all__ = sorted({name for name in globals() if not name.startswith("_")} | {...})` (line 57) with a comment explaining exactly why ("Preserve the historical `from mlframe.metrics import *` surface"). An AST scan of the star-imported submodules for name collisions found only benign ones (every module defines its own module-level `logger = logging.getLogger(__name__)`, which silently overwrites the previous submodule's `logger` binding in the aggregated package namespace on each subsequent `import *` -- harmless since nothing consumes `mlframe.core.logger` etc., but it is a live demonstration of the uncurated-namespace risk: any two submodules that happened to define a same-named public helper would silently shadow one another with no linter warning, `mypy --strict`/`pyflakes` would also not catch it without explicit `__all__`). Suggested fix: apply the `metrics/__init__.py` pattern (or an explicit hand-curated `__all__`) to the other 8 packages.

**F6** -- Three files exceed 1000 LOC in the in-scope tree: `training/core/_phase_composite_post_xt_ensemble/__init__.py` (1147, and notably this is an `__init__.py`, i.e. a subpackage facade file that has itself grown into a monolith rather than staying a re-export shim), `training/composite/transforms/nonlinear.py` (1045), and `training/_trainer_train_and_evaluate.py` (1002). This is the authoritative >1000-LOC list for the whole audited tree (excluding the two excluded MRMR packages) per this cluster's mandate; see Proposals for the much longer 900-999 LOC "approaching the limit" band. Suggested fix: apply this repo's own documented convention (CLAUDE.md: "Carve *before* a file nears ~800-900 LOC") -- split each into a sibling module/subpackage with re-export, per the monolith-split convention already used elsewhere (e.g. `_phase_composite_post_xt_ensemble` already being a subpackage suggests the split boundary just needs to go one level deeper).

**F7** -- Three separate, independently-implemented "stable per-component seed from a master seed" helpers exist in `training/`: `derive_seeds(random_state, components)` in `composite/ensemble/__init__.py:109` (sha256 of `f"{random_state}::{c}"`, first 4 bytes unpacked little-endian as unsigned 32-bit -- range `[0, 2**32)`); `_derive_seed(master_seed, key)` in `honest_diagnostics.py:70` (blake2b digest_size=4, interpreted big-endian, then `% (2**31 - 1)` -- range `[0, 2**31-1)`); and `_per_target_seed(base_seed, target_name)` in `baselines/_dummy_baseline_compute.py:37` (blake2b digest_size=4 of the target name only -- not the base seed -- masked to 16 bits, then added to `base_seed` and folded with `& 0x7FFFFFFF`). All three exist for the same underlying reason (avoid correlating multiple "independent" random draws that all trace back to one `random_state`), all three are well-commented and individually correct, but they disagree on hash algorithm, output range, and whether the base seed participates in the hash input (`derive_seeds`/`_derive_seed` hash `master_seed` together with the key; `_per_target_seed` hashes only the key/target name and adds the base seed afterward -- a materially different construction with different collision/correlation properties). `derive_seeds`'s own docstring already flags that it is currently unused in the actual discovery random draws ("discovery does NOT yet thread these sub-seeds into its randomness sources"), so this is a real but not yet load-bearing duplication for that one helper -- worth noting as the more benign reading. Suggested fix: consolidate on one canonical `derive_seed(master_seed, key) -> int` helper (e.g. promote `_derive_seed` or `derive_seeds` to a shared location such as `mlframe.core` or `mlframe.training.utils`) and have the other two call sites use it, or explicitly document why three different constructions are required (no such justification currently exists in any of the three docstrings).

**F8** -- `feature_engineering/numerical.py:761`: `def compute_numaggs_parallel(df: pd.DataFrame = None, ...)`. The default `None` is a legal, presumably intentional runtime value (the function is later used with `values=` instead of `df=` in some call patterns per the docstring's example), but the type hint claims `df` is always a `pd.DataFrame`, which is false at the type-checker level and will produce a false-negative under `mypy` for any caller passing `None` explicitly. This was the only live implicit-Optional hit found in the ~10-file spot-check band covering `models/`, `calibration/`, `preprocessing/`, `feature_engineering/`, `metrics/`, `evaluation/`, `core/`, `estimators/` (two other textual matches in `models/tuning.py:595,779` are inside a comment, not live code). Suggested fix: `df: Optional[pd.DataFrame] = None`.

**F9** -- `mlframe/metrics/__init__.py` re-exports the fast metric functions (`fast_roc_auc`, `fast_brier_score_loss`, `fast_r2_score`, `fast_mean_squared_error`, `compute_probabilistic_multiclass_error`, etc.) through `from mlframe.metrics.core import *` plus an explicit computed `__all__`, establishing `mlframe.metrics` as the intended public facade. In practice, roughly 20 call sites sampled across `calibration/`, `estimators/`, `evaluation/`, and `feature_selection/wrappers/rfecv/` (both `_benchmarks`-excluded and real code, e.g. `calibration/probabilities.py:13`, `calibration/quality.py:36`, `estimators/early_stopping.py:53`, `estimators/pipelines.py:69`, `evaluation/reports.py:29,574,632,706`, `evaluation/bootstrap.py:699`, `evaluation/adversarial_feature_audit.py:79`) reach directly into `mlframe.metrics.core` instead. `core.py` is not underscore-prefixed so this is not a "private module" violation in the strict sense, but it does mean the facade's curated `__all__` (which the codebase's own comment says was added specifically to "preserve the historical star-import surface") is not actually where most of the codebase gets these symbols from -- so a future rename/move inside `metrics.core` that keeps the facade's `__all__` intact would still silently break the majority of call sites, defeating the purpose of having a facade. Suggested fix: either have the facade re-export be the documented/enforced convention (grep-based lint rule or a note in CONTRIBUTING) and update the ~20 sites to `from mlframe.metrics import ...`, or drop the pretense of a curated facade if direct submodule access is the actual intended API.

## Coverage notes

- **LOC actually read vs. LOC in scope**: this cluster's brief explicitly authorizes broad Grep/Glob sampling over exhaustive line-by-line reading ("you are not expected to deep-read every file line-by-line"). 17 files (7345 LOC) were opened and read with enough context to support the findings above; the remaining 1789 files (744873 LOC) were covered only by the targeted structural greps listed in Scope, not read for content. Any correctness bug (as opposed to a structural/API finding) inside those unread files is out of this cluster's coverage by design -- that is the job of the per-subsystem clusters.
- **Test-coverage checklist item (9)** was not executed as a full mirror-check (see PR2) -- doing it properly (enumerate every public symbol in scope, then check for a corresponding test file/function) is a distinct, large task better suited to its own pass than a partial sample that could understate or overstate coverage gaps.
- **mypy-cleanliness spot-check (item f)** covered `: T = None` and `: object` patterns via full-tree grep (not just 10-15 files as suggested, since the full-tree grep was cheap and more reliable than sampling) but did not run `mypy` itself against the tree -- no attempt was made to verify overall mypy-clean status, only the two specific patterns named in the brief were checked.
- **Broad `except Exception:` and `.astype(int)`-truncation sweep** was run only against `calibration/`, `metrics/`, `preprocessing/` (the three packages item (a) names as "lower-level") as a sanity check while investigating the layering question, not as an exhaustive cross-cutting finding category -- the 9 bare `except Exception:` hits found there were not individually triaged (each needs its surrounding function's error-recovery contract inspected, which is subsystem-cluster-level work, not structural). Listed here so it isn't silently dropped: `calibration/ensembling.py:177`, `calibration/post.py:67`, `metrics/classification/_ordinal_cutpoints.py:122`, `metrics/_core_auc_brier.py:130`, `metrics/_core_numba_warmup.py:149`, `metrics/_gpu_metrics.py:217`, `preprocessing/cleaning.py:474`, `preprocessing/transforms.py:146,202`.
- **Public-API fit/predict signature census** (`def fit(self` grep, ~55 matches across the non-excluded tree) found the convention is overwhelmingly `fit(self, X, y, sample_weight=None, **kwargs) -> "Self"`, with one clearly-documented, intentional exception (`training/composite/sklearn_compat.py:232`, `CompositeTargetTransformer.fit(self, y, X=None)` -- deliberately mirrors sklearn's `TransformedTargetRegressor` convention where the transformer is fit on `y`, with an explicit docstring explaining the reversed argument order) and a few classes taking an additional required positional beyond `sample_weight` for genuinely extra required inputs (e.g. `composite/gated_regression_mixture.py:127` `subpop_label`, `composite/panel.py:210` `entity_id`, `composite/dual_direction.py:96` `scale_y`) which were not flagged as inconsistency findings since each is a distinct algorithm requiring genuinely distinct input, not an arbitrary naming drift.
