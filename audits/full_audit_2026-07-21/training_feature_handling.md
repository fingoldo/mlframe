### F1 -- WoE k-fold unseen-category fallback can emit `-inf`/`nan`

`_kfold_encode` (`target_encoders.py:766-789`) computes, per validation fold, `p = wp.get(c, prior_t)` and `q = wn.get(c, 1.0 - prior_t)` for a category unseen in that fold's training rows, then `out[j] = log(p) - log(q)`. `prior_t` is the RAW (unsmoothed) fold target mean -- if the training fold happens to contain no positives (`prior_t == 0.0`) or no negatives (`prior_t == 1.0`), which is realistic with small `cv` on an imbalanced target, `log(0)` is `-inf`. This poisons the OOF-encoded training column with `-inf`/`nan` for every row of the unseen category in that fold, silently, with no error. Contrast with the (correctly fixed) full-train-stat transform path `_encode_per_row` (lines 854-877), which explicitly clips the unseen-category probability into `[1e-12, 1-1e-12]` before taking logs, citing exactly this failure mode ("Wave 47... with smoothing=0 a category that has zero positives OR zero negatives in train yields p==0 or q==0... clip with the same Laplace cushion"). That fix was never applied to the k-fold OOF path. `tests/training/test_target_encoders.py`'s `test_fit_transform_returns_correct_shape` does assert `np.isfinite(out).all()` for `woe`, but its `synthetic_train` fixture (200 rows, 5 categories, balanced random binary target, `cv=5`) is not adversarial enough to hit an all-0/all-1 fold, so the gap is untested. Suggested fix: apply the same probability clip inside `_kfold_encode`'s woe branch that `_encode_per_row` already uses.

### F2 -- `ordered_target_encode_batch` ignores a `Generator` `random_state`

`ordered_target_encode_batch` (`ordered_target_encoder.py:109-180`) computes `base_seed = random_state if not isinstance(random_state, np.random.Generator) else None` (line 154), then calls `np.random.SeedSequence(base_seed).spawn(...)`. When a caller passes their own `np.random.Generator` (a type the function's signature explicitly accepts, `Union[int, np.random.Generator]`), `base_seed` becomes `None`, and `SeedSequence(None)` draws OS entropy -- non-reproducible -- on every call. The single-column sibling `ordered_target_encode` correctly reuses the caller's `Generator` directly (`ordered_target_encode.py:97`: `rng = random_state if isinstance(random_state, np.random.Generator) else np.random.default_rng(random_state)`). No test in `tests/training/feature_handling/test_biz_val_ordered_target_encoder_batch.py` passes a `Generator` instance for `random_state` -- only plain ints -- so the gap is untested. Suggested fix: when `random_state` is already a `Generator`, derive column-specific generators from it deterministically (e.g. via `Generator.spawn` if available, or hash the column index into a `SeedSequence` built from the generator's own bit stream) instead of discarding it.

### F3 -- Disk-tier cache never evicts

`cache.py`'s module docstring (lines 30-34) documents disk-tier eviction: "Disk: LRU within `cache.disk_evict_when_free_below_gb` / `cache.disk_min_free_gb` thresholds." `CacheConfig` (`config.py:104-105`) defines both fields with real defaults (50 GB / 5 GB). But `FeatureCache._write_to_disk` / `_read_from_disk` (`cache.py:325-364`) never reference either field, and no eviction routine exists anywhere in `cache.py` for the disk tier -- only `_evict_if_needed_locked` (`cache.py:246-277`), which acts exclusively on `self._mem` (the in-memory tier). A long-running suite with `cache.persistence in ("auto", "read_write")` -- precisely the scenario `CacheConfig.persistence`'s own docstring recommends it for ("long-running prod suites where the 10+ minute transformer inference cost amortises over re-runs") -- will grow the disk cache directory without bound until the filesystem fills up, contradicting the documented safety net. Suggested fix: either wire disk-tier writes through a policy that checks free space / directory size against the two configured thresholds (the already-implemented `LocalDiskBackend._evict_to_caps` in `cache_backend.py:251-303` does effectively this and could be reused), or update the docstring/config to mark disk eviction as not-yet-implemented.

### F4 -- `group_columns` ("group-aware encoding") is a documented no-op

`axis.py:70-72` states the `HandlerSpec` ABC contract requires subclasses to "declare `group_columns: Optional[List[str]] = None` for group-aware encoding (ABC contract is symmetric across cat and text)". `CatHandlerSpec` does declare it (`handlers.py:254`, commented `# group-aware encoding`). But: (1) `TextHandlerSpec` (`handlers.py:166-188`) has no `group_columns` field at all, directly violating the ABC's own "symmetric across cat and text" claim; (2) nowhere in `apply.py`'s cat-axis handling loop (`apply.py:296-330`, the only place `CatHandlerSpec` instances are consumed) is `cat_spec.group_columns` ever read; (3) `CustomHandler.__init__` (`custom_handler.py:69-79`) accepts and stores a `group_columns` parameter but never uses it in `fit`/`transform`, and its only caller (`apply.py:271`) doesn't even pass it. A user who configures `CatHandlerSpec(method="target_mean", group_columns=["region"])` expecting per-group target statistics gets ordinary global-fold target encoding with zero warning that grouping was ignored. Suggested fix: either implement group-aware fitting in `LeakageSafeEncoder`/the cat-axis loop, or raise/warn when `group_columns` is set but unsupported, and add the missing field to `TextHandlerSpec` for genuine ABC symmetry.

### F5 -- Target-encoder cache key lacks column-content disambiguation (asymmetric vs. the text-encoder fix)

`_apply_text_encoder`'s `InMemoryKey.train_idx_token` (`apply.py:396`) is deliberately built from `_text_column_content_token(train_df, column)` -- a content-derived hash of the specific column -- with an explicit comment explaining why: "discriminate different OD masks / row subsets even when `train_id` (id(df)) coincides... Literal `0` collided across OD-masked vs full-train fits." `_apply_target_encoder`'s `InMemoryKey.train_idx_token` (`apply.py:478`), by contrast, is built only from `_target_content_token(train_target)` -- the target array's content, not the categorical column's. If two calls share `id(train_df)` (recycled or a mutated-in-place frame) and an identical `train_target` but the categorical column's actual values differ (e.g. two different binnings/recodings of the same column name scored against one fixed target, or an OD mask that happens to preserve the target values but changes the column), the cached `(encoder, oof_train)` from one call would be silently replayed for the other's column, producing wrong encodings with no error. This is exactly the failure class the text-encoder path was hardened against; the target-encoder path was not given the same treatment. Suggested fix: fold a column-content token (reusing `_text_column_content_token`'s sampling strategy) into `_apply_target_encoder`'s cache key alongside the existing target-content token.

### F6 -- `PolarsNativeDispatcher` docstring overclaims a "try-invoke" probe it doesn't implement

`polars_capability.py`'s module docstring (lines 4-10) and the `_BLUEPRINT_OPS`/`_PDS_TOPLEVEL_OPS` in-code comment (lines 35-37, "Each entry: (capability_key, probe_lambda)... the probe accepts a Blueprint or pds module and runs the actual call on a tiny synthetic frame") both describe a "try-invoke" capability check specifically because plain `hasattr` is "brittle (a method may exist but raise NotImplementedError on certain dtypes / platforms)". The actual implementation (`detect_polars_ds_capabilities`, lines 112-118) does exactly the rejected-as-brittle thing: `if hasattr(Blueprint, op): caps.add(...)`. `_BLUEPRINT_OPS`/`_PDS_TOPLEVEL_OPS` are plain string tuples, not `(key, probe_lambda)` pairs as the comment claims. A consumer calling `dispatcher.has("blueprint.polynomial_features")` can get `True` for an op that exists on the class but actually raises at call time on the caller's data/platform -- exactly the failure mode the docstring says is guarded against. Suggested fix: either implement the described try-invoke probe on a tiny synthetic frame, or update the docstring/comments to describe the actual (cheaper, hasattr-only) contract so callers know they still need to handle a runtime failure.

### F7 -- `cb_native_only` preset never gives CatBoost native categorical handling

`presets.py`'s `cb_native_only()` docstring (lines 55-67) states it is "Equivalent to" a config with `default_cat=[CatHandlerSpec(method="native")]`. The actual returned config (lines 71-80) sets `default_cat=[CatHandlerSpec(method="ordinal", params=NoParams(kind="ordinal"))]` for every model, with no `cb`-specific cat override (only `text` gets a `cb`-specific override to `method="native"`, line 76). `compat.py:48` confirms `"native"` is a real, distinct, valid `cb`-axis-CAT method separate from `"ordinal"`. So a user calling `cb_native_only()` -- whose entire stated purpose is native handling for CatBoost -- gets CatBoost's categorical columns ordinal-encoded, not passed through natively, silently defeating the preset's name and documented behavior for every model including `cb`. Suggested fix: add a `cb`-specific `ModelHandlingOverride(cat=[CatHandlerSpec(method="native", ...)])` alongside the existing text override, and fix the docstring to match whichever behavior is intended.

### F8 -- WoE Laplace-alpha benchmark sweeps a parameter the WoE method ignores

`_benchmarks/bench_woe_laplace_alpha.py:63` constructs `LeakageSafeEncoder(method="woe", smoothing=alpha, random_state=0)` and sweeps `alpha` over `[0.5, 1.0, 2.0, 5.0, 10.0, 20.0]` to determine the best WoE Laplace cushion. But `target_encoders.py` confirms (via exhaustive grep for `self.smoothing`) that the WoE code path -- `_compute_woe_per_category` and the WoE branch of `_encode_per_row` / `_kfold_encode` -- never reads `self.smoothing`; only `self.woe_smoothing` (a separate constructor parameter, defaulted to `0.5` when unset) controls WoE's Laplace cushion. Since `.fit()` (used by the bench, not `.fit_transform()`) has no K-fold randomness either, every `alpha` value in the sweep produces a bit-identical `LeakageSafeEncoder` internal state and therefore identical `transform()` output -- the bench measures no signal at all for the parameter it claims to be tuning. `target_encoders.py:472-477`'s comment cites this exact bench file's result ("0.5 wins 9/15 cells, +0.06 AUC on 1pct") as the empirical justification for the production default `self.woe_smoothing = 0.5` (line 477). That justification is not reproducible from the checked-in bench script as written today. Suggested fix: change line 63 to `LeakageSafeEncoder(method="woe", woe_smoothing=alpha, random_state=0)` and re-run the sweep to obtain a real result; until then, treat the `woe_smoothing=0.5` default as unvalidated by this bench (it may still be a reasonable choice, just not evidenced by the cited numbers).

### F9 -- `LocalDiskBackend` is fully implemented but never used

`cache_backend.py` implements a complete `CacheBackend` Protocol + `LocalDiskBackend` with atomic writes, PID-aware cross-process locking, and a working LRU-eviction sidecar (`_evict_to_caps`, lines 251-303) gated on `max_entries`/`max_size_mb`. The module docstring in `__init__.py` (lines 15-22) frames this as the pluggable seam future backends (S3/GCS/NFS) will slot into. But `FeatureCache` (`cache.py`) never imports or instantiates `LocalDiskBackend`/`CacheBackend` -- its disk tier (`_disk_dir`, `_path_for`, `_read_from_disk`, `_write_to_disk`, `cache.py:305-364`) is a separate, hand-rolled implementation using `atomic_write_bytes` directly. This duplicates logic (atomic writes, path safety) across two independent code paths and means the "pluggable backend" promise is currently false for the only real consumer -- swapping in an S3 backend would require rewriting `cache.py`'s disk tier rather than implementing a new `CacheBackend`. It also means `LocalDiskBackend`'s working eviction logic sits unused while `FeatureCache`'s own disk tier has none (see F3). Suggested fix: route `FeatureCache`'s disk tier through `CacheBackend`/`LocalDiskBackend`, or remove/relabel the abstraction if it's genuinely still a future-phase placeholder.

### F10 -- `PolynomialFeatureExpander` never dispatches to polars-ds despite claiming to

`polynomial.py`'s module docstring says the class "Wraps either `sklearn.preprocessing.PolynomialFeatures` or polars-ds `Blueprint.polynomial_features` when available." The constructor builds `self._dispatcher = PolarsNativeDispatcher(prefer_polarsds=prefer_polarsds)` (line 65), but `self._dispatcher` is never referenced again anywhere in the file -- `fit()` (lines 84-174) unconditionally imports and uses `sklearn.preprocessing.PolynomialFeatures`. The `prefer_polarsds` constructor parameter is therefore a complete no-op: passing `True` or `False` produces identical behavior. (Contrast with `text_encoder.py`, which honestly documents the same situation with an explicit `TODO` and a docstring caveat -- "Phase C v1: sklearn-only fallback... audit confirmed" -- `polynomial.py` makes no such disclaimer.) Suggested fix: either wire the dispatcher in (branch on `self._dispatcher.has("blueprint.polynomial_features")`) or drop the dead `_dispatcher`/`prefer_polarsds` and correct the docstring.

### F11 -- `fingerprint_df` per-process memo ignores `n_sample`

`_fp_cache_key` (`fingerprint.py:85-99`) builds the memo key from `(id(df), len(cols), hash(tuple(cols)))` only. `fingerprint_df` (lines 286-425) accepts a public `n_sample: int = 4096` parameter that changes which rows get sampled into the content hash, but the memo lookup/store (`_fp_cache_get`/`_fp_cache_put`, only skipped when `columns is not None`, not when `n_sample` differs from a prior call) does not include `n_sample` in the key. Two calls on the identical `df` object with different `n_sample` values within one session would incorrectly return the first call's cached `ContentFingerprint` for the second call. No current caller in `src/` passes a non-default `n_sample` (both call sites use the default), so this is a latent gap rather than an active bug today. Suggested fix: fold `n_sample` into `_fp_cache_key`, or skip the memo whenever `n_sample != 4096` (the function's own default).

### F12 -- `torch_dtype` type inconsistency in the CUDA-context-lost CPU fallback

`hf_provider.py:192` computes `torch_dtype = torch.float16 if dtype_str == "fp16" else torch.float32` (a real `torch.dtype` object) for the primary `acquire()` load path (line 227). The CUDA-context-lost recovery path (`_batched_inference`, line 404) instead passes the literal string `torch_dtype="float32"` to a fresh `AutoModel.from_pretrained(...)` call. Whether `transformers.from_pretrained` accepts a bare dtype-name string for `torch_dtype` (vs. requiring `torch.dtype` or the literal string `"auto"`) is version-dependent; if the installed `transformers` version doesn't accept it, this already-rare recovery path (only reached after a driver-level CUDA context loss) would itself raise, uncaught, instead of falling back to CPU as the surrounding code and warning message promise. Flagging as unconfirmed since I did not execute this path against a live `transformers` install. Suggested fix: use `torch.float32` (matching line 192's pattern) instead of the string literal for certainty across versions.

F1, F2, F3, F5, F8, F11, F12 would benefit from an actual runtime repro to fully confirm the exact numeric/exception behavior; I've noted F12 explicitly as unconfirmed for this reason, and F1/F8/F11 are logically airtight from the code alone (no branch exists that could avoid the described outcome) even without running them.
- Downstream consumers of this cluster (`mlframe.training.core.*`, which actually wires `feature_handling_apply`/`FeatureHandlingConfig` into the real training loop) are out of scope per the cluster boundary; I did not verify whether any of F3/F4/F9's gaps are silently compensated for by code outside `training/feature_handling/**` (e.g. a separate disk-cleanup cron, or a `group_columns`-aware wrapper elsewhere in `training/core`). I checked the two most plausible spots (`_phase_train_one_target_body.py` et al. for `group_columns`) via grep and found no such compensation, but a full downstream audit was not performed.
- `src/mlframe/feature_selection/filters/**` and `src/mlframe/feature_selection/shap_proxied_fs/**` (and their test mirrors) were excluded per instructions; I did not read them. Where this cluster's code imports a symbol from those packages I did not find any such import (this cluster has no dependency on them).
- The excluded MRMR/shap_proxied audit's 143-finding log (`audits/mrmr_audit_2026-07-20/`) was not consulted, since it covers different files.
