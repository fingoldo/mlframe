### P0 findings

**F1 -- `compute_cross_sectional_neighbor_features` corrupts output when snapshot count <= k.** `knn_search` is queried for `k_query = min(k_max + 1, n_snapshots)` neighbors, then self-matches are masked to `-1` (index) / `inf` (distance) and sorted so self drops to the tail (`cross_sectional_neighbors.py:113-124`). When `n_snapshots <= k_max` (very plausible with the default `k=10` on a dataset with few distinct snapshot values, or in any small test fixture), there are fewer real neighbors than `k_max`, so the sliced `neighbor_idx_sorted`/`dists_sorted` arrays are padded with the placeholder `-1`/`inf` values instead of being masked out. `_snapshot_stats_for_k` then does `vectors[idx_k, col_j]` (`cross_sectional_neighbors.py:47`), and numpy's fancy indexing silently treats `-1` as "last row" -- pulling an unrelated snapshot's feature values into the neighbor-aggregate mean -- while `distance_ratio = first_nn / kth_nn` (`cross_sectional_neighbors.py:53-57`) divides by `inf`, collapsing to ~0, the *opposite* of the documented "near 1 for an isolated snapshot" semantics. Verified directly: with 5 snapshots and the default `k=10`, every row's `xsnn_f0_mean` output (28.0, 26.0, 24.0, 22.0, 20.0) is visibly not each snapshot's own neighborhood mean, and `xsnn_distance_ratio` is exactly `0.0` for every single row. All of this module's existing tests (`tests/feature_engineering/test_biz_val_cross_sectional_neighbors.py`) use `n_snapshots` from 20-1500 against `k` from 5-10, i.e. `n_snapshots` is always comfortably `> k`, so this path is untested. Fix direction: mask out `idx_k == -1` (e.g. via `np.where` to NaN before the reduction) and special-case `kth_nn` being `inf`/insufficient into the same "maximally isolated" `1.0` fallback the `kth_nn == 0` case already gets.

**F2 -- `boolean_pair_interactions` silently mis-casts NaN into wrong boolean logic.** `is_binary_column` (`boolean_pair_interactions.py:26-31`) explicitly tolerates NaN (`series.dropna()` before checking `{0, 1}` membership) and is the function's own auto-detection default. But the caller then does `df[c].to_numpy(dtype=np.int8)` (`boolean_pair_interactions.py:77`) with no NaN handling. Verified directly: for columns `a=[1,0,1,NaN,0]`, `b=[0,1,1,0,NaN]`, both pass `is_binary_column` as `True`, and the resulting AND/OR/XOR columns are deterministic-looking but silently wrong at every row containing an input NaN (row 3: `a=NaN,b=0` yields `and=0, or=0, xor=0` -- NaN was cast to some int8 sentinel rather than propagated), with only a `RuntimeWarning: invalid value encountered in cast` (easily filtered/lost in a real pipeline) rather than a raised error or a NaN in the output. None of the module's tests (`tests/feature_engineering/test_biz_val_boolean_pair_interactions.py`) exercise a column containing NaN. Fix direction: either raise if any selected/detected column has missing values, or NaN-propagate the affected output rows explicitly before casting to int8.

**F3 -- `graph_spectral_features`'s isolated-node eigenvalues pollute the shape fingerprint.** The code computes the normalized Laplacian via `Ln = I - D^-1/2 A D^-1/2` with `dinv_sqrt` zeroed for degree-0 nodes (`graph_spectral_features.py:119-121`), and the inline comment at line 118 claims "isolated nodes contribute a 0 (own component)". This is mathematically wrong for THIS convention: an isolated node's row/column becomes the standard basis vector `e_i` (diagonal entry 1, all else 0), contributing an eigenvalue of exactly **1**, not 0 -- a well-known property of the symmetric-normalized Laplacian (matching `scipy.sparse.csgraph.laplacian(normed=True)`'s own convention). Because `nontrivial = evLn[evLn > zero_tol]` (line 125) only filters values near 0, the isolated node's eigenvalue=1 is *not* filtered and is treated as a genuine structural eigenvalue, displacing real small eigenvalues out of the top-k fingerprint window. Verified directly: for a triangle (nodes 0-1-2, true smallest non-trivial normalized eigenvalue 1.5) plus one isolated node 3, `graph_spectral_features(4, edges, k=5)` returns `norm_lap_eig_1 = 1.0` (the isolated node's spurious artifact) ahead of the triangle's genuine `1.5`. The existing test `tests/feature_engineering/test_graph_spectral_features.py::test_components_of_disconnected_graph` uses an isolated node but only asserts `n_components`/`algebraic_connectivity` (both computed from the *unnormalized* Laplacian, which is unaffected), never inspecting `norm_lap_eig_*` in that scenario. Fix direction: exclude the `n_components`-many trivial eigenvalues by identifying isolated/degree-0 nodes directly (or by excluding eigenvalues within `zero_tol` of 1 that correspond one-to-one with isolated nodes) rather than assuming they sit near 0, and correct the comment.

### P1 findings

**F4.** `_pf_single_segment` is called once per group inside `particle_filter_posterior`'s grouped branch (`bayesian.py:487-494`) with `rng=np.random.default_rng(seed)` re-created from the SAME `seed` argument every iteration, instead of deriving a per-group sub-stream (e.g. `seed + g` or `rng.spawn()`). Every group's particle-filter noise realization is therefore drawn from an identical RNG stream, correlating the Monte-Carlo noise across all groups even though each group's underlying data differs -- a reproducibility/statistical-independence gap for a feature whose whole purpose (per its docstring) is "each group gets its own particle ensemble". Fix direction: derive a distinct sub-seed per group, e.g. `np.random.default_rng([seed, g])`.

**F5.** `compute_mps_targets` (`mps.py:683-751`) builds `targets_dfs` per group and returns `pl.concat([el for el in targets_dfs if el is not None and len(el) > 0])` unconditionally. If every group is skipped (all `final_prices[0] is None`, i.e. every group's price history is entirely null, or every group has <2 rows so its per-group output is 0-length), `targets_dfs` filters down to an empty list and `pl.concat([])` raises, even though the function's own signature promises `Optional[pl.DataFrame]`. `safely_compute_mps` happens to swallow this via its broad `except Exception`, but `compute_mps_targets` is itself public (in `__all__`) and callers invoking it directly get an uncaught crash instead of `None`. Fix direction: return `None` (or an empty typed frame) when `targets_dfs` is empty, before the `pl.concat` call.

**F6.** `_peak_trough_counts` (`windowed_shape.py:353-365`), used by `rolling_n_peaks`/`rolling_n_troughs`/`rolling_extrema_density`, compares `wmid > wleft`/`wmid < wright` with no NaN handling at all. A NaN comparison silently evaluates to `False`, so a NaN anywhere inside a window silently undercounts peaks/troughs (never raises, never yields NaN) instead of propagating a "can't compute" signal. Every other rolling function in the SAME file (`rolling_total_variation`, `rolling_zero_crossings`, `rolling_quantile_spread`, `rolling_shannon_entropy_binned`) explicitly checks `np.isfinite(...).all()` and either falls back to a NaN-safe numpy path or documents the fallback semantics in its docstring; `_peak_trough_counts` has neither. Fix direction: add the same `np.isfinite` gate (or document/propagate NaN explicitly) as the sibling functions.

**F7.** `local_linear_detrend`'s `_fit_segment` (`stationarity.py:325-350`) does `seg_f = np.where(np.isfinite(seg), seg, 0.0)` before running the rolling-OLS sliding-window fit, with no mention of this in either the outer docstring or the inner `_fit_segment` docstring. Injecting 0.0 for a missing observation directly biases the fitted slope/intercept toward the imputed zero rather than propagating NaN or excluding that row from the window's regression -- a materially worse failure mode than the same zero-fill trick used (and explicitly documented) in `ewma_residual`'s `_ewma_single` in the very same file, since here it distorts a *fitted trend*, not just one smoothed value. Fix direction: document the behavior explicitly (matching `ewma_residual`'s convention), or better, exclude NaN rows from the window's covariance sums instead of zero-filling them.

**F8.** `cusum_features`'s default-threshold branch (`stationarity.py:391-399`) computes `mad`/`threshold` from `arr` -- the FULL, ungrouped input array -- even when `group_ids` is supplied and the rest of the function (and the module's own header docstring: "Per-group support ... avoiding bleed across group boundaries") is built around strict per-group isolation. For a panel where groups differ substantially in scale (e.g. different tickers/sensors), one shared global MAD-derived threshold is applied to every group's CUSUM walk, which can make the change-point signal meaningless for groups far from the global scale (near-never-resetting for low-scale groups, near-instant-resetting for high-scale ones). Fix direction: when `threshold is None` and `group_ids` is given, compute the MAD per group instead of once globally.

**F9.** `_bands_for` (`spectral.py:38-58`) computes the default lo/mid/hi band edges via integer-division formulas (`n_freq // 6`, `n_freq // 2`) that only produce a valid, non-overlapping, DC-excluding 3-way split for large-enough `window_K`. At `window_K=10` (`n_freq=6`): band1=`(1,2)`, band2=`(1,3)` -- band2 overlaps band1 at bin 1 and starts where band1 starts, rather than at the mid-frequency region; at very small `K` the mid band can even include bin 0 (DC), which the module docstring explicitly claims is always skipped. This silently double-counts/omits spectral energy in `rolling_spectral_band_energies` and `rolling_hf_lf_ratio` for any caller using a small `window_K` (a legitimate choice for short lookback windows). Fix direction: validate/guard `window_K` below some minimum for the default 3-band split, or derive the split from `n_freq - 1` (post-DC count) with explicit non-overlap invariants.

**F10.** `rolling_spectral_rolloff` (`spectral.py:293-328`) computes `total = cum[:, -1:] + 1e-12` then `targets = percentile * total` and `idx = (cum >= targets).argmax(axis=1)`. At `percentile=1.0` (allowed by the function's own validation `0.0 < percentile <= 1.0`), `targets == cum[:, -1] + 1e-12`, which is strictly greater than the true (un-perturbed) cumulative total at every bin including the last -- so `cum >= targets` is `False` everywhere, and `argmax` of an all-`False` row returns index `0`. The function then silently reports rolloff bin `0` for every window instead of the mathematically correct answer (the last bin, `n_freq - 1`), for the boundary value the caller's own validation explicitly permits. Fix direction: compare against `cum[:, -1:] * percentile` without the `+1e-12` on the target side, or clamp `idx` to `n_freq - 1` when no bin satisfies the inequality.

**F11.** `_windowed_stats_by_time_njit` (`entity_inter_event.py:109-149`) accumulates `total`/`total_sq` in a single sliding pass and derives `var = total_sq/cnt - m*m`, the classic "sum-of-squares minus square-of-sum" formula that is prone to catastrophic cancellation for larger-magnitude inputs (its own `if var < 0.0: var = 0.0` guard is a symptom of exactly this risk). Its sibling `_windowed_stats_by_count_njit` (used for `window_size`) instead does a genuine two-pass mean-then-deviation computation, which is numerically stable. Both kernels are dispatched from the same public function (`_windowed_group_stats`) for both the `time_since_prev` delta feature AND the (potentially large-magnitude) `value_col` feature, so a caller using `window_time` on a large-magnitude `value_col` (e.g. transaction amounts) gets a materially less precise (and, at pathological scale, near-zero-clamped) std than the same caller would get using `window_size` on the identical data. Fix direction: rework the time-windowed kernel to a numerically-stable incremental variance (e.g. Welford-style add/remove) rather than the raw sum-of-squares recurrence.

**F12.** `latent_interaction_features` (`latent_interaction_svd.py:221-223`) computes `n_eff = min(n_components, max(1, min(n_rows, n_cols) - 1))`. When the interaction matrix degenerates to a single distinct row-entity OR a single distinct column-entity (`min(n_rows, n_cols) == 1`), this yields `n_eff = max(1, 0) = 1`, which is passed to `TruncatedSVD(n_components=1)` on a matrix with 1 feature. Confirmed empirically: `TruncatedSVD(n_components=1).fit_transform(scipy.sparse.random(1, 1, ...))` raises `ValueError: Found array with 1 feature(s) ... while a minimum of 2 is required by TruncatedSVD`. A single-item item-catalog or a heavily-filtered small dataset can realistically hit this. Fix direction: raise a clear, actionable `ValueError` earlier when `min(n_rows, n_cols) < 2` (SVD is meaningless there anyway), rather than letting sklearn's internal error surface.

**F13.** `_grouped_ewma_njit` and `_grouped_ewma_adaptive_njit` (`ewma_multi_alpha_features.py:28-93`) contain no NaN handling whatsoever: `out[i] = alpha * values_sorted[i] + one_minus * out[i-1]` propagates NaN forward through `+`/`*` arithmetic once it first appears, so a single missing observation anywhere in an entity's history permanently NaN-poisons every subsequent EWMA value for that entity for the rest of its history -- not just that one row. The sibling recurrence in the SAME package, `_ewma_recurrence_njit` (`stationarity.py:37-48`), explicitly zero-fills NaN via `seg_f = np.where(np.isfinite(seg), seg, 0.0)` before running the identical recurrence pattern, and that fix is documented. This module's own docstring frames the target use case as "e.g. a binary product-presence indicator" over real per-client event histories, where missing rows are realistic. The adaptive-alpha variant compounds this: an alpha whose walk-forward SSE becomes NaN is never selected as `best_alpha` (NaN comparisons are always `False`), silently defaulting to `alphas_grid[0]` regardless of true fit quality once any NaN enters the validation window. Fix direction: zero-fill (or otherwise NaN-guard) `values_sorted` before the recurrence, matching the sibling kernel's convention.

**F14.** `nearest_past_join`'s fallback-chain loop (`nearest_past_join.py:131-156`) computes, for each tier, `resolved = OR over all attached_names of (that column's own notna & not-sparse)` (line 148-153), then uses this single combined `resolved` mask to update BOTH `matched_tier` AND `unresolved` for every row (lines 155-156). When `right_value_cols` has 2+ columns with different null patterns in `right_df`, a row where only ONE of the several requested value columns got a match at the current (fine) tier is marked fully "resolved" and removed from `unresolved` -- so it is never retried at a coarser tier for its OTHER, still-NaN columns, even though a coarser tier might have filled them. This directly contradicts the function's own documented contract: "``NaN`` where no eligible past row exists for that group, across every tier tried" implies per-column exhaustion of the tier chain, but the actual implementation exhausts the chain per-ROW. Fix direction: track `unresolved` per (row, column) pair (or loop the fallback chain once per value column) instead of a single row-level `resolved` flag.

**F15.** `_select_significant_lags_per_group` (`acf_lag_selection.py:96-149`) computes `pacf_by_lag = {lag: total / n_groups_scored for lag, total in lag_pacf_sums.items()}` (line 141), dividing every lag's PACF sum by `n_groups_scored` -- the TOTAL number of groups that met `min_group_size`, regardless of how many of those groups actually had enough points (`min(max_lag, group_series.shape[0] - 1)`) to score that particular lag. The function's own docstring explicitly promises `pacf_values` is "the group-wise average PACF among groups where the lag was scored" -- but for panels with heterogeneous group lengths (a common real scenario: some entities have short histories, others long), higher lags scored by only a few long groups get silently diluted by dividing by the full `n_groups_scored` count, systematically under-estimating `pacf_by_lag` at those lags relative to what the docstring describes. Fix direction: track a per-lag scored-group counter (incrementing only when that lag's key is present in `group_result["pacf_values"]`) and divide by that instead of the fixed `n_groups_scored`.

### P2 findings

**F16.** `find_best_mps_sequence`'s docstring (`mps.py:194-203`) documents a 3-tuple return: "positions ... total_profit ... cumulative_profits", but the function body only ever `return`s a 2-tuple, `positions, profits` (`mps.py:279`), matching every one of its callers. The docstring is simply stale relative to the actual contract.

**F17.** Neither `find_best_mps_sequence` nor its public wrapper `find_maximum_profit_system` (`mps.py:185-192`, `328-336`) document the `shift` parameter at all, despite it materially changing the returned positions (shifting them left and backfilling). A caller reading only the docstring has no way to know this parameter exists or what it does.

**F18.** `compute_numaggs`'s `return_lintrend_approx_stats` recursive call (`numerical.py:553-576`) builds its `params` dict with `return_float32=True` hardcoded, regardless of what the OUTER caller passed for `return_float32`. When the outer caller requested `return_float32=False` (full float64 precision), the recursive lintrend-approx block is still computed and rounded through a `np.float32` array before being folded back into the (otherwise float64) output tuple -- a silent, undocumented precision downgrade for that subset of columns only.

**F19.** `rolling_integral_above_baseline` (`windowed_shape.py:646-650`) contains:
```python
if baseline is not None and baseline_fn != "median":
    # The two are mutually exclusive -- one chosen explicit baseline
    # wins, fn is only used when no array supplied.
    pass
```
This branch does nothing (`pass`) and has no observable effect on behavior (the real dispatch happens later via a separate `if baseline is not None:` check) -- it's dead, confusing code that reads like it should warn or validate but doesn't.

**F20.** `rolling_spectral_centroid`, `rolling_spectral_bandwidth`, `rolling_spectral_rolloff`, `rolling_spectral_flatness`, `rolling_spectral_flux`, and `rolling_periodicity_score` (`spectral.py:229-448`) each independently repeat the identical 3-line NaN-imputation snippet (`seg_mean = float(np.nanmean(seg)) if np.isfinite(seg).any() else 0.0; seg_f = np.where(...)`), including the identical inline comment, six times. `_detrend_zero_mean` already shows the pattern for extracting shared helpers in this file; this block is a good candidate for the same treatment.

**F21.** `predictor_disagreement_var` (`ensemble_features.py:117-120`) does `arr.var(axis=1, ddof=1) if arr.shape[1] > 1 else np.zeros(arr.shape[0])`, but `_coerce_preds` (called immediately above) already raises `ValueError` when `arr.shape[1] < 2` (`ensemble_features.py:78-79`), so the `else` branch is unreachable dead code.

**F22.** `recency_weighted_rolling.py`'s module docstring and both public function docstrings never mention NaN handling. `values` is not NaN-checked anywhere in `_rolling_recency_weighted_mean_sorted`/`_rolling_recency_weighted_std_sorted`, so a NaN anywhere in the trailing window silently poisons `num`/`den` to NaN for that row (an honest failure, not a silently-wrong value, but undocumented -- a caller has no way to know from the docstring whether NaN is tolerated).

**F23.** The `long_window_weight_power` parameter (`ma_crossover.py:27, 82-88`) is, per its own extensive docstring, deliberately applied to the pair's `short` window (`float(short) ** long_window_weight_power`, line 88) -- the docstring explains this was a deliberate FIX from "an earlier, buggy version of this parameter" that weighted by `long`. The parameter's NAME was never updated to match the corrected semantics, so a caller skimming the signature (without reading the full docstring) would reasonably but incorrectly assume it weights by the LONG leg.

**F24.** `variance_gated_pairwise_diff`'s docstring (`variance_gated_pairwise_diff.py:36-38`) describes the pruning rule as "``np.var(diff) > min_variance``" (numpy's `np.var` default is `ddof=0`, population variance), but the actual optimized implementation derives `pair_var` from `np.cov(X, rowvar=False)` (`variance_gated_pairwise_diff.py:65-72`), whose default is `ddof=1` (sample variance). The identity `var(a-b) = var(a) + var(b) - 2*cov(a,b)` used in the accompanying comment is valid under either convention as long as it's applied consistently (which it is), but the actual threshold comparison silently differs from the one described in the docstring by a factor of `n/(n-1)` -- immaterial at large `n`, but undocumented and, unlike this codebase's other "optimized path replaces the naive one" comments elsewhere in the same cluster (see `_total_variation_kernel` in windowed_shape.py, or `compute_moments_slope_mi` in _numerical_numba.py), there is no note here acknowledging or bounding the delta.
