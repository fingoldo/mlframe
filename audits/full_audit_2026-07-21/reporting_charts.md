# reporting/charts (charting & visualization) -- mlframe audit

## Scope

All files under `src/mlframe/reporting/charts/` were read in full, except the `_benchmarks/` subpackage which received a lighter pass (see Coverage notes).

Total files reviewed in depth: 40. Total LOC reviewed (`wc -l` on these 40 files): 14081.

`_benchmarks/` (24 files, 1509 LOC): one file (`bench_error_bias_column_pull.py`) was read in full; the rest received a grep-only scan for the highest-risk patterns (`except Exception`/bare `except`, `eval`/`exec`/`os.system`, unsanitized `subprocess`). ### F1 -- `_reliability_curve` NaN mis-binning (calibration.py)

`build_reliability_overlay_spec` (calibration.py:513) calls `_reliability_curve(raw_probs, y, edges)` without any finite-row filter, unlike `smoothed_reliability_curve`/`bootstrap_reliability_band` earlier in the same file which explicitly do `finite = np.isfinite(s) & np.isfinite(t)`. `np.digitize` sends a NaN probability to the same overflow index as a value greater than every bin edge; after `- 1` and `clip(0, nbins-1)` that NaN lands in the LAST bin, and its (real) label is silently added to that bin's `bincount`/`weights` sum. Empirically confirmed: `p = [0.5, nan, 0.99]` -> `digitize` -> `[6, 11, 10]` -> `bin_ids` (after `-1`/clip) -> `[5, 9, 9]`, so the NaN row and the 0.99 row land in the same bin. The only production caller, `mlframe.calibration.policy._emit_reliability_plot`, passes raw OOF probabilities/labels straight through with no NaN filter (`src/mlframe/calibration/policy.py:627-638`). Fix direction: filter `isfinite(raw_probs) & isfinite(y_true)` (and each `calibrated_probs[name]` series) before binning, mirroring `smoothed_reliability_curve`.

### F2 -- Unfiltered NaN rows in three quantile panels (quantile.py)

`_reliability_panel` computes `emp[k] = mean(y <= P[:, k])` over the full unfiltered `y`/`P`; a NaN comparison is always `False`, so a NaN row is silently counted as "not covered" in the denominator instead of being excluded, deflating the reported empirical coverage. `_coverage_panel` has the identical pattern via `inside = (y >= q_lo) & (y <= q_hi)` with `n = y.shape[0]` unfiltered. `_quantile_crossing_panel` computes `rates[j] = c / n` with `n = P.shape[0]` unfiltered; a NaN prediction never registers as a crossing violation (comparison is `False`) but is still counted in `n`, diluting the reported violation rate. All three are inconsistent with `_quantile_reliability_panel` and `_pinball_decomp_panel` in the same file, which both do `row_ok = np.isfinite(y) & np.all(np.isfinite(P), axis=1)` up front. No test in `tests/reporting/test_charts_quantile.py` or `test_charts_quantile_variants.py` exercises a NaN in `y_true`/`preds_NK` (confirmed via grep: 0 hits for `nan`/`NaN`), so this is untested as well as unfixed. Fix direction: add the same `row_ok` filter (or a shared helper) to all three panels.

### F3 -- Silent broad-except fallback in the weak-segment tree ranker (error_analysis.py)

`_top_split_features` fits a `DecisionTreeRegressor` and falls back to a per-feature median-split surrogate on ANY exception, with no `logger.warning`/`logger.debug` call in the except block at all (confirmed by reading the source; the sibling `_is_multi_output_catboost` in `shap_panels.py` DOES log its except branches). `DecisionTreeRegressor.fit` tolerates `NaN` in modern sklearn (verified: sklearn 1.8.0 here fits fine with `NaN` features) but still raises `ValueError` on `Inf` (verified live: "Input X contains infinity..."). Since `weak_segment_heatmap` only filters `err`'s finiteness (not `mat`'s -- error_analysis.py:283-286), an `Inf` in any feature column silently drops the tree-based multi-way-split ranking for the whole weak-segment heatmap, with the caller having no way to know the fallback fired. Fix direction: narrow the except to what sklearn actually raises for bad input (or filter/clip `Inf` from `mat` first, matching `_bin_matrix` in `slice_finder.py` which already handles this), and log a warning on fallback.

### F4 -- Silent length-truncation instead of validation (calibration_heatmap_2d.py, calibration_by_feature.py, fairness_calibration.py)

All four call sites open with `n = min(a.size, b.size, c.size); a, b, c = a[:n], b[:n], c[:n]` with no warning. Every other length-sensitive entry point audited in this cluster (`calibration_drift.py:124-125`, `category_discriminability_table` in `category_discriminability.py:184-185`, `class_structure_matrix` in `class_structure_heatmap.py:76-77`, `separability_panel` in `engineered_separability.py:154-155`, `compose_ltr_figure` in `ltr.py:469-470`) explicitly validates and raises `ValueError` instead. A caller who passes a `feature_values`/`subgroups` array computed against a stale/different row filter than `y_true`/`y_score` gets a silently truncated (and potentially row-misaligned) calibration read instead of an error. Fix direction: raise `ValueError` on mismatch, matching the rest of the cluster's convention.

### F5 -- Silent inter-model row truncation before Spearman correlation (model_comparison.py)

`_corr_heatmap_panel` builds the correlation input via `n_rows = min(c.shape[0] for c in cols); cols = [c[:n_rows] for c in cols]`. This assumes every model's score array is already aligned row-for-row on the SAME underlying dataset; if one model was scored on a different-length subset, the truncation silently pairs unrelated rows and reports a real-looking but meaningless Spearman correlation, with nothing in the title/annotation flagging that truncation happened. Fix direction: raise on mismatch, or annotate the panel when a truncation occurred.

### F6 -- Five diagnostic submodules missing from the package facade (`__init__.py`)

`interaction_strength.py`, `prediction_stability.py`, `risk_coverage.py`, `shap_interactions.py`, and `pdp_2d.py` each define a complete, `__all__`-exported, docstring-documented composer function, but none of the five is imported/re-exported by `mlframe/reporting/charts/__init__.py`. Every other diagnostic in the cluster (including very similar late additions like `decision_curve.py` and `shap_per_instance.py`) IS re-exported. Not a functional break -- `diagnostics_dispatch.py`, `_risk_coverage_diagnostic.py`, and the tests already import these five modules directly by their private submodule path -- but it means a caller following the documented "import from `mlframe.reporting.charts`" pattern silently gets an `ImportError`/`AttributeError` for these five specifically. Fix direction: add the five modules' public composers/results to `__init__.py`'s imports and `__all__`.

### F7 -- Triplicated SHAP figure-save helper (shap_panels.py, shap_interactions.py, shap_per_instance.py)

Three independent, near-identical implementations of "resolve matplotlib format(s) via `_matplotlib_formats`, loop `fig.savefig` in a try/except that logs a warning on failure". `shap_interactions.py` and `shap_per_instance.py` already import `_matplotlib_formats` and `_close_figs` from `shap_panels.py` -- the save-figure loop itself should be imported the same way instead of being copy-pasted a third time. Fix direction: keep one `save_figure(fig, base, plot_outputs)` in `shap_panels.py` and import it from the two siblings.

### F8 -- Three files past this repo's own "carve early" LOC convention

`error_analysis.py` (886 LOC), `binary.py` (826 LOC), and `multiclass.py` (821 LOC) are all past the ~800-900 LOC point CLAUDE.md says to carve a sibling module at (still under the hard 1000-LOC backstop test). `error_analysis.py` in particular has two largely-independent halves (weak-segment-heatmap machinery vs. error-bias/worst-k/target-dist-overlay machinery) that would split cleanly. Fix direction: no immediate action needed, but the next feature added to any of these three should go in a new sibling module.
