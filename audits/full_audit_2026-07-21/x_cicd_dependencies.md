Below, one paragraph per finding with the concrete failure scenario and a fix direction.

**F1 -- duplicated tests/ pre-commit hook block, one copy on a stale ruff.** `.pre-commit-config.yaml` defines the "tests/ blocking lint bundle" (ruff, black-filtered, bandit, interrogate, codespell scoped to `files: '^tests/'`) at lines 403-456, then repeats the identical five hooks again at lines 467-520 under a second `repo: astral-sh/ruff-pre-commit` entry pinned to `rev: v0.8.6` (versus `v0.15.20` for the first copy and for `pyproject.toml`'s `dev` extra, which pins `ruff==0.15.20` specifically "matching .pre-commit-config.yaml's ruff-pre-commit rev AND CI's `uvx ruff==0.15.20`... an open range here is exactly how the prior v0.8.6 pre-commit-vs-CI drift went unnoticed"). Concretely: (1) every commit touching a `tests/` file now runs ruff/black/bandit/interrogate/codespell on that file twice, doubling that portion of pre-commit wall time for no benefit; (2) the second run uses ruff 0.8.6, a release from well before the rules/behavior the project's pinned 0.15.20 and `tests/ruff.toml` were tuned against -- it can silently pass something 0.15.20-era rules would catch, or fail on a rule/behavior that has since changed, reintroducing precisely the version-drift bug class the nearby `pyproject.toml` comment documents as already fixed. Fix: delete the second block (lines 458-520) entirely; the first copy (394-456) already covers `tests/`.

**F2 -- `ci.yml` `build` job has no `timeout-minutes`.** The `build` job runs `uv build`, `twine check`, a venv install, and a wheel-smoke-import after the full `test` matrix (`needs: test`) succeeds. It has no `timeout-minutes` set, so a hang anywhere in that chain (e.g. a `pip install` step waiting on a stalled network resolve) runs up to GitHub's default 360-minute per-job ceiling before being killed. `release.yml`'s own header comment (line 26-28) explicitly documents that its `build` job used to be "the sole omission" of `timeout-minutes` across the repo's workflows and was fixed in the 2026-07-18 architecture review -- but `ci.yml`'s `build` job was apparently missed in that same pass and still has the gap today. This matters more than usual here because the repo's own `ci.yml` matrix comments describe the account having already exhausted its monthly free Actions minutes once. Fix: add `timeout-minutes: 15` (or similar), matching `release.yml`'s own `build` job.

**F3 -- `release.yml` `publish` job has no `timeout-minutes`.** Same gap as F2, but on the job that actually uploads to PyPI via Trusted Publishing (OIDC). It only downloads a pre-built artifact and calls `pypa/gh-action-pypi-publish`, so a genuine hang is unlikely, but it is also the single highest-consequence job in the repo (an unbounded hang here would sit against the `pypi` GitHub Environment without a safety net). Fix: add a short `timeout-minutes` (e.g. 10) for consistency with the rest of the repo's own documented convention of an explicit timeout on every job.

**F4 -- Dependabot's `pip` ecosystem is fully disabled.** `.github/dependabot.yml`'s `pip` entry sets `open-pull-requests-limit: 0`, which stops Dependabot from opening any update PRs for Python dependencies (it still scans, but never proposes a fix) since 2026-07-10, per the comment, because every PR/branch it opened triggered the full CI matrix and contributed to Actions-minutes exhaustion. Concretely: if a transitive or direct pip dependency gets a CVE advisory, the *only* automated signals left are `dependency-review.yml` (which only inspects dependencies newly introduced by an incoming PR's diff, not drift in already-declared version ranges) and `pip-audit` inside `lint-advisory.yml` (explicitly advisory/`continue-on-error`, never blocks a merge or opens a fix PR). The `github-actions` ecosystem entry stays fully active, so this is an asymmetric gap specifically on the Python dependency surface. This is a documented, reasoned tradeoff (not an oversight) -- but the comment's own "Raise back to 1-3 once the Actions-minutes budget allows" has no owner or trigger date attached, so there's a real risk it silently becomes permanent. Fix direction: either raise the limit back to a small nonzero value (1-2) now that the Windows/macOS CI legs and the nightly numba-coverage job have already been dropped/gated to reclaim most of the minutes budget the original comment blamed, or add a tracked follow-up (issue/TODO with a revisit condition) so the 0-limit doesn't quietly become the permanent state.

**F5 -- inert "FUTURE SKETCH" block left in a live workflow file.** `numba-coverage.yml` lines 120-162 are ~43 lines of commented-out YAML (a full job definition prefixed with `#`) explicitly labeled "inert, not real YAML". It's harmless (never parsed, doesn't affect the active job), but it's dead content sitting in a file GitHub Actions actually executes, and a design sketch like this reads better as a tracked issue/PROJECTS.md note than as a permanent code comment. Fix direction: move the sketch's rationale into a tracked follow-up item and trim the file to the active job, or leave as-is if the team prefers workflow-file-local design notes (stated as the less-preferred but valid alternative reading).

**F6 -- nightly numba-coverage job's scheduled trigger never actually runs the job body.** The workflow declares `on.schedule: "0 3 * * *"`, but the job itself is gated `if: inputs.run`, and `github.event.inputs` is empty (falsy) for a schedule-triggered run -- so the cron fires the workflow shell every night but the actual coverage job never executes unless a human does a manual "Run workflow" dispatch with the `run` checkbox ticked. This is explicitly documented as an intentional stopgap (Actions-minutes budget) with a comment noting a meta-test pins the workflow's mere *existence* so it can't be silently deleted -- but that meta-test can't detect that the job has been effectively off for every scheduled invocation since 2026-07-10, only that the trigger declaration exists. Fix direction: since this is already flagged as "restore once budget allows," the meta-test could be strengthened to also assert `inputs.run`'s `default` is `false` intentionally (so a future accidental flip-back is visible), or a calendar/backlog reminder could be attached so this doesn't silently persist past the point where the budget constraint was actually resolved.

**F7 -- `gpu-extras-install-matrix.yml`'s cuda12x rows aren't marked experimental for 3.13/3.14 (unverified, flagged as a possible policy inconsistency).** The matrix cross-products `python-version: ["3.10"..."3.14"]` x `extra: ["gpu", "gpu-cuda11"]`, then adds `include` entries marking only `gpu-cuda11` + {3.13, 3.14} as `experimental: true`, with the file's own header comment explaining cupy has no cuda11x wheel for Python 3.13+. The `gpu` (cupy-cuda12x) row for 3.13/3.14 is left non-experimental, i.e. a real CI failure there turns the check red with no escape valve -- inconsistent with `ci.yml`'s own blanket policy of marking every 3.14 leg experimental repo-wide "since the heavy stack does not yet reliably ship cp314 wheels." I could not verify from within this sandbox (no network access) whether `cupy-cuda12x>=14.0,<15.0` currently ships a cp314 wheel; if it does, this is a non-issue and the asymmetry is correct. Flagging as a possible drift between this workflow's per-extra experimental policy and the rest of the repo's blanket per-Python-version policy, worth a one-time confirmation.
