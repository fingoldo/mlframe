### F1 -- grouped+recurrent forward misaligns `groups` against the full-length sequence (P0)

`CompositeTargetEstimator.fit()` always slices group labels to the valid rows up front: `groups_train = _extract_groups(X, self.group_column)[valid]` (`_estimator.py:635`). For transforms that are both `requires_groups=True` and `recurrent=True` (`ewma_residual_grouped`, `rolling_quantile_ratio_grouped`, `frac_diff_grouped` -- see `_registry_extended.py`), whenever `not bool(valid.all())` (any row fails `domain_check`, e.g. a single NaN anywhere in `base`, or in `y` for `frac_diff_grouped`), `fit()` takes the recurrent "full-then-mask" branch: it rebuilds `y_seq`/`base_seq` at the ORIGINAL full length `n` via `_carry_forward_fill`, then calls `transform.forward(y_seq, base_seq, transform_params, groups=groups_train)` -- but `groups_train` still has length `valid.sum() < n`. Inside `_grouped_extra.py`'s `_group_segments(groups)` / `_ewma_residual_grouped_apply` / `_rqr_grouped_median` / `_frac_diff_grouped_forward`, the row indices are derived from `range(len(groups))` = `range(valid.sum())` and used to index the full-length `arr_f`/`base_f`/`out` arrays (size `n`). This means: (a) every group's window is computed over the WRONG subset of the true sequence (row identity is scrambled), and (b) `out[valid.sum():n]` is never written at all, so it retains whatever `np.empty` happened to allocate. Back in `_estimator.fit()`, `t_train = t_full[valid]` then reads from arbitrary positions of this partially-uninitialized array -- the inner GBDT is silently trained on garbage T for a subset of rows, with no guarantee the "T must be finite" guard at `_estimator.py:693` catches it (`np.empty` garbage is frequently a finite-looking float). No test in `tests/training/composite/` exercises any of these three transforms through `CompositeTargetEstimator.fit()` with a `group_column` AND a domain-violating row -- every existing test (`test_transforms_t_batch.py`, `test_composite_transforms_registry_contract.py`) calls the transform functions directly with 100%-valid synthetic data, so the bug is untested. Suggested fix: keep a FULL-length groups array alongside `y_seq`/`base_seq` for the recurrent branch (only the `[valid]`-sliced version should go to `transform.fit()` and to the non-recurrent forward path), then slice `t_full` (not `groups`) by `valid` at the end, exactly as already done for `y`/`base`.

### F2 -- `seasonal_residual` phase drifts when the domain filter drops a mid-series row (P1)

`seasonal_residual` is registered with `recurrent=False` (`_registry_extended.py`), so `_estimator.fit()` takes the ordinary (compacting) path: `transform.forward(y_train, base_train, ...)` where `y_train = y_arr[valid]` has already had any non-finite-`y` rows removed. `_seasonal_residual_fit` / `_forward` / `_inverse` all compute `phase = np.arange(y_f.size) % period` -- i.e. the position WITHIN THE ARRAY PASSED IN, not the row's true calendar position. If the training target has even one NaN scattered mid-series (a routine real-world gap), every row after it is silently re-indexed to the wrong phase relative to its true position, corrupting the learned per-phase means and every downstream residual with no error raised. This is exactly the class of bug the `recurrent=True` full-then-mask machinery exists to fix elsewhere in the same package (EWMA / frac-diff / rolling-quantile-ratio), but `seasonal_residual`'s dependency is on absolute ROW POSITION rather than neighbour VALUES, so the existing `recurrent` flag's "pointwise, no neighbour reads" justification for leaving it `False` does not actually rule out the bug. Suggested fix: either flag `seasonal_residual` `recurrent=True` (the forward already tolerates being run on the full sequence) so it gets the full-then-mask treatment, or thread an explicit absolute row-index array through fit/forward/inverse that survives the domain filter instead of relying on `arange(len(compacted_array))`.

### F3 -- external-holdout OOF path ignores the caller's `random_state` (P1)

`compute_oof_holdout_predictions(..., random_state=rs, external_holdout_X=..., external_holdout_y=...)` -- the branch the function's own docstring calls "preferred" -- routes to `_compute_oof_with_external_holdout` (`ensemble/__init__.py:269`). That helper's signature has NO `random_state` parameter at all, and its two `_carve_inner_eval_split(...)` calls (lines 335-340 and 360-365) hardcode `random_state=0`. Every other OOF-split branch in the same file (`kfold` branch at line 601/639, the group-outer-holdout branch at line 745/755, the plain random-outer-holdout branch at line 763/840/866) correctly threads `random_state=int(random_state)`. As a result, re-running the exact same suite call with different top-level seeds produces an IDENTICAL internal early-stopping eval-carve every time on the external-holdout path, silently defeating any multi-seed variance / bootstrap analysis that assumes each seed perturbs every randomness source (this repo's own `derive_seeds` helper exists precisely to keep these sources independent-but-reproducible). Suggested fix: add a `random_state` parameter to `_compute_oof_with_external_holdout` and pass the caller's value through to both `_carve_inner_eval_split` calls instead of the literal `0`.

### F4 -- T-clip envelope width changes silently after a streaming drift refit (P1)

At `fit()` (`_estimator.py:783-797`) and in `from_fitted_inner()` (`_estimator.py:404-413`), `t_clip_low`/`t_clip_high` are computed as `median(T) +/- 10 * median(|T - median|)` -- raw (unscaled) MAD. `_update.py`'s post-drift-refit envelope refresh (line 176-187) recomputes the SAME two keys as `median(T) +/- 10 * (median(|T - median|) * 1.4826)`. The two formulas both claim to be "the fitted T-train envelope" the docstrings describe identically, but the `update()` path is ~48% wider for the same underlying spread (1.4826 is the normal-consistent scale factor for MAD, absent from the two `_estimator.py` sites). Practically: `predict()`'s T-clip guard (documented as bounding heavy-tail blow-up, e.g. "Huber-slope miscalibration") becomes measurably less aggressive purely as a side effect of ever calling `.update()` with a firing drift correction, with no comment anywhere explaining or intending the discrepancy. Suggested fix: apply the same `* 1.4826` scaling at all three sites, or drop it from `_update.py`, so the envelope's statistical meaning (and width) is consistent across the initial fit and any later refresh.

### F5 -- `predict_quantile` under-reports y-clip hits (P2)

`_invert_one` inside `predict_quantile` (`_predict.py:493-503`) applies `np.clip(y_col, low, high)` on every quantile column, but every call site (`_predict.py:509-511`, `518-520`, `526-528`) passes `low_hits=0, high_hits=0` to `_record_runtime_stats`. `runtime_stats_["y_clip_low_hits"]` / `["y_clip_high_hits"]` therefore never move for quantile predictions even when clipping is silently happening on every batch, while the same physical event is correctly counted by `predict()`. An operator monitoring `runtime_stats_` (the documented observability surface) for distribution drift via the quantile API gets a false all-clear. Suggested fix: compute `int(np.sum(y_col < low))` / `int(np.sum(y_col > high))` inside `_invert_one` before clipping and accumulate them into the closure totals alongside the existing T-clip counters.

### F6 -- dead duplicate EWMA / frac-diff numba kernels in `transforms/__init__.py` (P2)

`transforms/__init__.py:28-60` defines its own `_ewma_kernel` and `_frac_diff_inverse_kernel` `@numba.njit` functions, byte-identical in body to the ones in `nonlinear.py` that are actually used by `_ewma_compute` / `_frac_diff_inverse_compute` and their dispatchers. Grepping the whole `src/` and `tests/` trees turns up zero references to the `__init__.py` copies outside their own definition -- every real caller and every test (`tests/training/test_ewma_frac_diff_parallel.py`) imports the `nonlinear.py` versions. This is leftover duplication from before the `nonlinear.py` module-split (the module's own comments describe the split explicitly for other symbols) that was never cleaned up. Suggested fix: delete the two dead kernel defs (and the `else: _ewma_kernel = None; _frac_diff_inverse_kernel = None` fallback) from `transforms/__init__.py`.

### F7 -- `_seasonal.py`'s recurrent=False justification doesn't match the actual failure mode (P2)

See F2. The module docstring reasons "Given the phase, the transform is pointwise (no neighbour reads), so `recurrent=False` is correct: dropping a row shifts later phases exactly as it would shift them in the caller's own frame." The second clause is not accurate: the row that gets dropped is dropped by the WRAPPER's own `domain_check`-driven compaction inside `fit()`, not by anything visible in "the caller's own frame" -- the caller passed a full, gap-free series and the wrapper silently introduced the gap. Suggested fix: correct the comment once F2 is fixed (either flip to `recurrent=True` or document the real position-index contract), so the stated design rationale matches what the code does.

## Coverage notes

- The 15 `_benchmarks/*.py` scripts in `ensemble/` and `transforms/` (1,437 LOC) were spot-checked for import sanity and assertion-correctness (all pin bit-identity/round-trip invariants consistent with this repo's benchmark convention) but not read line-by-line for algorithmic correctness, since they are never imported by production code (developer-run microbenchmarks only) and are explicitly out of the hot runtime path this audit's checklist targets. If a future audit wants to verify the specific speedup numbers cited in the reviewed production docstrings/comments (e.g. "2.7-3.8x at K>=10, N>=100k" for the batched EWMA kernel), those benchmark scripts are where the numbers come from and would need to be re-run, not re-read.
- `discovery/` and `post_shim.py` / `conformal.py` / `conformal_online.py` / `streaming.py` under `src/mlframe/training/composite/` (referenced by lazy imports from files in scope, e.g. `_soft_shrink.py`'s `_resolve_lag_column` reaching into `..discovery._causal_lag`, and `_estimator.py`'s bottom-of-module binding of `..conformal` / `..conformal_online`) are OUTSIDE the assigned cluster (`transforms/`, `ensemble/`, `estimator/` only) and were not audited; only enough of their signatures were read to confirm the call sites in scope use them correctly.
- Did not run the test suite or any benchmark script (read-only audit constraint); F1-F5 are argued from static code reading plus a targeted grep confirming the absence of a covering test, not from an observed test failure.
