### F1 -- `collapse_rare_categories` has no fit/apply split (rare_count_pruning.py:63-126)

`collapse_rare_categories(df, columns, min_count, ...)` recomputes `df[col].value_counts()` (and, in `target_aware=True` mode, the WoE/z-test against `y`) from whatever single `df` it is given -- there is no way to learn the rare-category set on train and replay the SAME mapping on test/inference data, unlike every other "learned" transform in this package (`UnseenCategoryImputer.fit`/`.transform`, `align_feature_direction`/`apply_feature_direction`, `cleaning.analyse_and_clean_features`/`apply_features_cleaning`). The module's own test file (`tests/preprocessing/test_biz_val_rare_count_pruning.py:51-52,126-131`) calls it TWICE -- once on `X_train`, once on `X_test`, each with its OWN `min_count`/`y` -- exactly the pattern that breaks category-space consistency: a category common enough to stay itself in train but rare in test gets silently mapped to `"__other__"` in test while the train-side encoder never learned an `"__other__"` slot (or vice versa), reproducing the exact "unseen category" failure mode `train_test_support_screen`/`UnseenCategoryImputer` in this same package exist to prevent. In `target_aware=True` mode this is worse: calling it on the test split with `y=y_test` uses the TEST LABELS to decide which test-side rare categories get collapsed -- information not available at real inference time. Fix direction: return (or accept as input) a persisted per-column collapse mapping (`{col: {rare_value: other_label}}` plus, for `target_aware`, the fitted collapse set) so train fits once and test/inference replays the identical mapping, mirroring `cleaning.apply_features_cleaning`.

### F2 -- `match_missingness_rate` crashes on any non-numeric column (degradation_augment.py:28-52)

`values = X_train.to_numpy(dtype=np.float64, copy=True)` operates on ALL of `X_train`'s columns, not `X_train.select_dtypes(include=[np.number])` the way the sibling `match_noise_level` (line 62-64) correctly does. Any `X_train` with a categorical/object/string column -- the normal case for a real feature frame -- raises `ValueError: could not convert string to float` immediately. Since `match_missingness_rate` is the first element of `augment_to_match_test_distribution`'s default `degradation_fns=(match_missingness_rate, match_noise_level)` (line 85), the default call on a realistic mixed-dtype frame crashes. No test in `test_biz_val_degradation_augment.py` exercises a non-numeric column. Fix direction: restrict to `numeric_cols` the same way `match_noise_level` already does, and pass the untouched non-numeric columns through unchanged.

### F3 -- Whole-frame copy inside the per-column analysis loop (cleaning.py:108-122)

`_update_sub_df_col` is called after every rare-value merge and every 2-valued NaN-fill inside `analyse_and_clean_features`'s per-column loop (lines 668-670, 748-750, 787-789) whenever `update_data=True`. Each call does `sub_df = sub_df.copy() if col in sub_df else sub_df` -- and `col` is always in `sub_df`, so this is unconditionally a copy of the FULL, potentially wide, potentially huge (`analyse_mask`-selected) analysis frame, just to refresh one column's `value_counts()`. On a frame with many columns and many rare-value-merged columns this is O(n_cols x n_calls x full_frame_size) instead of the O(1-column) work actually needed. Fix direction: compute `df.loc[analyse_mask, col].value_counts(dropna=False)` directly as a fresh Series, never touching/copying the rest of `sub_df`'s columns.

### F4 -- `calculated_quantiles` truthiness crash (cleaning.py:313)

`is_variable_truly_continuous`'s signature documents `calculated_quantiles: Optional[np.ndarray] = None`, and lines 327-334 explicitly handle the case where the caller supplies it directly (requiring `tukey_fences_multiplier` too) -- a clearly intended, documented call pattern. But `if calculated_quantiles or use_quantile:` evaluates `bool(calculated_quantiles)` first; for a 2-element ndarray (exactly the type this parameter is documented to hold) Python/numpy raises `ValueError: The truth value of an array with more than one element is ambiguous`. No internal caller or test in the repo currently exercises this path (all call it via `use_quantile` only), so the bug is latent but 100%-reproducible for any external caller following the documented contract. Fix direction: `if calculated_quantiles is not None or use_quantile:`.

### F5 / F6 -- Unnecessary deep `.copy()` of large frames (outlier_policy.py:116,127; regime_conditioned_imputation.py:68)

Both functions do a full deep `.copy()` of the input frame to modify/add only a handful of columns, while every directly comparable sibling in this package (`outlier_cap_or_missing`, `collapse_rare_categories`, `align_feature_direction`, `gaussian_power_transform_search.apply_gaussian_power_transform`) uses `.copy(deep=False)` specifically to avoid doubling peak RAM on the "100+GB frame" case this codebase explicitly designs around (see `cleaning.py:462-464`'s `_DEFRAG_COPY_MAX_BYTES` comment for the same concern elsewhere in this very package). Fix direction: switch both to `.copy(deep=False)`; `regime_conditioned_imputation.py` additionally has no way to persist/replay the train-learned regime medians onto test data (same architectural gap as F1/F7).

### F7 -- `impute_with_missing_indicator` has no persisted fit state (missing_indicator_pairing.py:67-103)

`global_fill`/`group_fill` are computed fresh from `out[col]`/`group_by[col]` on every call (lines 83-99). There is no `fit`/`transform` split and no returned mapping of learned fill values, so a caller who imputes train and test in two separate calls (the natural workflow for this helper, since it is a plain function not a stateful class) gets test rows filled with TEST's own median/mean/mode/group-stats, not train's -- a real train/serve statistic mismatch, and (if `df` passed spans train+val/test together) direct look-ahead leakage of val/test rows into the imputed values used for train. `cleaning.analyse_and_clean_features` documents this exact risk with an explicit "LEAKAGE WARNING" docstring section and solves it via `features_transforms`/`apply_features_cleaning`; this module has neither the warning nor the persisted-state solution. Fix direction: add a `fit`/`transform` pair (or return the learned fill-value mapping) mirroring `UnseenCategoryImputer`.

### F8 -- Single-class `y` division by zero (align_feature_direction.py:35-39)

`n_pos = int(is_pos.sum()); n_neg = len(y_arr) - n_pos; ... / (n_pos * n_neg)`. If `y` has only one class (a realistic edge case for a small/filtered fold), `n_pos*n_neg == 0`, producing `inf`/`nan` AUCs via a `RuntimeWarning: divide by zero` that is not raised as an exception. Downstream, `auc < 0.5` is `False` for NaN, so `align_feature_direction` silently decides "no flip" for every column instead of surfacing an error, and `check_feature_direction_stability` would silently record meaningless `mean_fold_auc=nan` too. Fix direction: raise a clear `ValueError` (or short-circuit to all-`sign=1`/skip with a logged warning) when `y_arr` is single-class, matching this codebase's established "assert/silent-NaN -> ValueError" convention (see `cleaning.py`'s "Wave 31" fixes at lines 217, 320, 428).

### F9 -- Scale-mismatched real vs. augmented rows (temporal_drift_augment.py)

`augment_temporal_drift` never modifies `first_frame`'s (the real, non-augmented rows') `feature_cols` -- they stay at whatever raw scale the caller passed in (line 88, `ordered.assign(...)`). The synthetic rows, however, have `feature_cols` OVERWRITTEN with `(raw - expanding_mean) / expanding_std` (lines 100-105), i.e. a truncated-history Z-SCORE. If a caller follows the docstring literally -- "Return df concatenated with synthetic 'earlier vintage' rows... plus a bool `_temporal_drift_augmented` column... so callers can filter/weight them separately if desired" -- and trains directly on the full returned frame (using `_temporal_drift_augmented` only for optional weighting, as the wording suggests), half the training rows carry raw-scale values and half carry z-score-scale values in the exact same column, which would badly corrupt any model's use of that feature. The module's own `tests/preprocessing/test_biz_val_temporal_drift_augment.py:44-55` never actually consumes the function's real rows this way -- it discards them and reconstructs a separately-standardized `train_last` externally, using ONLY `augmented.loc[augmented["_temporal_drift_augmented"]]` from the function's output. This working pattern is never stated in the docstring. Fix direction: either document prominently that the real rows must be discarded/re-standardized by the caller before combining with the synthetic rows, or (better) have the function itself z-score the real rows too (against their full history) so the returned frame is internally scale-consistent and usable as documented.

### F10 -- Box-Cox/Yeo-Johnson refit-on-apply (gaussian_power_transform_search.py:166-181)

`gaussian_power_transform_search` picks `best_transform` per column via `_apply_transform`, but never persists the fitted Box-Cox lambda / Yeo-Johnson power parameter -- only the transform NAME goes into `search_result`. `apply_gaussian_power_transform` then calls `_apply_transform(finite_fill, info["best_transform"])` again, which for `"boxcox"` calls `scipy.stats.boxcox(x)` (no fixed `lmbda=`, so it refits from `x`'s own distribution) and for `"yeo_johnson"` constructs a brand-new `PowerTransformer().fit_transform(x)` (refits again). Calling `apply_gaussian_power_transform` on a DIFFERENT frame than the one passed to `gaussian_power_transform_search` (e.g. search on train, apply to val/test -- the natural intended workflow for a "search" + "apply" pair) silently applies a DIFFERENT function than the one whose skew/target-correlation was measured and selected. All tests apply to the exact same `df` used for search, so this gap is untested. Contrast with `align_feature_direction`/`apply_feature_direction` in the same package, which correctly persist `(mode, center, sign)` and replay it without recomputation. Fix direction: persist the fitted `lmbda`/`PowerTransformer` object in `search_result` and have `apply_gaussian_power_transform` reuse it instead of recalling `_apply_transform`.

## Coverage notes

- `src/mlframe/preprocessing/_benchmarks/**` (~30 files) was intentionally not read for findings: the task's own file/LOC budget (23 files, ~4.2k LOC) matches exactly the 23 non-benchmark files under `preprocessing/`, and these are timing/profiling scripts (`bench_*.py`, `prof*.py`, `check*.py`), not library code exercised by production call paths.
- Every finding above was cross-checked against the corresponding `tests/preprocessing/test_*.py` file where one exists, to confirm the bug is real (not already guarded/tested) and, where possible, to confirm the intended call contract from the test's own usage pattern (this is how F1, F9, F10, F13 were confirmed rather than just inferred from reading the source).
- `_cleaning_kernels.py`'s three `numba.njit` kernels were read in full and reasoned through by hand for NaN/edge-case parity against the numpy expressions they replace (no numba runtime was executed, since this is a read-only audit); no discrepancy found.
- Did not execute any test or benchmark script (per the read-only audit constraint); all findings are from static reading plus test-file cross-reference, not from an observed failure/traceback. Severity/impact statements (e.g. "crashes with ValueError") are derived from tracing the exact operations (`np.to_numpy(dtype=float64)` on a string column, `bool()` on a multi-element ndarray) against documented Python/numpy semantics, not from a captured stack trace.
