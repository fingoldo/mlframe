# Cross-cutting: test-suite architecture & coverage map -- mlframe audit

## Scope

This cluster's scope is enormous (tests/ contains ~15,395 `def test_*` functions across hundreds of files once the MRMR/shap_proxied exclusions are applied), so per the task instructions this was a **structural audit**: directory-by-directory characterization, targeted grep sweeps across the whole in-scope tree, and deep reads of a representative sample of files rather than a line-by-line read of every file. ## Findings

| ID | Severity | Category | File:Line | Summary |

| F1 | P1 | flaky-pattern | `tests/training/test_biz_val_training_core.py:57-67,96-97` | `_try_import_suite()` catches `ImportError`/`AttributeError` on the actual import of `train_mlframe_models_suite` and converts it to `pytest.skip`; the per-test `try/except (TypeError, ImportError): pytest.skip(...)` around the real call under test does the same for the call itself -- a genuine kwarg-contract break in the public training API silently downgrades a `biz_val` test to a skip instead of a failure. |
| F2 | P1 | flaky-pattern | `tests/training/test_suite_api_ergonomics.py:34-42`, `tests/training/test_precompute_bundle.py:32-41` | Same `_try_import_suite`/`except (ImportError, AttributeError): pytest.skip` pattern duplicated verbatim in two more files exercising the same `train_mlframe_models_suite` public surface. |
| F3 | P1 | suspicious-skip | `tests/training/test_shortlist_transformer_adapter_suite.py:14-20` | Module-level `except Exception: pytest.skip(..., allow_module_level=True)` around the imports of `train_mlframe_models_suite`, `OutputConfig`, `ReportingConfig`, `FeatureSelectionConfig` -- broader than `ImportError`/`AttributeError` (catches *any* exception raised during those imports, e.g. a config-validation error at class-definition time) and skips the WHOLE file, not just one test. |
| F4 | P1 | ci-integration | `.test_durations` (root, 4 bytes, content `{}`); `.github/workflows/update-test-durations.yml`; `.github/workflows/ci.yml:37-39,130-131` | The pytest-split duration file that `ci.yml`'s `--splits 8 --group N` 8-way shard relies on for balanced sharding across 5 (Linux, Python 3.9-3.13) matrix legs is empty. `gh run list --workflow=update-test-durations.yml` shows exactly one historical run (2026-07-19, scheduled), which was **cancelled** after 5h33m and never completed -- the file has apparently never been successfully populated. Every CI shard is currently falling back to pytest-split's naive equal-file-count split, which the repo's own comments in `ci.yml` (and the fact that a numba-cold shard needs ~65s/file vs a trivial file's sub-second run) say is exactly the imbalance this mechanism exists to avoid; a hot shard risks the 120-minute cap while others finish in a fraction of the time. |
| F5 | P2 | test-gap | `src/mlframe/integrations/mlflow.py` (whole file, 173 LOC) | Zero test coverage anywhere under `tests/` for the mlflow integration module -- confirmed via repo-wide case-insensitive grep for "mlflow" in `tests/`, the only 3 hits are unrelated string matches in other files' output/config, none of which import or exercise `mlframe.integrations.mlflow`. The module includes untested security-adjacent logic: `_strip_userinfo` (credential-URL redaction before logging) and `get_or_create_mlflow_run`'s `_dsl_escape` (quote/backslash escaping to prevent DSL-filter-string injection via a caller-supplied `run_name`). `pyproject.toml`'s `[tool.coverage.run] omit` list also explicitly excludes `*/integrations/*`, so this module is invisible to the coverage report too -- nothing would flag the gap even if coverage were reviewed. |
| F6 | P2 | test-gap | `src/mlframe/system/kernel_tuning_cache/__init__.py:1-40` (180 LOC total) | The `mlframe-tune-kernels` console-script entry point (registered in `pyproject.toml:411`, subcommands `list`/`show`/`explain`/`refresh`/`refresh-all`/`clear`) has no test invoking `main()` or any subcommand; a broken `argparse` wire-up or a renamed `pyutilz.performance.kernel_tuning` symbol would only surface when a user runs the CLI by hand. |
| F7 | P2 | weak-assertion | `tests/evaluation/test_evaluation_salvage.py:169-202` | `test_catboost_stopfile_smoke`, `test_lightgbm_stopfile_smoke`, `test_xgboost_stopfile_smoke`, `test_lightning_stopfile_smoke` each construct a `*StopFileCallback` and assert only `cb is not None` -- a constructor that doesn't raise always returns a non-None instance in Python, so this assertion can never fail; none of the four tests exercises the callback's actual behaviour (does it detect the stop-file, does it actually halt training). |
| F8 | P2 | suspicious-skip | `tests/training/test_automl.py:110,133,234,251` | The 4 real-training tests for AutoGluon (`test_basic_training`, `test_training_with_test_df`) and LAMA are `@pytest.mark.skip(reason="... heavy dependency - run manually if needed")` with no CLI opt-in flag, unlike this same test suite's own established pattern for other heavy tests (`fuzz`/`biz_transformer` markers gated behind `--run-fuzz`/`--run-biz-transformer` in `tests/conftest.py:363-378`, collected and explicitly deselected so they're at least visible and runnable). These four are permanently inert in every mode; only the "library not installed -> returns None" fallback path is exercised, so a real regression in the AutoGluon/LAMA training call shape (e.g. a renamed `fit_params` key) would never be caught even by a maintainer who explicitly wants to run the heavy tests. |
| F9 | P2 | test-gap / doc-quality | `tests/interrogate.toml:6-9`; representative sites e.g. `tests/inspection/test_interaction.py:14,24,30,35,50,57,64,72` | `tests/interrogate.toml` documents a "full docstring-authoring campaign" that took whole-tree docstring coverage from 53.1% to 100% in one day (2026-07-17 -> 2026-07-18) to satisfy a `fail-under = 100` gate. A repo-wide grep for the literal filler pattern `"""Helper that ` finds 499 occurrences across 369 test files (e.g. `"""Helper that predict."""`, `"""Additive model has near zero interaction."""` restating the function name rather than adding information). The `test_public_docstrings.py` meta-test (F5 in this list's sibling checks) only scans `src/mlframe`, so this pattern is specific to the `tests/interrogate.toml` 100%-of-tests gate; the gate optimizes for docstring *presence*, and the campaign's output demonstrates it was satisfied with content-free auto-paraphrased docstrings on a large scale rather than genuine documentation. |
| F10 | P2 | suspicious-skip (documented, lower confidence) | `tests/training/test_audit_dict_duplicate_keys.py:105-107`, `tests/training/test_audit_if_elif_missing_else.py:113-115,135-137`, `tests/training/test_audit_sentinel_collision.py:158-160,180-182` | Five behavioural-sensor tests guard against a "stale build path" (`__file__` not under `src/mlframe`) and skip rather than fail when the imported module resolves to a different copy on `sys.path`. Each site is individually commented ("namespace-package gotcha", wave 49), so this looks like a genuine, previously-diagnosed environment quirk rather than a fresh bug -- flagged for visibility per this repo's own "never accept an undocumented skip" convention, since the skip converts a potential real regression (if it were ever the *live* source failing, not a stale one) into silence with no distinguishing signal in the test output between "correctly detected stale path" and "silently masking a live-source failure that happens to also resolve from a stale path in CI." |

## Coverage notes

- The MRMR/filters/shap_proxied exclusion (per this audit's scope contract) removes by far the largest chunk of `tests/` and `src/mlframe/feature_selection/` from this review; `src/mlframe/feature_selection` alone is 228,117 LOC across 819 files, versus every other `src/mlframe/` subpackage combined (~301,000 LOC). The coverage-gap comparison above is therefore necessarily silent on that package's internals -- the two dedicated audit sessions cover it (see `audits/mrmr_audit_2026-07-20/`).
- `src/mlframe/training/` is comparably large (178,254 LOC across 646 files, `composite/` alone 60,104 LOC / 249 files) and its `tests/training/` mirror is correspondingly huge (~370 loose files plus 17 subdirectories); a genuine per-file coverage-gap audit of that subtree at the granularity done for the smaller packages (`data`, `system`, `inspection`, `integrations`) was not attempted -- it would require either scripted symbol-level cross-referencing (which this read-only, no-execution audit could have built as a static AST script but did not, given the scope budget) or another dedicated cluster's worth of effort. The `test_meta/` LOC-budget exempt-list excerpt (read in full) shows the maintainers are already tracking several `training/` files at or near the 1k LOC carve ceiling with real justification, which is a reasonable proxy signal that the area is actively maintained, not neglected.
- Weak-assertion spot-checking used `grep "assert .* is not None$"` (661 hits in scope) as a cheap proxy and sampled ~15 hits across `calibration`, `estimators`, `evaluation`, `feature_engineering`, `votenrank`; the large majority sampled were precondition checks followed by real behavioral assertions (not weak), with the exception of the 4 stopfile-callback smoke tests in F7. A full read of all 661 sites was not attempted; this is a sample, not an exhaustive sweep, and more weak-assertion-only tests likely exist among the un-sampled remainder (in particular `tests/training/`'s ~370 loose files were not sampled for this pattern at all).
- Did not attempt to run `pytest --collect-only` or any other pytest invocation (explicitly disallowed for a read-only audit), so all "does this test actually pass / is it actually collected" claims are inferred from static reading, not confirmed execution. In particular, F4's claim about `.test_durations` being empty is a filesystem fact (`{}`, 4 bytes) plus one `gh run list` read-only query, not a live CI observation.
- `gh run list` was used once (read-only, no state change) to check `update-test-durations.yml`'s run history; no other GitHub API calls were made.
