## Proposals

### (fe_idea) Chatterjee's Xi rank correlation scorer

Idea: Chatterjee (2021, "A New Coefficient of Correlation", JASA) defines xi(X,Y) = 1 - 3*sum_i |r_{i+1} - r_i| / (n^2-1), where r_i is the rank of Y reordered by ascending X. Unlike Pearson/Spearman it is 0 iff X,Y independent (asymptotically) and 1 iff Y is a MEASURABLE FUNCTION of X (not just monotone) -- it is a genuinely different construction from every scorer already shipped (plug-in MI is binning-based, KSG is kNN-distance-based, copula-MI is rank-uniformized-MI, dCor is U-centred-distance-matrix-based, HSIC is RKHS-kernel-based). Why it catches a shape the catalog misses: on y = sin(20*x) + noise (a smooth but highly-OSCILLATORY, many-cycles function), the plug-in MI's fixed quantile bins average many oscillation cycles into each bin (near-zero MI), KSG's kNN balls at moderate k similarly smear across cycles, and dCor/HSIC (which weight by an RBF/distance kernel calibrated to typical global scale) underweight the fine local structure -- but xi's SORT-THEN-WALK construction is scale-free and directly sees every local up/down rank flip, so it stays high even at very high oscillation frequency where every distance/kernel/binning scorer decays toward the null floor. Sketch: `def score_by_xi(X: np.ndarray, y_bin: np.ndarray, *, ties: str = "average") -> np.ndarray:` per column, argsort by x, take y ranks in that order, `xi = 1 - 3*np.sum(np.abs(np.diff(r)))/(n**2-1)`; wire it into the Layer 68 auto-scorer dispatch table (_orthogonal_scorer_auto_fe.py) alongside plug-in/KSG/copula/dCor/HSIC as a 9th scorer, since that family already has 8 members -- a proven, actively-extended pattern. GPU residency: HIGH. The whole computation is one argsort (cupy `cp.argsort`, batched per-column via `axis=0` sort of a (n,K) matrix) + one `cp.diff` + one reduction; no CPU library is structurally required, fully cupy-native and stays resident alongside the existing resident-operand cache.

### (fe_idea) Random Fourier Features (random kitchen sinks) multi-column kernel-approximation block

Idea: Rahimi & Recht (2007, "Random Features for Large-Scale Kernel Machines"): draw a random Gaussian projection matrix W (p x m) and phases b ~ Uniform(0,2*pi), emit phi(x) = sqrt(2/m) * cos(X @ W / bandwidth + b) as m new columns; the inner product phi(x_i).phi(x_j) approximates an RBF kernel k(x_i,x_j) = exp(-||x_i-x_j||^2/(2*bandwidth^2)) in expectation. Why it catches a shape the catalog misses: every existing basis (Hermite/Legendre/Chebyshev/Laguerre/wavelet/hinge/Fourier/spline) is a PER-COLUMN expansion, and every cross-basis family (pair/triplet/quadruplet/adaptive-arity) is a PRODUCT of per-leg bases -- none of them build a feature that is jointly a smooth function of MANY (5+) raw columns simultaneously without combinatorial blow-up. Concrete scenario: y = exp(-||x||^2 / 2) on p=10 jointly-informative numeric columns (a radial/Gaussian-bump target in 10-D) -- no pairwise product term or even the quadruplet arity-4 cell captures a genuinely 10-way radial structure (it would need C(10,4) or worse cells and still only see products, not the exponential radial form), while a handful of random Fourier features linearly recovers it because the RBF kernel IS the radial-Gaussian target class. Sketch: `def random_fourier_features(X: np.ndarray, m: int = 64, bandwidth: float | None = None, seed: int = 0) -> np.ndarray:` bandwidth defaults to the median pairwise distance heuristic (computed on a subsample); `Z = X @ W / bandwidth + b; return np.sqrt(2.0/m) * np.cos(Z)`; screen the m emitted columns through the existing Tier-1 local_mi_gate / Tier-2 unified_second_pass_gate (_unified_fe_gate.py) exactly like the other recipe-emitting families, so only genuinely useful random directions survive. GPU residency: HIGH. `X @ W` is a single cupy matmul, `+ b` and `cos` are elementwise -- the entire computation is 2 kernel launches, trivially fused with the resident operand matrix, no CPU library needed at all.

### (fe_idea) GBDT leaf-membership target-encoded feature (Facebook GBDT+LR trick)

Idea: He et al. (2014, "Practical Lessons from Predicting Clicks on Ads at Facebook"): fit a small, shallow gradient-boosted tree ensemble (e.g. 20-50 trees, depth 3-4) on the CURRENT selected-feature support, extract each row's terminal LEAF INDEX per tree (a high-order, non-parametric, arbitrary-depth partition of the joint feature space), and K-fold target-encode each tree's leaf-id column (reusing the existing Layer 33 K-fold target-encoding machinery in _target_encoding_fe.py verbatim -- only the categorical column being encoded is synthetic). Why it catches a shape the catalog misses: a leaf id encodes an AND-of-thresholds rule of depth up to the tree depth over an arbitrary, GBM-CHOSEN subset of columns -- e.g. y = 1{(x1>3 AND x5<0.2) OR (x2>7 AND x9>1 AND x3<0)}, a depth-3, disjunctive rule over 5 distinct columns picked by the boosting split search. No fixed-order product-of-bases family (pair/triplet/quadruplet/adaptive-arity, which are all MULTIPLICATIVE, never a disjunction of conjunctions) or fixed cat-interaction family (pair/triple/k-way target-encoded CROSSES, which need the interacting columns specified up front) can discover this rule without already knowing which 5 columns and which OR/AND structure to search -- the GBM's own greedy split search does that discovery for free. Sketch: `def gbdt_leaf_features(X: np.ndarray, y_fold_safe, cols: list[int], *, n_trees: int = 30, max_depth: int = 3, n_folds: int = 5) -> tuple[np.ndarray, list[str]]:` fit `n_folds` OOF LightGBM/XGBoost models (leak-safe, mirroring the K-fold target-encoding OOF discipline already used by Layer 33), call `.predict(..., pred_leaf=True)` to get an (n, n_trees) int matrix, then K-fold-target-encode each tree column via the existing `_compute_target_encoding` helper in _cat_target_encoding_and_weighted.py. GPU residency: PARTIAL/HONEST-NO for the core fit. Training the GBM itself needs a real boosted-tree library (LightGBM/XGBoost/CatBoost); those DO have GPU histogram-building backends (XGBoost `tree_method=gpu_hist`, LightGBM `device=gpu`), so the tree-fit step can be GPU-accelerated, but it is fundamentally a call into an external tree-boosting library's own kernel, not a cupy/numba.cuda kernel mlframe could write itself -- leaf-index extraction and the subsequent target-encoding pass are cheap host-side / reuse the existing (CPU) target-encoding path.

### (fe_idea) Row-wise ordinal-pattern (Bandt-Pompe permutation) encoding

Idea: Bandt & Pompe (2002, "Permutation Entropy: A Natural Complexity Measure for Time Series"): for a chosen K-tuple of columns (x_{i1},...,x_{iK}), encode each row by the FULL RELATIVE ORDER (permutation) of the K values, i.e. which of the K! orderings the row realizes, and target/count-encode that permutation-id like a categorical. Why it catches a shape the catalog misses: the shipped ROW-ARGMAX feature (_conditional_gate_fe.py) only reports WHICH column is the row maximum -- for K=3 that collapses 6 possible total orderings into 3 buckets, discarding the second-vs-third order entirely. Concrete scenario: y = 1{x1 > x2 > x3} (exactly one of the 6 orderings is positive) -- argmax_row(x1,x2,x3) only tells you x1 is the max in 2 of those 6 orderings (x1>x2>x3 and x1>x3>x2) and cannot distinguish the target-positive one from the target-negative one; only the full permutation id resolves it. This is a natural extension of the argmax/conditional-gate family, which the codebase's own docstring already frames as "two multi-column operators the rich catalog cannot express" -- ordinal-pattern is operator #3 in that same family, generalizing #1 (argmax) to the full ranking. Sketch: `def ordinal_pattern_features(X_cols: np.ndarray, *, K: int = 3, seed_tuples: list[tuple[int,...]] | None = None) -> tuple[np.ndarray, list[str]]:` for each candidate K-tuple (capped like the existing triplet/quadruplet seed_count cost guard), compute `perm_id = np.argsort(X[:, tuple_idx], axis=1)` reduced to a base-K! integer via a lexicographic-rank lookup table, then feed the resulting low-cardinality categorical straight through the existing K-fold target-encoding / count-encoding path. GPU residency: HIGH. For small fixed K (3-5, matching the existing triplet/quadruplet arity cap) `cp.argsort(..., axis=1)` on a (n,K) resident slice is a single batched sort kernel; the lexicographic-rank reduction to perm_id is a small fixed-size elementwise/table-lookup step, both cupy-native with no host round trip.

### (fe_idea) Multivariate Mahalanobis / Gaussian-copula joint density anomaly score

Idea: compute a single new feature d(row) = sqrt((x-mu)^T Sigma^{-1} (x-mu)) over ALL (or a correlated cluster of) numeric raw columns jointly -- the classical multivariate-normal quadratic form -- and, for a copula-robust variant, the log-density under a fitted Gaussian copula on the rank-uniformized marginals (handles heavy-tailed/skewed columns the raw Mahalanobis form is not robust to). Why it catches a shape the catalog misses: y depends on whether a row sits inside or outside an ELLIPSOIDAL level-set of the joint distribution of p=15-30 correlated numeric columns (e.g. a multivariate process-control / fraud "jointly-typical vs jointly-atypical" target) where NO single column, pair, triplet, or even the quadruplet arity-4 cross-basis is individually extreme -- each column can sit comfortably within its own marginal range while the JOINT combination is far in Mahalanobis distance (the classic multivariate-outlier-invisible-to-univariate-checks scenario). The existing group_distance / conditional-dispersion families condition one column's deviation on ONE other (binned) column; this is the p-way generalization using the FULL covariance structure. Sketch: `def mahalanobis_density_feature(X: np.ndarray, cluster_cols: list[int], *, shrinkage: float = 0.1) -> np.ndarray:` fit mu, Sigma (Ledoit-Wolf shrunk to avoid the p>n ill-conditioning) on TRAIN rows only (leak-safe, mirrors the existing K-fold-fit-then-apply discipline), then `d = sqrt(einsum('ni,ij,nj->n', X-mu, Sigma_inv, X-mu))`; naturally targets the existing DCD (_dynamic_cluster_discovery) correlated-cluster groups the cluster_basis family (Layer 61) already discovers, so it is a drop-in sibling scorer for those same clusters rather than a brand-new column-selection mechanism. GPU residency: HIGH. Covariance estimation is `X.T @ X` (one cupy matmul), the Ledoit-Wolf shrinkage target is a scalar trace computation, and `Sigma_inv` via `cp.linalg.cholesky` + `cp.linalg.solve` (cuSOLVER-backed) or `cp.linalg.inv` on a p x p matrix (p is small -- tens, not thousands) is cheap and fully cupy-native; the final quadratic form is a batched einsum/matmul over the resident (n,p) operand block.

### (fe_idea) Sliced Inverse Regression (SIR) oblique-direction projection feature

Idea: Li (1991, "Sliced Inverse Regression for Dimension Reduction"): slice y into H bins, compute the per-slice mean of X, form the between-slice-mean covariance matrix M = Cov(E[X|slice]), then solve the generalized eigenproblem Sigma^{-1} M v = lambda v (Sigma = overall covariance of X); the top eigenvector(s) v give the LINEAR COMBINATION direction(s) w.x along which y varies most -- an effective dimension-reduction direction, not restricted to any 2 or 3 named columns. Why it catches a shape the catalog misses: y = 1{0.6*x1 + 0.5*x2 + 0.4*x3 + 0.3*x4 + 0.4*x5 > c} -- a genuinely OBLIQUE (rotated) threshold spread thinly across 5 correlated columns, where EVERY individual weight is too small for that column's marginal MI to clear the screening floor, and no pairwise/triplet/quadruplet product of any 2-4 of the 5 columns reconstructs the linear combination (axis-aligned bases multiplied together cannot represent a rotated hyperplane economically). SIR recovers the direction w = (0.6,0.5,0.4,0.3,0.4) directly as a single new feature w.x, after which the existing argmax/gate/threshold machinery (or a plain MI screen) picks it up trivially. Sketch: `def sir_direction_features(X: np.ndarray, y: np.ndarray, *, n_slices: int = 10, n_directions: int = 2, is_classification: bool = False) -> tuple[np.ndarray, list[str]]:` bin y into n_slices (reuse the existing _quantile_bin helper from _mi_greedy_cmi_fe.py), compute per-slice means, form M and Sigma, solve `scipy.linalg.eigh(M, Sigma)` (or `cp.linalg.eigh` on the whitened form `Sigma^{-1/2} M Sigma^{-1/2}`), project X onto the top n_directions eigenvectors, and screen the resulting columns through the standard Tier-1/Tier-2 unified FE gate. GPU residency: HIGH for the linear-algebra core (covariance matmuls + `cp.linalg.eigh` on a small p x p symmetric matrix are cupy-native, cuSOLVER-backed); the only CPU-preferred step is the slicing/bin-mean bookkeeping which is a cheap groupby-mean already available as a device-resident bincount-style reduction elsewhere in the codebase (_grouped_agg_fe pattern).

### (fe_idea) Genetic-programming / symbolic expression-tree deep feature synthesis

Idea: Koza-style genetic programming (gplearn-equivalent): evolve a POPULATION of small expression TREES (not just one binary op over two unary-transformed operands) via mutation/crossover/tournament-selection, fitness = MI-uplift or CMI-gain against y, seeded from the operand pool already assembled by build_fe_operand_pool (_step_pool.py). Why it catches a shape the catalog misses: the entire unary/binary FE mechanism (feature_engineering.py) is structurally capped at ONE binary op combining exactly TWO (possibly-unary-transformed) operands -- `binary_op(unary_a(x_i), unary_b(x_j))`. It cannot express a 3-operator, 4-operand nested formula like `(a - b) * c / (d + 1)` in one shot; only degenerate two-level compositions are reachable via multiple FE steps re-selecting an already-engineered column as a new operand, which the pool/step-budget machinery actively rate-limits. A GP search over arbitrary-depth trees (depth <= 4, arity-2 internal nodes, leaves = pool operands) discovers such compound expressions directly. Distinct also from the smart-polynom CMA-ES optimizer (which tunes COEFFICIENTS of a fixed orthogonal-polynomial-basis cell for a fixed PAIR, not the discrete tree STRUCTURE over 3+ operands). Sketch: `def gp_synthesize_features(operand_pool: np.ndarray, operand_names: list[str], y_bin: np.ndarray, *, pop_size: int = 200, n_gens: int = 15, max_depth: int = 4, elite_frac: float = 0.1) -> list[EngineeredRecipe]:` standard GP loop (random tree init -> batched vectorized fitness eval via _mi_classif_batch on the whole population's outputs at once -> tournament select -> subtree crossover/mutation -> repeat), returning the elite trees as EngineeredRecipe objects (reusing the existing recipe/replay contract so transform() on new data works unchanged). Cap total tree evaluations tightly (mirrors the existing fe_synergy_max_pairs / rung-schedule cost guards) since GP is combinatorially more expensive than the fixed-shape searches. GPU residency: HYBRID/HONEST. The FITNESS EVALUATION (applying a batch of candidate trees to the resident operand matrix and scoring MI) is fully vectorizable on GPU -- each tree is just a sequence of elementwise cupy ops fused across the whole population -- but the SEARCH LOOP itself (tree mutation/crossover, tournament bookkeeping, population management) is inherently host-side Python control flow over tree data structures; no cupy/numba.cuda kernel replaces that orchestration, so this is CPU-orchestrated with a GPU-accelerated inner evaluation, not a pure-GPU kernel.

### (fe_idea) Graph co-occurrence spectral embedding for high-cardinality categoricals

Idea: build a weighted co-occurrence graph over the LEVELS of a high-cardinality categorical column (edge weight = how often two levels appear in the same group/session/basket, or how similar their target-conditional distributions are), then embed levels via truncated SVD of the (levels x levels) PMI matrix (GloVe-style, Pennington et al. 2014) or a few power-iterations of the normalized graph Laplacian (Belkin & Niyogi 2003 Laplacian Eigenmaps) -- emit the top-d embedding coordinates as new numeric columns, broadcast per row by the row's category. Why it catches a shape the catalog misses: the shipped K-fold target-encoding (Layer 33) and rare-category/freq-band features (Layer 104) both summarize a level by its OWN target-mean / frequency statistics ONLY -- for a column with 50,000 distinct merchant ids where each id has ~5-10 rows, the per-level target-mean estimate is dominated by sampling noise (a handful of labels per level) and gets shrunk toward the global mean, discarding whatever LATENT similarity structure exists between merchants that never co-occur with enough shared labeled rows individually but DO co-occur heavily with each other in an unlabeled transaction log. The graph embedding pools information ACROSS the whole co-occurrence structure (a semi-supervised-style transfer, complementing the existing Layer 80 semi-supervised basis-preprocessing family which fits marginal stats on unlabeled data but never a cross-LEVEL relational structure) rather than per-level in isolation. Sketch: `def cooccurrence_spectral_embedding(cat_col: np.ndarray, cooccur_key: np.ndarray, *, d: int = 8, min_count: int = 5) -> tuple[np.ndarray, list[str]]:` build a sparse (n_levels x n_levels) co-occurrence count matrix keyed on a grouping id (session/basket/entity), PMI-weight it, truncated-SVD (or sparse Laplacian eigsh) to d dimensions, broadcast the d embedding columns back to rows by category membership; feed through the standard local_mi_gate. GPU residency: PARTIAL/HONEST-MIXED. Building the sparse co-occurrence matrix and a crude power-iteration embedding (dense small-d, e.g. `cupyx.scipy.sparse` matmuls) can run on GPU, but a numerically robust, ARPACK-quality sparse eigensolve for tens-of-thousands of levels is materially more mature and reliable on CPU (`scipy.sparse.linalg.eigsh`/`svds`) than in cupy's current sparse eigensolver support -- for large level-counts this fundamentally leans on a CPU sparse-linear-algebra library, with GPU only cost-effective at moderate level-counts via the crude power-iteration fallback.

### (fe_idea) Local Outlier Factor / k-NN local density-ratio feature

Idea: Breunig et al. (2000, "LOF: Identifying Density-Based Local Outliers"): for each row, compute the ratio of its local k-NN density to the AVERAGE local density of its k neighbors (reachability-distance based); rows in locally sparse regions relative to their neighborhood score high even when the neighborhood itself is not globally extreme. Why it catches a shape the catalog misses: distinct from the Mahalanobis/copula-density idea above (which assumes a single global elliptical/Gaussian-copula shape), LOF is LOCAL and non-parametric -- it catches anomalies in a MULTI-MODAL joint distribution (e.g. 3 well-separated Gaussian clusters of normal behavior, where a row is anomalous for sitting in a locally-sparse gap BETWEEN clusters even though its raw distance to the GLOBAL mean/covariance is unremarkable, since the global Mahalanobis ellipsoid straddles all 3 clusters and a between-cluster point can have a perfectly ordinary global Mahalanobis distance). This is the k-NN analogue used by the KSG MI scorer (Layer 65) but reused as an ENGINEERED FEATURE (a per-row anomaly score) rather than a scorer over a fixed candidate column -- a genuinely different consumption of the same underlying k-NN machinery already in the codebase (info_theory kNN infrastructure), so it reuses proven internals in a new role. Sketch: `def lof_features(X_cluster: np.ndarray, *, k: int = 20) -> np.ndarray:` on TRAIN rows compute pairwise k-NN (reuse the KSG scorer's existing kNN backend), reachability distances, local reachability density `lrd(p) = 1 / (mean(reach-dist(p, o) for o in kNN(p)))`, `LOF(p) = mean(lrd(o) for o in kNN(p)) / lrd(p)`; apply the fitted kNN index (not refit) to test rows at transform time for leak safety. GPU residency: HIGH for moderate n. Pairwise distance via the matmul trick (`||a-b||^2 = ||a||^2+||b||^2-2a.b`) is a single cupy matmul + broadcast, k-NN selection via `cp.argpartition`/`cp.sort` along an axis, and the density-ratio arithmetic is elementwise -- all cupy-native; only brute-force O(n^2) distance at very large n (>~200k) would need chunking or an approximate GPU-kNN library (e.g. cuML/FAISS-GPU) rather than the naive matmul form.

### (fe_idea) Conditional quantile-rank feature (full conditional distribution shape, not just mean/std)

Idea: extend the existing conditional-dispersion family (z-within-group in _grouped_agg_fe.py / _composite_group_agg_fe.py, and the z-score / |z| features in _extra_fe_families_dispersion.py) from CONDITIONAL MEAN/STD ONLY to the full CONDITIONAL QUANTILE: for a numeric column x_i and a binned conditioning column x_j, compute `q(row) = empirical_rank(x_i within bin(x_j))` -- the row's percentile position WITHIN its conditioning bin, not its z-score. Why it catches a shape the catalog misses: on a heavy-tailed / skewed conditional distribution (e.g. x_i | bin(x_j) is log-normal, common for financial/count data), a z-score `(x-mu)/sigma` badly misrepresents "how extreme" a row is -- the mean/std pair is not a sufficient statistic for a skewed shape, so two rows with identical z-scores can sit at very different TRUE percentiles (e.g. z=2 might be the 99th percentile in the upper tail but only the 80th in the heavy lower tail of a skewed distribution). A target that depends on "is this row in the top-5% of its conditional peer group" (e.g. fraud/outlier detection conditioned on a merchant category) is exactly the shape z-score under-resolves and quantile-rank resolves directly. This is the 4th member of a family (grouped_agg mean/std -> composite_group_agg -> conditional-dispersion z-score/|z| -> conditional quantile-rank), a proven, actively-extended pattern in this codebase. Sketch: `def conditional_quantile_rank_fe(x_i: np.ndarray, x_j_bins: np.ndarray, *, n_quantile_bins: int = 20) -> np.ndarray:` fit per-bin quantile EDGES on TRAIN (reusing the existing discretize_2d_quantile_batch machinery), at apply-time map each row's x_i into its bin's quantile index via searchsorted, normalize to [0,1]. GPU residency: MEDIUM-HIGH. Exact per-bin quantiles need a per-group SORT (a segmented-sort, which cupy does not ship a highly-optimized primitive for natively) -- the fully-exact GPU form needs a hand-written RawKernel or sort-by-(bin,val)-key-then-walk-segments approach (implementable, matching the acceleration-ladder's RawKernel tier); a cheaper, already-GPU-native approximation is achievable via the existing binned-histogram infrastructure (coarse quantile via cumulative bincount, no sort needed), trading some quantile resolution for full residency.

### (fe_idea) CUSUM-style row-order changepoint / regime-indicator feature

Idea: Page (1954) CUSUM statistic, applied along the natural ROW ORDER (or an explicit time/entity-ordered index) of a numeric column: `S_t = max(0, S_{t-1} + (x_t - mu_hat) - k)`; a large S_t flags a structural break (regime shift) in the column's level at row t. Emit both the running CUSUM value and a binary "post-changepoint" regime indicator (which side of the detected break each row falls on). Why it catches a shape the catalog misses: the existing Layer 92 temporal_agg family (_temporal_agg_fe.py, _temporal_agg_fe_rolling.py) computes leak-safe EXPANDING/ROLLING-WINDOW aggregates (mean/std over a lookback window) -- a smoothly-adapting local statistic -- but has no mechanism for detecting a discrete STRUCTURAL BREAK (e.g. a sensor recalibration, a policy change, a regime switch in a financial series) where the right feature is not "the rolling mean" but "which SIDE of the break this row is on" / "how far past the most recent break". Concrete scenario: y depends on `1{row_index > true_changepoint}` where the changepoint is unknown and must be discovered from the data itself (e.g. a manufacturing-defect-rate series that jumps at an unknown maintenance date) -- no fixed-window rolling stat recovers a clean step-function regime label; a CUSUM-derived changepoint does. Sketch: `def cusum_regime_features(x: np.ndarray, order_idx: np.ndarray, *, k: float | None = None, h: float | None = None) -> tuple[np.ndarray, np.ndarray]:` sort/apply by `order_idx` (entity-scoped, leak-safe -- fit `mu_hat`/`k`/`h` on train only, mirroring the existing temporal leak-safety discipline), compute the running CUSUM via `np.maximum.accumulate`-style recursion, threshold at `h` (auto-tuned via a permutation null, reusing the codebase's existing permutation-null machinery) to emit the binary regime flag. GPU residency: MEDIUM-HIGH. The exact recursive CUSUM (`S_t` depends on `S_{t-1}`) is a SEQUENTIAL scan, not embarrassingly parallel, but has a well-known GPU-parallel prefix-scan formulation (a max-plus scan, expressible via `cupy`'s scan primitives or a segmented parallel-scan RawKernel, the same acceleration-ladder tier used for other sequential kernels in this codebase); a simpler embarrassingly-parallel APPROXIMATION (evaluate the two-sample mean-shift test statistic at every candidate split point via cumulative sums, i.e. `cumsum`-based rather than the exact recursive CUSUM) is trivially GPU-parallel and a reasonable practical substitute.

### (fe_idea) Tail-dependence coefficient scorer (extreme co-movement, distinct from average copula-MI)

Idea: from extreme-value / copula theory (Longin & Solnik 2001; McNeil, Frey & Embrechts 2005): the upper tail-dependence coefficient `lambda_U = lim_{q->1} P(Y > F_Y^{-1}(q) | X > F_X^{-1}(q))` measures how often X and Y are BOTH in their extreme tails simultaneously, estimated empirically as the co-exceedance rate at a fixed high quantile threshold (e.g. q=0.95) divided by (1-q). Why it catches a shape the catalog misses: the shipped Layer 66 copula-MI scorer (_orthogonal_copula_mi_fe.py) computes MI on the FULL rank-uniformized joint distribution -- an AVERAGE dependence measure over the whole [0,1]^2 unit square. On a target where two columns are co-dependent ONLY in their joint extreme tail and essentially independent in the bulk (a Gumbel-copula-like structure, common in financial contagion / extreme-weather-co-occurrence / catastrophic-failure data), the bulk's near-independence DILUTES the average copula-MI toward a middling, easy-to-miss value, while a dedicated tail-dependence statistic isolates exactly the co-exceedance signal that matters for a target like `y = 1{catastrophic joint extreme event}`. This is a genuinely different summary of the SAME rank-transformed data the copula-MI family already computes (reuses its rank-uniformization step), not a duplicate. Sketch: `def tail_dependence_score(u: np.ndarray, v: np.ndarray, *, q: float = 0.95, both_tails: bool = True) -> float:` `u, v` are the existing copula-MI family's rank-uniformized [0,1] transforms; `lambda_U = np.mean((u > q) & (v > q)) / (1 - q)` (and symmetrically for `lambda_L` at `1-q`), floored against a permutation-null co-exceedance rate (reusing the existing permutation-null infrastructure) to distinguish genuine tail dependence from finite-sample chance co-exceedance at small `n*(1-q)`. GPU residency: HIGH. Once `u, v` are already rank-uniformized (an argsort, already GPU-resident per the copula-MI family's own implementation), the co-exceedance rate is a single elementwise comparison + reduction (`cp.mean`), and the permutation null is `n_perm` independent shuffles of `v` -- all cupy-native, no host round trip, cheaper than the copula-MI computation it piggybacks on.
