## Rejected by majority vote (1)

- `evaluation.py:566` -- evaluate_candidate's baseline relevance-null calls (mi_direct / mi_direct_gpu, baseline_npermutations budget) also never receive a seed, so ctx.random_seed never affects the per-candidate baseline permutation gate that runs on every single scored candidate.
  - **Caveat**: all 3 verifiers independently confirm the CPU half of this finding (evaluate_candidate's `mi_direct` baseline-relevance call at evaluation.py:611 really does hardcode `base_seed=0`, `ctx.random_seed` never reaches it). Rejection was purely on a factual imprecision about the GPU half (the finding claimed `mi_direct_gpu` also "silently uses base_seed=0"; in reality `mi_direct_gpu` has no seed parameter at all and draws from unseeded `cp.random.default_rng()` entropy -- arguably a worse defect, non-reproducible rather than deterministic-but-wrong). Treat as **effectively confirmed** with a corrected mechanism description; not a false alarm.
