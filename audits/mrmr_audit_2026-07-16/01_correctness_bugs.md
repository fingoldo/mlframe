# MRMR Module Correctness Audit — Findings

Scope covered: full read of all 8 files in `mrmr/` (`_mrmr_class.py` 3706 lines in chunks, `_mrmr_class_config.py`, `_mrmr_class_fit_helpers.py`, `_mrmr_class_shared.py`, `_mrmr_class_transform.py`, `_mrmr_param_constants.py`, `_mrmr_setstate_defaults.py`, `__init__.py`). Note: `_fit_impl`, `_run_fe_step`, `_validate_inputs`, `transform` body, `partial_fit` etc. are implemented in sibling modules OUTSIDE `mrmr/` (bound onto the class in `__init__.py`) and were out of scope per the task's file list; only the code physically inside `mrmr/` was audited for defects.

## Findings

**1. `_mrmr_class.py:3095-3114` — broad `except Exception` silently converts an invalid/broken `stability_selection_method` into a silent fallback to classic (non-stability) fitting.**
`_stability_outer_fit` (in `_mrmr_class_fit_helpers.py:48-121`) raises `ValueError(f"unknown stability_selection_method={method!r}")` for any value other than `"classic"/"cluster"/"complementary_pairs"` (line 112). The caller in `fit()` wraps the whole call in `try/except Exception` and, on ANY exception (a typo'd config value, or a genuine bug inside `cluster_stability_selection`/`complementary_pairs_stability`), just emits a `UserWarning` and falls through to the classic single-fit path (lines 3106-3114).
- Failure scenario: a user sets `stability_selection_method="cluster"` (or any value) expecting bootstrap-stability-selected features, but a runtime error inside the stability machinery (or a caught typo) makes the fit silently run PLAIN classic MRMR instead — the returned `support_`/`stability_freq_`/`stability_info_` are absent/wrong relative to what was requested, with only an easy-to-miss `UserWarning` (not raised) as evidence. This masks real bugs in the stability path as "expected fallback," and a mistyped `stability_selection_method` value never gets its own validation error at all (an outright invalid string is indistinguishable from a legitimate transient failure).

**2. `_mrmr_class.py:3137-3138` (multi-output early return) — `degenerate_columns_` and other outer-`fit()` diagnostic/state attributes are never populated when `multioutput_strategy` triggers `_fit_multioutput`.**
The multi-output dispatch (`if _mo_strategy in ("union","intersect") and _mrmr_y_is_multioutput(y): return self._fit_multioutput(...)`) happens at line 3137-3138, BEFORE the degenerate-column diagnostic scan at line 3149-3154 (`self.degenerate_columns_ = audit_degenerate_columns(X)`), before the identity-cache logic, before SIS screening, before all the FE/toggle machinery. `_fit_multioutput` (`_mrmr_class_fit_helpers.py:321-362`) never sets `degenerate_columns_`, `provenance_`, `fe_provenance_`, `fe_rejection_ledger_`, etc. on `self`.
- Failure scenario: any 2D-`y` fit (the default `multioutput_strategy="union"`) returns an estimator missing `degenerate_columns_`/`provenance_`/`fe_provenance_` entirely — downstream code that unconditionally reads `mrmr.degenerate_columns_` (a documented public diagnostic attribute per the class docstring) raises `AttributeError` only for multi-output fits, an inconsistent fitted-attribute surface between single- and multi-output modes.

**3. `_mrmr_class_fit_helpers.py:283-287` (`_fit_identity_shortcut`) — synthesized placeholder feature names use `f"f{i}"`, inconsistent with every other synthesized-name convention in the codebase (`f"feature_{i}"`, e.g. `_fit_multioutput` line 329, and the `^feature_\d+$` regex fallback in `_mrmr_class_transform.py:77`).**
```python
_names = [f"f{i}" for i in range(n_cols)]
```
- This is guarded functionally by the explicit `_feature_names_in_synthesized_` sentinel set on the very next fit (`_mrmr_class_transform.py:74-78` prefers the sentinel over the regex when present), so `get_feature_names_out(input_features=...)` on a freshly-fit instance still resolves correctly. However, if such an identity-shortcut-produced estimator is pickled with an OLDER protocol/tooling that drops the sentinel (or a future refactor forgets to carry `_feature_names_in_synthesized_` through), the regex fallback (`^feature_\d+$`) will NOT match `"f0"`/`"f1"` style names and will misclassify the estimator as "not synthesized," incorrectly enforcing the strict column-drift check against opaque placeholder names. Low severity today (masked by the sentinel), but a latent inconsistency worth fixing to `f"feature_{i}"` for uniformity.

**4. `_mrmr_class.py:3336-3356` (`redundancy_aggregator == "auto"`) — broad `except Exception` around `detect_synergy` silently defaults `jmim_engaged=False` on any failure, indistinguishable from a legitimate "not synergistic" data-driven decision.**
```python
except Exception as _exc:
    warnings.warn(...)
    self._synergy_auto_decision_ = {"jmim_engaged": False, "error": str(_exc)}
```
- Failure scenario: a bug in `detect_synergy` (shape mismatch, GPU/thread issue, etc.) silently routes every "auto" fit to plain Fleuret instead of JMIM even on genuinely synergistic data, with only a `UserWarning`; callers inspecting `_synergy_auto_decision_["jmim_engaged"]` cannot distinguish "data judged non-synergistic" from "detector crashed." Lower severity than finding #1 since this only affects one scorer-routing decision (not a whole alternate selection strategy silently discarded), but the same error-swallowing pattern.

**5. `_mrmr_class.py:3370-3371` — `mi_correction="chao_shen"` is accepted by validation but silently degrades to plug-in `'none'` behavior, surfaced only via `logger.warning` (not raised, not a `UserWarning`).**
```python
if _mi_corr == "chao_shen":
    logger.warning("[MRMR] mi_correction='chao_shen' is not yet wired into the relevance/null path and falls back to plug-in MI ('none'); no bias correction is applied this fit.")
```
- Since `logging` is frequently unconfigured in caller scripts (the module's own `_print_fit_summary` docstring at `_mrmr_class_fit_helpers.py:180-184` explicitly notes `logger.info` calls are routinely swallowed for exactly this reason), a user who explicitly requests Chao-Shen bias correction gets ordinary plug-in MI with no visible indication in the common case, silently changing the accuracy characteristics of small-n/high-cardinality selection versus what was requested. This is documented/known behavior in the code, not accidental, but the visibility channel (logger.warning vs. `warnings.warn`) is inconsistent with how the module treats the sibling `chao_shen`-adjacent knobs and is worth promoting to a `UserWarning` so it survives unconfigured-logging setups.

## Areas checked and found correct (no bug)
- `_mrmr_setstate_defaults.py` / `__setstate__` D5 overlay (`_mrmr_class.py:2910-2952`): the three-layer defaulting (roster deep-copy → ctor-default overlay for shared non-override keys → fresh-instance injection for uncovered keys) is internally consistent; every documented `_SETSTATE_LEGACY_OVERRIDES` entry matches its stated legacy-vs-ctor divergence exactly (verified `fe_fallback_to_all`, `mrmr_identity_cache_ycorr_threshold`, `fe_pair_prewarp_enable`, `fe_hybrid_orth_enable`/`triplet`/`quadruplet`, `fe_kfold_te_enable`, `fe_conditional_dispersion_enable`, `fe_wavelet_enable` against the live `__init__` defaults).
- `_coerce_target_dtype` (`_mrmr_class_config.py:234-256`): int64→int16 downcast is correctly range-guarded (previously-noted bug already fixed).
- `_maybe_resample_for_sample_weight` (`_mrmr_class_fit_helpers.py:124-173`): NaN/negative/zero-sum/shape checks are all present and correctly ordered; uniform-weight short-circuit avoids resampling noise.
- `get_feature_names_out`/`transform`/`_append_usability_union`/`_usability_union_extra` width-consistency in `_mrmr_class_transform.py`: verified the two independent code paths (get_feature_names_out and transform) dedupe against the identical base-name set and stay in sync.
- `get_support` (`_mrmr_class_transform.py:165-173`) assumes `support_` is always an integer-index array; verified by grep across the whole `filters/` tree that every `self.support_ = ...` assignment in the MRMR codebase is integer-index (`dtype=np.int64`/`np.int32`), so this is never actually exercised with a boolean mask — not a live bug, despite `get_feature_names_out` defensively handling a boolean-mask case that `get_support` does not.
- `_mrmr_y_is_multioutput` / `_mrmr_y_columns` (`_mrmr_class.py:79-90`, `_mrmr_class_shared.py`): dtype/module-name dispatch for pandas/polars/ndarray is correct; n=1 feature and empty-selection paths in `_fit_identity_shortcut` and `_fit_multioutput` handled correctly (`X.ndim > 1` guard, `if not per_column_selected: raise`).
