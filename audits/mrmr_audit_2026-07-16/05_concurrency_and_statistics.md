## Audit: MRMR module — thread-safety and MI/redundancy math (2 angles only)

### ANGLE 1 — Thread-safety / shared mutable state

**1. GPU circuit breakers are unprotected process-global booleans, reset unconditionally at every `fit()` entry**
`_mrmr_class.py:2978-3002` calls `reset_cmi_gpu_circuit_breaker()` / `reset_mi_direct_gpu_circuit_breaker()` / `reset_pair_maxt_gpu_circuit_breaker()` at the very start of every `fit()`, with the comment "process-global... re-arming at the start of each fit() bounds the cost." The breakers themselves are plain module globals with no lock and no `threading.local`:
- `info_theory/_cmi_cuda.py:726` `_CMI_GPU_FAILED = False` (module global), reset at `:729-732`
- `permutation.py:44` `_MI_DIRECT_GPU_FAILED = False`, reset at `:47-50`
- same pattern for `reset_pair_maxt_gpu_circuit_breaker`

Scenario: thread A trips the breaker mid-fit after a genuine CUDA launch fault (poisoned context) — the breaker exists precisely to stop A from re-attempting GPU calls on that dead context for the rest of A's fit. Thread B starts a fresh, unrelated `fit()` concurrently (multi-model suite, web-service worker, nested joblib) and its entry-time re-arm silently clears the breaker while A is still in flight. A now re-attempts GPU launches on the still-poisoned context, reintroducing the exact retry-storm/crash risk the breaker was built to prevent. This directly contradicts the module's own stated design intent (breaker "permanently disable[s] GPU for the rest of the process" until deliberately reset) under any concurrent-fit usage.

**2. `_FIT_CACHE` replay reads the cached instance outside the lock**
`_mrmr_fit_impl/_fit_impl_core.py:228-240`: the lookup + `move_to_end` is protected by `_MRMR_FIT_CACHE_LOCK` (an `RLock`), but `_replay_fitted_state(self, _cached)` at line 234 runs **after** the `with` block exits, reading attributes off `_cached` with no synchronization at all. If the exact same cached `MRMR` instance is itself concurrently re-`fit()` (same object, re-fit on identical X/y/params so it maps to the same cache key — plausible with a shared/reused estimator in a service), a second thread's replay can read a torn mix of attributes: some already reset for the new in-flight fit, some still holding the old fitted values (`support_`, scores, etc. are not updated atomically as a unit). Result: a replayed selection that is neither the old nor the new fit's actual output.

**3. Cache-miss TOCTOU causes nondeterministic "canonical" cached instance**
`_fit_impl_core.py:228-233` (miss check) vs `:9155-9157` (`MRMR._FIT_CACHE[_cache_key] = self`, also lock-protected). Between the locked miss-check and the locked store, the actual fit work is unlocked — so two threads with an identical cache key both run a full independent fit and then race to write into `_FIT_CACHE`. This is safe from a dict-corruption standpoint (each write is locked) but the *winner* is nondeterministic, and combined with finding #2 means later replays of "the same" logical fit can silently pick up whichever thread's instance won the race, even if the two fits differed subtly (e.g. different sample-weight resampling draws, since the process global RNG containment context is per-fit but not for TOCTOU-duplicated concurrent fits).

**4. MI toggles ARE real `threading.local()` — correctly implemented, but the guarantee is convention-only**
`info_theory/_state_and_dispatch.py:25,42-43,70,87,108-111` — `_SU_STATE`, `_JMIM_STATE`, `_BUR_STATE`, `_MM_STATE`, `_GROUP_MI_STATE`, `_RELAXMRMR_STATE`, `_PID_STATE`, `_CMI_PERM_STATE`, `_CPT_STATE` are genuine `_threading.local()` singletons (not a plain dict/global masquerading as thread-local). Correctness across joblib workers is achieved only because `_evaluation_driver.py:evaluate_candidates()` (lines 198-271) explicitly snapshots the main thread's values, forwards them as plain kwargs, and re-publishes/restores them inside each worker thread's own `try/finally`. This is a fragile-by-convention pattern: nothing in the module *enforces* that every future call site spawning a thread over evaluation code goes through this republish wrapper — a new joblib dispatch that calls `_evaluate_candidates_inner` directly (bypassing `evaluate_candidates`) would silently see default `False`/`0.0` toggles regardless of the caller's actual settings. No such bypass currently exists in the audited files, but the guard is a discipline, not a structural guarantee.

**5. No per-instance lock guards `self`'s own mutable attributes during `fit()`**
`_mrmr_class.py` fit()'s thread-local snapshot/restore (`~3315-3322`, `~3567-3603`) correctly isolates the *toggles* per calling thread. But the same `fit()` call also mutates ordinary (non-thread-local) instance attributes in place with no lock at all: `self._fit_sample_weight_` (`:3215`), `self.<fe_flag>` toggles under `fe_auto` (`:3277-3302`, restored `:3656-3661`), `self.cluster_aggregate_enable` (`:3415-3418`, restored `:3613`), `self.random_seed` / `self.skip_retraining_on_same_content` (`:3453-3463`, restored `:3619-3622`), and the target-cleanup frame/columns (`:3662-3668`). If the **same MRMR instance** is `.fit()`-ed concurrently from two threads (nothing prevents or documents this as unsafe), these plain-attribute read-modify-write sequences race directly — e.g. thread A's `finally` restores `cluster_aggregate_enable` to A's pre-fit value just as thread B's active fit is relying on the DCD-suppressed value B set, corrupting B's FE behavior mid-flight. Only the process-wide `_FIT_CACHE` has a documented lock; per-instance concurrent-fit safety is undocumented and unguarded.

### ANGLE 2 — MI / redundancy math correctness

**6. CMI-permutation / CPT null tests use a seed derived ONLY from the candidate column index**
`evaluation.py:875` (`cmi_perm_stop` path) and `evaluation.py:911` (`cpt_test` path) both call `conditional_permutation_test(..., seed=int(cand_idx))`. The seed depends solely on the candidate's index — not on the MRMR instance's `random_seed`/`random_state`, not on the current greedy round, not on the conditioning set `Z`/`selected_vars`.
- **Correlated null draws across rounds**: the same candidate re-evaluated against progressively larger conditioning sets in later greedy rounds draws the *identical* row permutation every time (`_conditional_permutation.py:56` `rng = np.random.default_rng(int(seed))`), so the permutation-null samples used by the `cmi_perm_stop` early-stop criterion / CPT p-value are not independent across rounds for that candidate — understating true null variance and biasing the stopping decision.
- **Seed-invariance defeats stability audits**: a user varying `random_seed=` across runs to check selection robustness gets a bit-identical cmi_perm/cpt component regardless — that piece of the pipeline silently ignores the seed knob entirely, contrary to the "GLOBAL-RNG CONTAINMENT + SEED DETERMINISM" contract documented at `_mrmr_class.py:3465-3476` for the rest of the fit.

**7. `mi_correction='chao_shen'` is a silent-degrade API trap (self-documented but still a correctness hazard)**
`_mrmr_class.py:3364-3371`: the constructor accepts `mi_correction='chao_shen'` as valid, but it "is not yet wired into the relevance/null path" and silently falls back to the plug-in `'none'` estimator, surfaced only via `logger.warning` (not raised, not reflected in `get_params`/provenance). A caller who sets this expecting the Chao-Shen bias correction gets ordinary plug-in MI with no exception and no attribute marking the fallback — a numerical-correctness footgun distinct from the SU/CSU clamps (`_entropy_kernels.py:315-320,387-396`), which are otherwise carefully guarded (negative-MI floor at 0, denom `<=1e-12` zero-guard, CSU clamped to `[0,1]`) and did not show sign or normalization errors on inspection.

### Scope note
The deepest quantization/binning kernels (`merge_vars`, MDLP edges, discretization module) live entirely outside the listed `mrmr/` files, in `info_theory/` and `discretization/`; the audit checked the SU/CSU/Miller-Madow formulas reachable from the mrmr fit path (found well-guarded) but did not exhaustively re-audit bin-edge arithmetic in `discretization/_kernels.py`, since that file is outside the given file list and likely covered by a separate angle/auditor.
