## Findings (memory/scalability only, large n / large p)

**1. Cluster-stability-selection path builds dense O(n·p) copy + O(p²) correlation matrix, uncapped by p**
`src/mlframe/feature_selection/filters/mrmr/_mrmr_class_fit_helpers.py:48-121` (`_stability_outer_fit`, opt-in via `stability_selection_method="cluster"`/`"complementary_pairs"`) calls into `src/mlframe/feature_selection/filters/_stability_cluster.py:77-100` (`cluster_stability_selection`), which does:
- `Xn = np.zeros((n, p), dtype=np.float64)` (line 89) — a full dense float64 copy of X whenever any column isn't already a plain numeric ndarray (i.e. any DataFrame input, which is the common case), on top of the caller's original X.
- `C = np.abs(Z.T @ Z / n)` (line 100) — a dense **p × p** float64 matrix, not chunked or sparse.
- Trigger scenario: n=1,000,000 rows, p=5,000 columns → `Xn` copy alone is ~40 GB; `C` is 5000²×8B=200MB (fine at p=5000, but at p=50,000 — a realistic "100GB-class frame" column count — `C` alone is 50000²×8B=20 GB, and `Xn` is n·p·8B which dwarfs it). Neither is freed until the function returns, and this runs BEFORE any of the `sis_screen_threshold`/wide-p screening logic in `_mrmr_class.py` gets a chance to cut the pool down — the stability wrapper hands the full column count straight to the correlation clustering step.
- No cap/threshold analogous to `sis_screen_threshold` (used by the main fit path, `_mrmr_class.py:3241-3250`) protects this stability-selection entry point at large p.

**2. Each bootstrap/complementary-pairs replicate runs a full nested MRMR.fit, multiplying per-replicate cost/columns by `stability_n_bootstrap` (default 50)**
`_mrmr_class_fit_helpers.py:67-93` (`_inner_selector`): for every one of `n_bootstrap` (default 50, `_mrmr_class.py:272 stability_n_bootstrap: int = 50`) replicates, a brand-new sibling `MRMR` instance is constructed and `.fit()` on a row-subsample of X (`X_sub.reset_index(drop=True)` at line 72, a real copy for pandas frames). Each nested fit re-runs the *entire* pipeline (cat-FE, FE engineered-feature construction, screening) independently — so at large p, the per-replicate engineered-feature frame (raw + engineered columns) is rebuilt from scratch 50 times sequentially. This is not itself a "multiple simultaneous copies" bug (each `sub` is dropped after its loop iteration and is sequential, not parallel), but:
- Trigger scenario: n=200,000 rows, p=20,000 columns, `stability_n_bootstrap=50` → 50 sequential full nested fits each doing FE construction over 20k columns; wall-time scales linearly in bootstrap count with no shared/reusable screening state across replicates (each replicate re-screens from p=20,000 raw columns rather than reusing a screened candidate pool).
- Secondary effect: each nested `sub.fit()` inserts itself into the process-wide class-level `MRMR._FIT_CACHE` (`_mrmr_class.py:164`, populated in `_mrmr_fit_impl/_fit_impl_core.py:9156`), which is correctly LRU-bounded by `fit_cache_max` (default 4, `_mrmr_class.py:1079`) — so this is NOT an unbounded leak, but 50 distinct bootstrap-replicate cache keys thrash the shared 4-entry LRU cache, evicting any legitimately-reusable cached fit from an unrelated concurrent caller in the same process (e.g. a model-suite fit that would otherwise get a cache hit).

**3. `_fit_multioutput` (2D y) also runs one full nested fit per output column sequentially**
`_mrmr_class_fit_helpers.py:321-362`. Comment at line 327 explicitly confirms X itself is NOT copied (`sub.fit(X, y_col, ...)` shares X) — this part is fine. But same as finding 2, each `sub = clone(self); sub.fit(...)` is a full nested MRMR fit (FE, screening) per output column, and each also lands in the shared `_FIT_CACHE`, contributing to the same LRU-thrashing effect as finding 2 at large output-column counts (multilabel with many targets).

**4. No proactive GPU-memory check before allocation; circuit breaker is purely reactive (post-crash)**
Grepped all 8 target files plus the info_theory/permutation modules they call into (`reset_cmi_gpu_circuit_breaker`, `reset_mi_direct_gpu_circuit_breaker`, `reset_pair_maxt_gpu_circuit_breaker` at `_mrmr_class.py:2978-3002`) — there is no `mem_get_info`/`cudaMemGetInfo`/free-memory check anywhere in the mrmr package. The circuit-breaker mechanism only trips (and only re-arms once per fit, per the 2026-07-09 comment at line 2978) **after** a GPU launch has already faulted — i.e. it's a crash-then-fallback design, not a preflight capacity check. At large n×p, a GPU-resident FE/MI kernel (`fe_optimizer: str = "cupy_kernel"`, line 1021) or the exhaustive synergy sweep (`fe_synergy_exhaustive`, line 1508) can attempt to allocate a GPU buffer sized to the full candidate pool with no upfront guard, so the first sign of trouble is an actual CUDA OOM/launch fault caught by the (reactive) breaker rather than a graceful pre-sized fallback to CPU or a chunked GPU path.
- Trigger scenario: p=10,000+ candidate columns with `fe_synergy_exhaustive="force"` on a GPU with modest VRAM (e.g. 8GB) — the exhaustive C(p,2) sweep's per-batch buffer sizing (`MAX_K=200` per the comment at line 1460) bounds a single kernel call, but there's no check that even that batch size fits in currently-free GPU memory (which could be reduced by another process sharing the GPU) before launching.

**5. `fe_synergy_screen_max_features`/`fe_synergy_max_sweep_cost` gate wall-time, not memory, for the O(p²) synergy sweep**
`_mrmr_class.py:1477, 1515-1520`: `fe_synergy_max_sweep_cost` (default 5e8) gates `n * p²` as a **wall-time** budget (explicitly documented via measured seconds-per-pair-count), and `fe_synergy_screen_max_features` (250) caps the column count entering the O(p²) sweep. These are good time safeguards but there's no explicit memory-budget check (e.g. bytes-per-pair × pair-count vs. available RAM/VRAM) — the sweep is described as using the existing batched `batch_pair_mi_cuda` kernel (implying chunked, not a full p²-sized materialized matrix), so this is a lower-severity/likely-benign observation rather than a confirmed bug — flagging it since it wasn't possible to fully verify batching bounds memory (not GPU memory) without reading the kernel implementation outside the audited file set.

**Non-findings worth noting (already handled correctly, so not re-flagged as bugs):**
- `_FIT_CACHE` (`_mrmr_class.py:164`) is properly LRU-bounded via `fit_cache_max` (default 4) and `OrderedDict.move_to_end`/eviction in `_mrmr_fit_impl/_fit_impl_core.py:9151-9164` — no unbounded growth across repeated `fit()` calls in a long-running process.
- The polars→pandas FE bridge (`_mrmr_class.py:3052-3082`) is a documented zero-copy Arrow view, not a whole-frame copy.
- `sis_screen_threshold` (default 20,000 columns, `_mrmr_class.py:1067`, applied at `3241-3250`) chunks the O(p·n) screen before wide-p fits reach the expensive stages — but this protection is bypassed entirely by the stability-selection entry point (finding 1), which calls the correlation-clustering step directly on the full, unscreened p.
